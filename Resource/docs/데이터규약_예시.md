# 📑 CSV 데이터 규약 — 통합 예시

> [[알파_개발계획]] §2.6②-b 규약(top-down·세그먼트 ID·명명 사전)을 실데이터로 시연하는 예시.
> 파츠 데이터는 [[파츠_인벤토리]] 실측 결과와 1:1 매핑, 스킬 MotionRow는 실측 모션 팔레트(17행) 기준.
#projectTP/데이터규약

## ID 발급 규칙 — 8자리, 대분류별 하위 배치

| 첫 자리(대분류) | 도메인 | 나머지 7자리 배치 |
|---|---|---|
| 1 | 캐릭터종 | `TT`직업 `S`성별 `G`등급 `NNN`일련 |
| 2 | 파츠 | `C`카테고리(1스킨 2얼굴 3머리 4의상 5무기) `SS`스타일 `CC`색상 `NN`예비 |
| 3 | 스킬 | `K`계열(1물리 2마법 3방어…) `NNN`일련 `NNN`예비 |
| 4 / 5 | 아이템 / 보상 | `T`타입 `NNN`일련 `NNN`예비 |
| 9 | 어휘·메타 | `TT`테이블 `NNN`일련 `NN`예비 |

- hair 스타일 번호 통합: m1~m14 → 01~14, f1~f9 → 15~23 (성별 라벨은 `GenderTag` 컬럼으로 — 로직은 컬럼만 읽음).
- 안전장치 2규칙(재확인): ①ID에 넣는 축은 불변 취급(변경=신규 발급) ②로직은 ID 파싱 금지 — 진실은 컬럼.
- 런타임 인스턴스(뽑힌 캐릭터 등)는 스마트 ID가 아니라 증가값/GUID.

## `#` 주석 규약 — 사람용, 게임은 못 봄 (오너 확정 2026-07-08)
- `#`로 시작하는 **컬럼 헤더** = 주석 컬럼 / `#`로 시작하는 **행(첫 셀)** = 주석 행. 임포트 전처리(A0③ 파이프라인)가 제거 후 UE에 공급 — 원본 CSV는 사람 친화, 게임은 정제본만.
- **`#id_txt`**: Id 바로 옆 사람용 번역 슬러그 — 숫자만 보고 사람이 알 수 없는 문제 해결. `10112001` → `char_knight_m_rare`.
- `#id_txt`는 손으로 유지하지 않는다 — **검증 스크립트가 컬럼 값에서 자동 생성/대조**(불일치 리포트). 데이터 진실은 언제나 정식 컬럼.
- `strings.csv`는 Key 자체가 이미 사람이 읽는 텍스트라 `#id_txt` 불필요.

`#id_txt`에 리소스 원명(f7·spear)이 그대로 들어가 사람이 파일과 즉시 대조 가능. 마지막 행처럼 `#` 시작 행은 통째로 주석.
- `24150100` = 파츠·의상·스타일15·색1 = hero_knight 갑옷(`cloth15_c1`).
- `GenderTag`·`JobTags` = 어휘 값 사용(계층 위반 아님). **오너의 직업/성별 가이드가 채울 컬럼이 바로 이 둘.**


`10112001` = 캐릭터·직업01·남(1)·2등급·001. **소프트웨어는 숫자를, 사람은 옆의 `char_knight_m_rare`를 읽는다.** ID 세그먼트(성별=1)와 `GenderId` 컬럼이 공존 — 발급 규칙은 같지만 코드는 컬럼만 읽는다(이중 진실 방지). `#id_txt`도 검증 스크립트가 컬럼에서 생성.

## 참조 방향 요약
```
      [어휘] jobs·genders ← 값 사용은 어느 테이블이든 OK
            ↑값                ↑값
캐릭터종 ──SkillIds──▶ skills          ↓ 위가 아래를 참조, 아래는 위를 모름
    └──(생성 규칙)──▶ parts
items ──RewardId──▶ rewards ──ItemIds──▶ items   ★등록된 순환 예외만 허용
```
