# 🔄 개발 워크플로우 (단계별 게이트)

> projectTP 개발의 **정본 프로세스**. Director가 이 절차로 팀을 지휘한다.
> 요약은 상위 `CLAUDE.md`에. 이 문서는 상세 절차·템플릿이다.
> 관련: [[에이전트팀_설계]] · [[projectTP_허브]]

---

## 핵심 원칙

**개발을 단계(stage)로 쪼갠다. 각 단계는 테스트 게이트를 통과해야 다음으로 넘어간다.**

> 간단한 1~2줄 수정·오타·로그 추가는 이 프로세스를 생략하고 Director가 직접 처리한다. diff를 한 문장으로 설명할 수 있으면 생략.

---

## 기능별 폴더 구조

모든 기능(feature)은 자기 폴더를 갖는다:

```
docs/features/<기능명>/
├─ 청사진.md     ← Director: 큰 그림·목표·범위·비범위
├─ plan.md       ← 단계 분해 + 각 단계 TC + 진행 체크리스트
└─ 진행로그.md    ← (선택) 결정·이슈 기록
```

템플릿: `docs/features/_TEMPLATE/`

## 5단계 프로세스

### ① 청사진 (Director 주도)
기능 폴더 생성, `청사진.md` 작성 — 목표, 범위/비범위, 큰 흐름, 관련 에이전트, 열린 질문.

### ② 세부 plan (Director + 서포트 에이전트)
- 개발 요소를 **독립 검증 가능한 단계로 분해**.
- designer/필요 에이전트가 각 단계 명세 서포트 → `plan.md`.

### ③ plan 피드백 + TC 제작 (병렬)
- 관련 **모든 에이전트 + qa-critic**이 plan에 피드백 (한 메시지에서 병렬 호출).
- **qa-critic이 각 단계별 TC를 설계** → `plan.md`에 기록.
- Director가 피드백 종합 → plan 확정.

### ④ 단계별 개발 + 게이트
각 단계를 순서대로:
1. **개발** — gameplay-engineer / scene-builder / art-pipeline / hd2d-art-director
2. **테스트 실행** — verifier가 그 단계 TC를 빌드·PIE·스크린샷으로 실증
3. **게이트 판정** — Director가 TC 결과 확인 → 통과면 다음 단계로
- **이월(deferred)**: 이번 단계서 검증 불가하고 이후 단계서 검증되는 TC는, Director가 확인 후 "이월" 표기하고 넘긴다. 이월 TC는 `plan.md`에서 명시적으로 추적한다.

### ⑤ 최종 풀 테스트
- 전 단계 완료 후 **designer + qa-critic + verifier**가 전체 검증.
- designer: "의도대로 나왔는가" / qa-critic: 통합 엣지케이스 / verifier: 전체 실증.
- **이월 TC 전부 해소** 확인.

## 역할 요약 (게이트)

| 일 | 담당 |
|---|---|
| 단계별 TC 설계 | **qa-critic** |
| TC 실행·실증 | **verifier** |
| 게이트 통과 판정 | **Director** |
| 이월 승인·추적 | **Director** |

## TC 표기 규칙 (plan.md 안)

각 TC 한 줄로:

```
[단계][ID] 조건 → 기대결과 | 상태(대기/PASS/FAIL/이월)
예) [S1][TC-03] 타겟 없는 스킬 사용 → 액션 취소·게이지 유지 | 대기
```

- 상태는 개발 진행에 따라 verifier가 PASS/FAIL로 갱신, Director가 이월 판정.
- FAIL이 하나라도 있으면 그 단계 게이트는 통과 아님(이월 승인된 것 제외).
