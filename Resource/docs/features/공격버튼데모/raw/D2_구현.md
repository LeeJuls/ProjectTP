## ① BP_BattleSpawnPoint 런타임 확장 (`/Game/Blueprints/BP_BattleSpawnPoint`)

**Construction Script 무변경 확인**(`find_nodes`로 UserConstructionScript 노드 13개 — 기존 T1 구현과 동일 개수, 이번 세션에서 EventGraph만 조작).

변수 추가: `SpriteMID`(MaterialInstanceDynamic object ref, instance_editable=false 기본).

**BeginPlay**(EventGraph):
```
BeginPlay.then
  → CreateDynamicMaterialInstance(self=GetSprite, ElementIndex=0, SourceMaterial=GetMaterial(self=GetSprite, ElementIndex=0))
    .then → SetSpriteMID(CreateDynamicMaterialInstance.ReturnValue)
```

**커스텀 이벤트 `PlayAttack`**:
```
PlayAttack.then → IsValid(GetSpriteMID).exec
  Is Valid
    → SetScalarParameterValue(self=SpriteMID, ParameterName="TimeOffset", Value=GetGameTimeInSeconds)
    → SetScalarParameterValue(self=SpriteMID, ParameterName="FrameCount", Value=8.0)
    → SetScalarParameterValue(self=SpriteMID, ParameterName="RowIndex", Value=5.0)
    → RetriggerableDelay(Duration=0.95)
    → SetScalarParameterValue(self=SpriteMID, ParameterName="TimeOffset", Value=0.0)
    → SetScalarParameterValue(self=SpriteMID, ParameterName="FrameCount", Value=6.0)
    → SetScalarParameterValue(self=SpriteMID, ParameterName="RowIndex", Value=0.0)
  Is Not Valid → (미연결, null 가드)
```

`compile_blueprint(warnings_as_errors=true)` → **에러/경고 0**.

## 발견한 함정 (Director 보고용)

1. **`set_actor_transform`/`add_to_scene_from_asset`의 xform 부분 지정 시 identity 강제(중대, 재확인)**: `set_actor_transform`을 `xform={rotation: {...}}`만 지정(location/scale 생략)해서 호출하면, 스키마 설명("설정되지 않은 필드는 기존 오브젝트 수정 시 변경 없음을 의미")과 달리 **실제로는 location과 scale이 identity(0,0,0 / 1,1,1)로 강제 리셋**됐다. `CaptureViewport`의 `labeledActors[].worldLocation`으로 실측 검증해 발견(액터가 카메라 근처가 아니라 원점 부근에 있었음). **반드시 location/rotation/scale 3필드 전부 명시**하는 워크어라운드로 해결. T1T2 로그의 "결함 A"(`add_to_scene_from_asset`)와 동일 계열 버그가 `set_actor_transform`에도 존재함을 재확인 — 이 프로젝트의 트랜스폼 관련 도구 전반에 이 함정이 있다고 가정하고 작업할 것.
2. **`set_properties`의 Vector 구조체 다중 필드 설정 시 일부만 반영되는 비결정적 결함(재확인, 신규 사례)**: TextRenderComponent(레벨 배치된 액터의 서브 컴포넌트)의 `relativeLocation`을 `{x:0,y:0,z:-20}`처럼 3필드 동시 지정했더니 `x`만 반영되고 `y`,`z`는 무시됨(`returnValue=true`인데도). **필드별 개별 `set_properties` 호출**(z만 별도)로 100% 안정적으로 해결. T1T2 로그에서 발견된 결함 B가 Blueprint CDO뿐 아니라 **레벨에 배치된 액터의 컴포넌트에도 동일하게 재현**됨을 확인 — 이 프로젝트 전역 워크어라운드로 표준화 필요.
3. **`find_node_types`가 context_pins 유무·값에 따라 다른(때로 실제 생성 불가능한) type_id 문자열을 반환(신규)**: 예) `PlayAttack` 커스텀 이벤트 호출 노드를 찾을 때 context_pins에 IsValid 출력을 넣으면 `함수호출|PlayAttack`(존재하지 않음, create_node 실패)을 반환하지만, context_pins를 비워서 검색하면 `Class|BPBattleSpawnPoint|PlayAttack`(정상 생성됨)을 반환. `SetShowMouseCursor`/`SetEnableClickEvents`도 유사(처음엔 "Variables|마우스인터페이스|..."라는 존재하지 않는 카테고리, GetPlayerController 리턴값을 context로 재검색하니 "Class|플레이어컨트롤러|..."로 정정). **워크어라운드: type_id 후보가 여러 개 나오면 실제로 `create_node`를 시도해 실패하면 다른 context 조합(빈 배열, 다른 참조 오브젝트)으로 재검색.**
4. **MID(MaterialInstanceDynamic)의 SetScalarParameterValue와 MaterialParameterCollection의 동명 함수 오버로드 충돌**: `find_node_types(type_id_filter="SetScalarParameterValue")`가 반환하는 `렌더링|머티리얼|SetScalarParameterValue`는 기본적으로 **MPC(MaterialParameterCollection)용 오버로드**를 생성한다(Collection/ParameterName/ParameterValue 핀). MID용(self=MID, ParameterName, Value 핀)을 만들려면 **`create_node`에 `declaring_class={refPath:"/Script/Engine.MaterialInstanceDynamic"}`를 명시**해야 한다. 지시서의 "SetScalarParameterValue는 오철자 무음 무시" 함정 외에, **오버로드 자체를 잘못 생성하는 함정이 별도로 존재**함을 확인 — 이후 유사 작업 시 declaring_class 명시를 표준 절차로 삼을 것.
5. **[오진 정정, 공격버튼데모 마무리 세션에서 재검증] 에디터 뷰포트(및 CaptureViewport)가 TextRenderComponent를 표시하지 않는 것으로 추정(당시 판단)**: Label(TextRenderComponent)의 위치(z=2→20→200→-20 각 시도)·회전(yaw=0→180)·텍스트 내용("Text"→"HELLO")·worldSize(48→300)를 모두 바꿔가며 최소 8회 캡처로 검증했으나 **단 한 번도 텍스트가 화면에 렌더링되지 않았다**. 반면 ButtonBg(StaticMeshComponent, Unlit 머티리얼)는 정상 렌더링됨. `SearchCVars("TextRender")` 조회 결과 `ShowFlag.TextRender=2`(오버라이드 없음)를 확인 — 당시엔 UE 에디터가 뷰포트에서 TextRenderComponent를 기본적으로 숨기는 Show flag 동작일 가능성이 높다고 판단했었다.
   **→ 오진으로 판명(정정)**: 실제 원인은 Show flag가 아니라 **Label의 회전(RelativeRotation)이 부모(ButtonBg) 회전(pitch90,yaw84)을 상쇄하지 못해 텍스트 정면 법선이 하늘(월드 +Z)을 향해 카메라에서 전혀 안 보였던 것**(D3 결함2 조사에서 확정, 아래 "원인 1" 참고). `RelativeRotation=(pitch90,yaw0,roll180)`으로 수정한 뒤 CaptureViewport·PIE 양쪽에서 텍스트가 정상 렌더링됨을 재확인(공격버튼데모 마무리 세션, `D3fix2_label.png`). 즉 **CaptureViewport와 에디터 뷰포트는 TextRenderComponent를 정상적으로 표시하며, Show flag 관련 특수 동작은 없다.** 이 항목의 "D3에서도 안 보이면 근본원인 조사 필요"라는 인계는 조건 자체가 성립하지 않게 됐다(D3에서 회전 수정 후 정상 확인).

## 명세와 다르게 구현한 부분 (qa-critic 대조용)

1. **버튼 최종 위치가 지시서 시작점과 다름**: 지시서 시작점 `(-38,-7356,480)`에서 반복 캡처 검증 후 `(-38,-7300,420)`으로 조정(Y +56cm, Z -60cm). 근경 펜스 말뚝과의 시각적 겹침을 줄이기 위한 미관 조정이며, "캐릭터 미가림" 요구사항은 시작점에서도 이미 충족했었음 — 순수 개선 목적.
2. **버튼 스케일**: 지시서는 "2:1 예시"라고 했으나, 초기 스케일(2,1,1)로는 화면에서 너무 작아 최종적으로 **(4,2,1)**로 확대(400×200cm, 여전히 가로2:세로1 비율 유지).
3. **회전값**: 지시서는 "카메라 yaw84의 역방향 기준으로 시작"이라고 명시했으나, 실측 결과 Plane 메시(로컬 법선=Z)를 카메라 정면으로 향하게 하려면 **pitch=90, yaw=84**(카메라와 동일 yaw)가 정답이었다. pitch=-90 및 look_at 기반 계산은 전부 실패(안 보임)로 실측 배제.

## D3 게이트 결함 2건 수사 및 수정 (gameplay-engineer, 후속 세션)

Director 실클릭(D3) 결과 결함 2건 발견 → 근본 원인 수사 및 수정. 수정 범위는 이 2건에 한정(다른 에셋·레벨 요소 무변경).

### 결함 1 — "복귀 체인 미작동" 보고 → 정적 분석 결과 배선 자체는 완전 정상 (수정 불필요, 판정 재검토 필요)

`get_node_infos`/`get_connected_subgraph`로 `BP_BattleSpawnPoint.PlayAttack` 커스텀 이벤트부터 전체 exec 체인을 노드 단위로 정밀 추적:

```
PlayAttack.then → IsValid.exec → (IsValid) → SetScalar(TimeOffset=GetGameTime).execute
  → .then → SetScalar(FrameCount=8.0) → .then → SetScalar(RowIndex=5.0)
  → .then → RetriggerableDelay(Duration=0.95).execute → .then
  → SetScalar(TimeOffset=0.0).execute → .then → SetScalar(FrameCount=6.0)
  → .then → SetScalar(RowIndex=0.0) → .then (체인 끝)
```

**모든 구간의 output_pin.connected_pins ↔ input_pin.connected_pins 양방향 일치를 개별 확인** — exec 미배선 지점 전무. `self`(전 구간 GetSpriteMID 동일 참조)/`ParameterName`(TimeOffset/FrameCount/RowIndex 철자 정확)/`Value`(0.0/6.0/0.0)/`Duration`(0.95) 전부 지시서 기대값과 일치. 즉 **지시서가 예상한 "exec 미배선" 계열 결함은 이 Blueprint에 존재하지 않음.**

**시각적 재현 시도**: BP_AttackButton BeginPlay 끝에 임시 스캐폴드(Delay(1.0)→PlayAttack)를 붙여 PIE 캡처로 A1의 idle/attack 자세 전환을 확인하려 했으나, 배틀 카메라 각도(yaw84, A1 뒤통수만 보임)에서 idle 자세와 공격 자세(스프라이트시트 RowIndex 0 vs 5)의 육안 구분이 어려워 **100% 확정적 시각 판정은 이번 세션에서 얻지 못함**. 다만 t=1.3s와 t=3.3s+ 캡처를 픽셀 비교(diff mean=18.6)한 결과 팔/어깨 위치가 명확히 다름을 확인 — 애니메이션이 프리즈(고정)되어 있지 않고 계속 진행 중임은 확인됨.

**결론**: 정적 배선 분석 기준으로 결함 없음. Director 원 보고("~1.4s·~2.5s에도 공격 사이클 유지")가 재현 안 됨 — 재현 조건(연속 클릭 등) 차이였을 가능성. **수정 불필요 판정, qa-critic/Director 재검토 요청.**

### 결함 2 — Label 텍스트 미표시: 복합 원인 중 2개 확정 수정, 1개 근본원인 미특정(범위 내 워크어라운드 실패)

**원인 3 (미확정, 미해결) — SetText 함수 호출이 런타임에 반영 안 됨**: 위 2개 수정 후에도 텍스트 문자열이 "Attack"으로 바뀌지 않고 컴포넌트 디폴트값 "Text" 그대로 표시됨. 광범위한 격리 테스트로 다음을 확정:
  - `PrintString`으로 BeginPlay 실행 자체를 로그 확증(`LogBlueprintUserMessages: [BP_AttackButton_C_0] DIAG_BEGINPLAY_FIRED_12345` 매 PIE 세션 재현).
  - `IsValid(GetLabel())` 진단으로 `Label` 컴포넌트가 Valid함을 로그 확증(`LABEL_IS_VALID`).
  - `SetText`의 `self`/`Value` 배선 정상, `MakeTextfromStringTable` 대신 리터럴 텍스트로 교체해도 동일 증상.
  - `SetText` 호출 경로를 BeginPlay 직결/IsValid 우회/SetInputModeGameAndUI 우회 등 5가지 방식으로 재배선해도 전부 "Text" 고정.

### 함정 발견 (신규)

6. **Blueprint EventGraph exec output 핀은 다중 연결(fan-out) 불가 — `connect_pins`로 이미 연결된 output 핀에 새 연결을 추가하면 기존 연결이 조용히 대체(교체)됨**: 진단용 노드를 기존 배선에 "병렬로 추가"하려고 `BeginPlay.then`에 새 노드를 연결했더니, 원래 `IsValid`로 가던 연결이 사라지고 새 노드로만 연결됨(`get_node_infos` 재조회로 발견, 즉시 원복). exec input 핀은 다중 연결 허용(여러 소스에서 호출 가능)되지만 output 핀은 UE 표준대로 1개만 유지 — **fan-out이 필요하면 Sequence 노드를 명시적으로 삽입할 것.** 진단 배선 시 반드시 대상 output 핀이 비어있는지 사전 확인 필요.
7. **`get_node_type_pins`가 그래프에 실제로 노드를 생성했다가(또는 스크래치 상태로) 남기는 것으로 추정**되나, `find_nodes`로 실제 그래프를 재확인하면 해당 임시 노드는 존재하지 않음 — 그래프 오염은 없었으나 반환된 노드 refPath(`K2Node_CallFunction_21` 등, 실제 그래프의 다음 인덱스보다 앞서는 번호)가 혼란을 줄 수 있음. 이 도구가 반환하는 node refPath는 실제 그래프 노드가 아닌 것으로 간주하고 참조하지 말 것.
8. **`editor_toolset` 계열(`ObjectTools.get_properties`/`set_properties`, `ActorTools.SelectActors` 등)은 PIE 런타임 인스턴스가 아니라 에디터 월드(레벨에 배치된 CDO 오버라이드 인스턴스)를 대상으로 동작하는 것으로 강하게 추정됨**: PIE 중에도 이 경로로 조회한 `SpriteMID`/`Text` 등 BeginPlay가 세팅하는 프로퍼티는 항상 `None`/디폴트값을 반환(BeginPlay 미실행 증거가 아님, 단순히 에디터 인스턴스를 보고 있는 것). 반면 이 경로로 **설정한 값**(회전/크기/색상/위치 등 정적 프로퍼티)은 PIE 렌더링에 정확히 반영됨 — 즉 "쓰기"는 유효한 인스턴스에 적용되지만 "읽기"는 BeginPlay 실행 전 스냅샷을 보여주는 것으로 추정. **PIE 런타임에 세팅된 값(SetText 등 함수 호출 결과)을 이 경로로 검증하려 하지 말 것** — `GetLogEntries` + `PrintString` 조합이 유일하게 신뢰 가능한 런타임 상태 확인 경로.
