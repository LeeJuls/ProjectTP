# P2 단계별 TC 설계 — 옥토패스대치 v2 (qa-critic)

> 상위: [[../청사진|옥토패스대치 청사진]] · plan: [[../plan]] · 프로세스: [[../../../개발_워크플로우]]
> 전 단계 검토(C1·C2·H2·H3·H4·M1~M3)가 승인 plan에 반영됨. 본 문서는 그 엣지케이스를 **verifier가 MCP/스크린샷으로 실행 가능한 TC**로 구체화한 것이다.
> **작성 규칙**: 각 TC는 (a) 재현 조건 = verifier가 실제로 부를 도구 시퀀스, (b) 기대결과 = 판정 임계값, (c) 심각도, (d) 근원(어느 적대적 발견에서 왔는지)을 명시. 수정·구현·TC실행은 하지 않음(설계만).

## 0. verifier 실행 전제 (모든 게이트 공통 — 반드시 먼저 읽을 것)

이 전제가 깨지면 아래 TC 결과가 **거짓 PASS/FAIL**이 되므로 게이트 진입 조건으로 강제한다.

- **[전제-1] 에디터 기동 + MCP 연결**: `_Launch_MCP.bat`로 언리얼 에디터가 떠 있어야 MCP 동작(CLAUDE.md). PIE 아님 — 에디터 뷰포트 상태로 검증.
- **[전제-2] 캡처 도구 = 뷰포트 카메라 기반**: `CaptureViewport`/`WorldPosToScreenCoords`는 **레벨의 CameraActor가 아니라 에디터 뷰포트 카메라**를 쓴다. 따라서 카메라 검증 TC는 항상 **먼저 `SetCameraTransform`으로 A안 값(-90,-7850,750 / pitch-6,yaw84,roll0)을 세팅한 뒤** 투영/캡처해야 한다. `CaptureViewport(captureTransform=...)` 인자로 포즈를 직접 넘기는 방식도 허용(권장: captureTransform 명시로 상태 누수 차단).
- **[전제-3] base64 토큰 한도 회피(P1 실패 교훈)**: `CaptureViewport`의 `image.data`(base64)를 **디코드하지 않아도** 반환 메타데이터 `labeledActors[]`(name·screenPosition(px)·worldLocation·distanceCm)·`cameraFOV`로 수치 판정이 가능하다. **수치 TC는 메타데이터로 판정**하고, **육안 TC(도트 무손상·seam·명암·어색함)만 이미지 디코드가 필요**하다. 육안 캡처는 해상도를 낮추거나(뷰포트 축소) 대상 1~2기로 좁혀 토큰 한도(약 6.2M자) 내로 떠라. 한 캡처에 8기 전원 풀프레임 고해상도는 P1에서 이미 한도 초과했다 — **분할 캡처**가 기본.
- **[전제-4] 레벨 스위칭 = 뷰포트 상태 리셋**: `load_level`로 다른 레벨(map_battle_village)을 열면 octopath 뷰포트 세팅이 날아간다. 따라서 **정면레벨 회귀 TC(village 대상)는 octopath 캡처를 전부 끝낸 맨 마지막에** 배치하고, 끝나면 octopath를 다시 로드해 후속 TC 재개. 게이트 내 TC 실행 순서를 이 문서 기재 순서대로 지킬 것.
- **[전제-5] 좌표 규칙 리마인더**: 월드 **-X = 화면 우측(아군/flip)**, **+X = 화면 좌측(적/원본)**. screen.x 0=좌, 1=우. 이 부호를 헷갈리면 "마주봄" 판정이 통째로 뒤집힌다.
- **[전제-6] 애셋 경로**: 마스터 `M_Sprite_Flipbook_Lit`(경로는 S1 산출 시 확정, 통상 `/Game/Materials/`), flip MI 4종 `/Game/Materials/MI_Party_A1_flip`..`A4_flip`, 원본 아군 `MI_Party_A1`..`A4`, 적 `MI_Enemy_B1`..`B4`, 텍스처 `T_Party_A1`..`A4`. 경로 불일치 시 그 자체가 FAIL(명세-구현 불일치, High).

## G1 — flip 게이트 (S1 산출물 검증, 최중요)

> 목적: flip MI 4종이 **구조적으로 올바르고**(부모·파라미터), **렌더가 실제로 좌향**이며, **원본/정면레벨을 오염시키지 않았음**을 증명. C1(정면레벨 공유) 회귀가 이 게이트의 핵심 안전장치.
> 실행 순서 권장: TC-01~04(애셋 구조, 레벨 무관) → TC-05~07(마스터 그래프·회귀, village 로드 전이면 순서 자유) → TC-08~10(octopath 로드+뷰포트 렌더 육안) → **TC-11(village 회귀, 맨 마지막) → octopath 재로드**.

### 구조 검증 (레벨 로드 불필요, 애셋 조회만)

`[G1][TC-01]` **flip MI 4종 존재·부모 확인** — 조건: `AssetTools`로 `/Game/Materials/MI_Party_A1_flip`..`A4_flip` 4개 애셋 존재 확인 + 각각 `MaterialInstanceTools.list_parameters` 호출 성공(로드 가능). 부모 확인은 `ObjectTools`로 `Parent` 프로퍼티 조회 → 4종 모두 부모 = 마스터 `M_Sprite_Flipbook_Lit`. → **기대**: 4/4 존재, 4/4 부모=마스터. 하나라도 누락/오부모면 FAIL. | 심각도 **High**(명세위반) | 대기

`[G1][TC-02]` **flip MI FlipX=1 오버라이드** — 조건: 각 flip MI에 `get_scalar_parameter(name="FlipX")` → 값 확인 + (가능하면) `set_parameter_override` 상태가 override=true인지 `list_parameters` 결과로 교차. → **기대**: 4종 전부 FlipX **effective=1.0**. (주의: `get_scalar_parameter`는 상속값도 반환하므로, 마스터 기본이 0인 상태에서 1.0이 나오면 이 인스턴스가 오버라이드한 것으로 확정.) 하나라도 ≠1이면 그 캐릭만 우향으로 렌더 → FAIL. | 심각도 **Crit**(방향 오류 = 기능 목적 실패) | 대기

`[G1][TC-03]` **flip MI SpriteTex 매핑 1:1** — 조건: 각 flip MI에 `get_texture_parameter(name="SpriteTex")`(파라미터명은 S1 확정값 사용) → 반환 텍스처 경로 확인. → **기대**: `MI_Party_A1_flip`→`T_Party_A1`, A2_flip→T_Party_A2, A3_flip→T_Party_A3, A4_flip→T_Party_A4. **뒤섞임(A1_flip이 T_Party_A2를 참조 등) 금지**. 매핑 어긋나면 캐릭터 얼굴이 뒤바뀜 → FAIL. | 심각도 **High**(명세위반) | 대기

`[G1][TC-04]` **flip MI 잔여 파라미터 원본과 동일(회귀)** — 조건: flip MI 1종(A1_flip)과 원본 MI_Party_A1의 `list_parameters` 전수 비교 — FlipX만 다르고(1 vs 0/상속) col·GridX·GridY·RowIndex·FPS 등 배선 파라미터는 동일해야. → **기대**: FlipX·SpriteTex 외 파라미터 값 원본과 일치(플립북 재생 파라미터가 flip 때문에 바뀌면 안 됨). 불일치 파라미터 발견 시 FAIL(그 파라미터가 애니에 영향 주면 재생 깨짐). | 심각도 **Med** | 대기

### 마스터 그래프 무결성 (§2 삽입지점 버그 — 구조적 검증)

`[G1][TC-05]` **마스터 FlipX 파라미터 존재·기본0** — 조건: 마스터 `M_Sprite_Flipbook_Lit`에 `list_parameters` → "FlipX" Scalar 존재 확인 + `get_scalar_parameter(마스터, "FlipX")`=0. → **기대**: FlipX 존재 & 기본 **0**(C1 완화의 근거 — 기본0이라 기존 MI/정면레벨 무영향). 기본이 0이 아니면 정면레벨 오염 위험 → FAIL. | 심각도 **Crit**(C1 근거 붕괴) | 대기

`[G1][TC-06]` **FlipX Lerp 삽입 지점 = col 오프셋 더하기 전(§2 재생순서 버그)** — 조건: `MaterialTools.get_property_input(MP_CustomizedUVs0)`로 UV0을 구동하는 익스프레션 추적 → `get_expressions`/`get_expression_inputs`로 그래프를 거슬러 올라가 **Lerp(A=baseU, B=OneMinus(baseU), Alpha=FlipX) 출력이 U에 반영된 뒤, 그 결과에 col 오프셋(Add)이 더해지는 순서**인지 확인. 즉 배선 위상: `TexCoord→ComponentMask(U)→[Lerp/OneMinus 분기]→Add(col오프셋)→...GridX`. **OneMinus/Lerp가 col오프셋 더한 후에 걸리면 프레임 재생순서 역전 버그**. → **기대**: Lerp가 col-Add **이전**. 순서 뒤바뀌면 FAIL(TC-09 애니 재생에서 증상으로도 잡히지만, 여기서 구조로 선검출). | 심각도 **High**(잠재 재생 버그) | 대기 | 근원: §2 삽입지점 경고

`[G1][TC-07]` **[C1 회귀-A] 원본 아군 MI 4종 FlipX 미오염** — 조건: `MI_Party_A1`..`A4`(원본) 각각 `get_scalar_parameter("FlipX")` → **effective=0**(상속 or 명시 0). + `list_parameters`로 원본 MI에 FlipX 오버라이드가 **켜져 있지 않음** 확인. → **기대**: 원본 4종 전부 FlipX=0. **S1 작업이 마스터에 파라미터 추가하면서 원본 MI를 건드리지 않았음을 증명**. 원본 중 하나라도 FlipX=1이면 정면레벨에서 그 캐릭이 뒤집힘 → FAIL. | 심각도 **Crit**(C1 직접) | 대기 | 근원: C1

### 렌더 실증 (octopath 로드 → 뷰포트 캡처, 육안)

> 주의: S1 시점에는 아직 아군 액터에 flip MI가 배정 안 됐을 수 있음(액터 재배치는 S2). 그렇다면 **G1 렌더 검증은 임시 쿼드 1~2기에 flip MI를 얹어** 수행하거나(권장: 원본 액터 불변 유지), 원본 아군 액터의 OverrideMaterial을 flip으로 **임시 교체 후 캡처하고 즉시 원복**. 어느 쪽이든 **map_battle_octopath에서만**, village 미접촉. 아래 TC는 "flip MI가 적용된 쿼드"를 대상으로 기술한다.

`[G1][TC-08]` **flip 렌더 = 좌향(무기 방향 반전)** — 조건: octopath 로드 → flip MI 적용 쿼드 1기를 카메라 정면(정투영 근사: 임시로 뷰포트를 해당 쿼드 정면 -X 쪽에 배치)에서 `CaptureViewport`(대상 1기로 좁혀 저해상도) → 디코드하여 **무기가 화면 반대쪽(원본 대비 좌우 반전)**·눈/가르마 등 3/4뷰 비대칭 요소가 반대편에 있는지 육안. 원본 MI 동일 쿼드 캡처와 나란히 비교. → **기대**: flip 쿼드의 무기/얼굴 방향이 원본의 **거울상**. 반전 안 됐거나(FlipX 미적용) 상하반전 등 엉뚱한 축이면 FAIL. | 심각도 **Crit**(기능 목적) | 대기

`[G1][TC-09]` **애니 재생 중 flip 유지 + 프레임 순서 정상** — 조건: flip 쿼드를 PIE 없이 에디터에서 플립북 재생시키거나(머티리얼 시간 기반 애니면 뷰포트 리얼타임 on), 최소 **2개 이상 시점(t0, t0+Δ)** 캡처해 프레임이 진행하는지 + 진행 방향이 정상(거꾸로 감기 아님)인지 육안. → **기대**: (a) 재생 내내 좌향 유지(중간에 우향으로 튀지 않음), (b) 프레임이 **정방향** 진행(TC-06 삽입지점 버그의 런타임 증상 = 역재생/프레임 점프 없음). 역재생/깜빡임 시 FAIL. | 심각도 **High** | 대기 | 근원: §2 삽입지점

`[G1][TC-10]` **도트 무손상 + 셀 경계 seam 없음** — 조건: flip 쿼드를 **확대 캡처**(쿼드가 화면 대부분 차지하도록 뷰포트 근접) → 디코드하여 (a) 픽셀 에지가 뭉개지지 않음(필터링/밉맵으로 인한 blur 없음 — 도트 샤프), (b) **스프라이트시트 셀 경계에서 인접 셀 픽셀이 새어 들어오는 seam/이중선 없음**(flip=OneMinus(U)가 셀 경계 U를 뒤집으면서 샘플링이 반 픽셀 밀리면 seam 발생 가능 — 적대적 관심 지점). 원본(비flip) 동일 확대 캡처와 seam 유무 대조. → **기대**: 도트 샤프·seam 없음. flip에서만 seam이 새로 생기면 FAIL(원본에도 있으면 상속, Director 판정). | 심각도 **High**(flip 고유 아티팩트) · seam이 원본에도 있으면 Med로 강등 | 대기

`[G1][TC-M3]` **[오너 육안] 비대칭 디테일 반전 어색함** — 조건: TC-08 캡처(flip 4종 전부 1회씩)를 오너에게 제시 → 눈·가르마·장비 등 비대칭 요소가 반전되어 "어색/거부감" 있는지 **오너 주관 판정**. → **기대**: 오너 OK. qa-critic은 **자동판정 불가 → 오너 플래그**로만 넘김. 오너가 특정 캐릭 반려 시 Director가 개별 대응(해당 캐릭만 비flip 유지 등) 결정. | 심각도 **Med**(주관) | 대기 | 근원: M3

### 정면레벨 회귀 (맨 마지막, village 로드)

`[G1][TC-11]` **[C1 회귀-B] map_battle_village 아군 4기 무반전·MI 불변** — 조건: **(octopath 캡처 전부 완료 후)** `load_level(/Game/Stages/map_battle_village)` → village의 아군 4기 액터 OverrideMaterial 조회(`ObjectTools`) → **원본 MI_Party_A1..A4 그대로(flip MI로 안 바뀜)** 확인 + village 카메라(0,-7850,90도)에서 `CaptureViewport` 저해상도 → 아군이 정면대치에서 **여전히 우향(무반전)**. + village fence/배치 불변. → **기대**: village 아군 MI 원본 유지·우향·정면구도 불변. **이 TC가 없으면 정면레벨 오염을 F까지 모른다**(오너 지시 강조점). village에서 아군이 좌향으로 보이거나 MI가 flip으로 바뀌었으면 FAIL(C1 위반, 즉시 S1 롤백). 완료 후 `load_level(octopath)`로 원복. | 심각도 **Crit**(C1 최종 방어선) | 대기 | 근원: C1

## S2 — 사선 배치 게이트 (좌표·방향·카메라·fence)

> 목적: 8기 신규 좌표 정확 배치 + 마주봄 + A안 카메라 프레이밍 + z-fight/fence 무간섭 + **정면대치 레벨 무오염**.
> 실행 순서 권장: TC-12~14(좌표·접지·유니크Y, 뷰포트 무관) → SetCamera(A안) → TC-15~20(방향·프레이밍·사선·실루엣, 뷰포트 기반) → TC-21~22(fence·z-fight) → **TC-23(village 무오염 회귀, 맨 마지막)**.

## 이월(deferred) 후보 — Director 판정용

아래는 **해당 단계에서 검증 불가하고 이후 단계에서 자연 검증되는** TC. Director가 게이트 통과 시 "이월" 표기 후 추적.

- **G1 TC-15(마주봄 최종형)**: G1 시점엔 아군 액터가 아직 사선 재배치·flip 배정 전일 수 있음(배치는 S2). G1에선 "flip MI 자체가 좌향 렌더"(TC-08)까지만 확정하고, **8기 동시 마주봄은 S2 TC-15에서 최종 검증** → G1에서 이월 가능.
- **G1 TC-M3(비대칭 어색함) / S3 TC-M-대조(명암) / F TC-M-방향부합**: 전부 **오너 주관** → 자동 게이트 PASS 대상 아님. Director가 "오너 리뷰 대기"로 이월·추적. verifier는 캡처 제공까지만.
- **S2 TC-18(축소율)·TC-20(중심)**: A안 확정 전 오너 A/B 앵글 선택이 남아있으면(B안 참고캡처 요구), 최종 수치는 **오너 앵글 확정 후 재측정** → 잠정 이월 가능.

## verifier 유의 (도구 특성 — 오판 방지)

- `get_scalar_parameter`/`get_texture_parameter`는 **상속값도 반환**한다. "오버라이드 여부"는 `list_parameters` 결과와 교차하거나, 마스터 기본값(FlipX=0)과 다른 값이 나오는지로 판단.
- `get_expression_inputs` **반환이 부정확할 수 있음**(§2 도구 주의). TC-06 삽입지점 검증은 `get_property_input`+`get_expressions`로 이중 확인하고, 애매하면 **PLAUSIBLE로 플래그**(런타임 증상은 TC-09가 잡음).
- `CaptureViewport` base64는 **디코드 없이 메타데이터만으로 수치 TC 대부분 판정 가능**(screenPosition/worldLocation/distanceCm/cameraFOV). 육안 필수 TC만 이미지 디코드 — **분할·저해상도**로 토큰 한도 회피.
- 카메라 검증은 항상 **captureTransform 명시 or SetCameraTransform 선행**. 뷰포트 상태 누수로 이전 TC의 카메라가 남아 오판하는 것 방지.
- 레벨 스위칭 TC(village 회귀)는 **각 게이트의 맨 마지막**. 끝나면 octopath 재로드 확인(`get_current_level`).
