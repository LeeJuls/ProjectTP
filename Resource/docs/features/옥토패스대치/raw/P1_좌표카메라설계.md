# P1 좌표·카메라 설계 — map_battle_octopath (수치 설계·뷰포트 검증, 액터 무이동)

> 상위: [[../청사진|옥토패스대치 청사진]] · 전 단계: [[P0_상속스냅샷|P0 상속스냅샷]]
> 작업 방식: 레벨 액터는 일절 이동하지 않았다. 뷰포트 카메라(`SetCameraTransform`)와 `WorldPosToScreenCoords`/`trace_world`만으로 좌표·카메라값을 설계·검증했다. 레벨 저장 없음.

## 0. 사전 확인 사항 (설계 전 실측)

### 0-1. 좌표계·좌우 부호 확정
`WorldPosToScreenCoords`로 기존 8기 원좌표를 카메라(0,-7850,750)/(pitch-6,yaw90,roll0)에서 투영해 실측:

| 월드 X | screen.x | 판정 |
|---|---|---|
| -600 (Party_A1 원좌표) | 0.855 | 화면 우측 |
| +600 (Enemy_B4 원좌표) | 0.141 | 화면 좌측 |
| 0 | 0.500 | 중앙 |

**확정: 월드 -X = 화면 우측(아군), 월드 +X = 화면 좌측(적).** 청사진 문구("-X=화면우측")와 정확히 일치. 기존 P0 배치의 X부호(Party 음수, Enemy 양수)가 이미 이 규칙을 따르고 있었으므로, 이번 사선 설계도 **진영별 X부호는 기존 그대로 유지**하고 사선감은 Y(진영 내 depth 인터리브)로 부여했다.

### 0-2. Z 오프셋 컨벤션 불일치 발견 (중요, 보고 필요)
P0 스냅샷의 기존 8기 Z값을 `trace_world`로 역산 검증한 결과, **기존 액터 Z는 순수 지형 히트 Z 그대로이며 +129.5 오프셋이 적용되어 있지 않다**(예: Party_A2 X=-450,Y=-7003 → trace_world 히트 Z=732.8 = 기존 액터 Z와 완전 일치). 이번 지시서는 "Z=trace_world+129.5"를 명시했으므로, **신규 8기 좌표표는 지시서 규칙(+129.5)을 따라 산출**했다. 이는 기존 컨벤션과 다른 새 기준의 시작이므로 S2 구현 시 이 차이를 인지해야 한다(기존 8기와 신규 8기가 같은 규칙이 아님).

### 0-3. 스프라이트 액터 실측 치수 (겹침 판정 근거)
`get_actor_bounds`(Party_A1, 원위치): X범위 -924~-276(폭648cm, 피벗기준 좌우 ±324 대칭), Z범위 631.4~890.4(높이259cm).
쿼드(SM_SpriteQuad) 전체 폭은 648cm이나, 이는 배경 여백을 포함한 메시 전체 폭이다. 쿼드 경계를 직접 투영해 겹침을 판정해본 결과 **정면대치 원본(P0 기존배치)에서도 인접 캐릭터의 쿼드 경계는 이미 크게 겹쳐 있었다**(B3/B4 원좌표 쿼드 투영 시 겹침폭 0.296, 화면비). 알파마스크 처리로 실제 그림(실루엣)만 안 겹치면 시각적으로 문제없는 기존 설계 방식으로 판단되며, **쿼드 전체 경계 겹침은 신규 설계의 결함 판정 기준으로 부적절**하다. 따라서 본 문서의 겹침 판정은 "인접 캐릭터 피벗의 스크린x 간격을 정면대치 기준선(약 0.088~0.090, 아래 0-4 참조) 대비 상대압축률로 비교"하는 방식을 채택했다.

### 0-4. 정면대치 기준선 (겹침 판정 기준값)
카메라(0,-7850,750)/(pitch-6,yaw90,roll0)에서 기존 8기 원좌표(X스텝150, Y거의 일렬) 투영 시 진영 내 인접 스크린x 간격은 **균일하게 약 0.088~0.090**이었다. 이를 "정상 간격" 기준선으로 채택.

## 1. 8기 제안 좌표표

### 1-1. 배치 원칙
- 진영별 X부호는 기존 유지: 아군(Party) X ∈ {-600,-450,-300,-150}, 적(Enemy) X ∈ {150,300,450,600}. X스텝 150cm(>149cm, 실루엣 미가림 기준 충족).
- **V자 사선 방향**: 각 진영에서 중앙에 가까운(|X| 작음) 캐릭터일수록 카메라에 가깝게(앞열), 바깥쪽(|X| 큼)일수록 멀게(뒤열) 배치. 즉 아군 A4(X=-150,최안쪽)=최앞 / A1(X=-600,최바깥)=최뒤, 적 B1(X=150,최안쪽)=최앞 / B4(X=600,최바깥)=최뒤. 두 진영이 중앙에서 앞으로 튀어나오며 마주보는 ▽(V자) 구도 — 옥토패스 레퍼런스(중앙 캐릭터가 카메라 최근접)와 부합.
- **Y 깊이(depth) 설계**: 카메라-앞열거리 Dn≈850cm 기준, 뒤열축소 25~30% DoD 역산 → Df(카메라-뒤열거리) 허용범위 850<Df≤1133~1214cm → Y스팬(앞~뒤) 허용 ≤283~364cm. 보수적으로 총 300cm 채택(4단 균등 100cm/단): 앞열기준 Y=-7000 ~ 뒤열기준 Y=-6700.
- **전역 유니크 Y 인터리브**: 같은 "단"의 아군/적을 서로 ±5~10cm 어긋나게 배치해 8개 Y값이 전부 유니크하고 인접 Y차 ≥3cm(H3, z-fight 방지) 되도록 함.

## 2. 카메라 A/B 후보 설계·검증

### 2-1. 설계 과정에서 발견한 핵심 트레이드오프 (반드시 읽을 것)

오블리크 카메라를 "yaw만 틀고 카메라 위치를 한쪽으로 옮기는" 방식으로 설계하면, 카메라에서 멀어지는 쪽 진영(카메라를 아군 쪽으로 옮기면 적 쪽)의 원근압축이 반대쪽보다 훨씬 심해지는 근본적 비대칭이 발생한다. 이를 완화하려 카메라를 후퇴(Y 증가)시키면 비대칭은 개선되지만 **원근감(뒤열축소율) 자체가 함께 죽어버려 축소율 DoD(25~30%)를 잃는다.** 즉:
- **진영간 실루엣 간격 대칭성**과 **뒤열축소율 25~30%**는 카메라 위치·각도만으로 동시에 최적화하기 어려운 트레이드오프 관계다.
- 이 트레이드오프의 기울기는 **오프축 각도가 클수록 가팔라진다**(12°는 관리 가능, 25°는 거의 불가능한 수준으로 실측됨).

이 사실은 다수의 카메라 위치/거리 조합(약 15회 반복 실측, 계산 과정은 작업 스크래치에 보존)을 시도해 확정한 것이며, 카메라 배치 실수가 아니라 **이 스트립 치수(X±600cm, Y스팬300cm)와 12°/25° 각도 조합의 구조적 특성**이다.

### 2-2. A안 (보수적 사선, yaw 오프축)

**중요: 청사진 원 스펙은 "yaw 오프축 ~12°"였으나, 실측 결과 12°(yaw=78/102)로는 두 DoD(실루엣미겹침+축소율25~30%)를 동시 만족하는 조합을 찾지 못했다. 아래 상세 비교 후 6°(yaw=84)로 완화한 값을 A안 대표로 채택했다 — 이 각도 완화는 Director/오너의 명시적 확인이 필요한 사항이다.**

**12° 원스펙 시도 결과(참고, 채택 안 함)**:
- 카메라(X=-270,Y=-9500,pitch-3,yaw78, 1650cm 후퇴)로 실루엣 간격을 개선(최소간격 0.020, 기준대비22%)하는 데는 성공했으나, 그 대가로 **뒤열축소율이 6.8%까지 떨어져(DoD 25~30% 대비 크게 미달) 원근감이 거의 사라짐**. 후퇴를 줄이면 축소율은 회복되지만 실루엣 간격이 다시 기준대비 27~41%로 악화(X=-300,Y=-7850,yaw78 조합).

## 4. 액터·레벨 상태 확인

- 레벨 8기 스프라이트 액터, fence 7개 등 **어떤 실제 액터도 이동·수정하지 않았다.** 사용한 도구는 `SetCameraTransform`(뷰포트 카메라, 레벨 데이터 아님), `WorldPosToScreenCoords`, `trace_world`뿐이다.
