## 핵심 팩트 요약 (4줄)

1. **"의상 일체형"이 맞다** — 상의/하의를 따로 조합하는 구조가 아니라 `cloth` 17종 중 하나를 통째로 고르는 단일 슬롯. `_bot`/`_top`은 신체 부위가 아니라 렌더링 순서(뒤/앞) 구분이며 hair·cloth·weapon 3개 카테고리에 공통 적용.
2. **17행은 IDLE1/2·RUN1/2·JUMP·ATK1/2/3·AIRATK1/2·CASTING1/2·HURT·DYING·DASH·BLOCK·ROLL** — 원본 가이드 이미지(`frameguide_v2.png`) 라벨과 761개 PNG 알파 전수스캔이 정확히 일치.
3. **공격 계열은 1개가 아니라 7개 행(ATK1~3, AIRATK1~2, CASTING1~2)이고, 5개 무기 전부가 17행 전체에 그림을 갖고 있다** — 무기가 "쓸 수 있는 행"을 제한하지 않는다. 대신 기존 데이터(`skills.csv`)는 **"무기" 단위가 아니라 "스킬" 단위**로 행 1개를 매핑하는 구조(`MotionRow` 컬럼)를 이미 갖고 있다.
4. **현재 코드에서 행을 실제로 전환하는 유일한 경로(`PlayAttack`)는 무기·스킬과 무관하게 RowIndex=5(ATK1) 고정값**이다 — `skills.csv`의 `MotionRow` 데이터와 아직 배선되어 있지 않다. 또한 현재 유일한 무기 배정 사례(`compose_party.py` 8종)는 캐릭터 컨셉과 무기가 뒤섞여 있음이 확인됨(질문3-6).

## 질문 1 — 파츠 카테고리 정확 목록

### 1-1. "머리/상의/하의" 구조가 아니라 "의상 일체형"이다

heroes99에는 상의·하의를 따로 갈아입는 슬롯이 **없다**. 장비 카테고리는 **skin·face·hair·cloth·weapon 5개뿐**이고, `cloth`는 17종의 **완전한 한 벌 의상**(스타일) 중 하나를 고르는 단일 슬롯이다(예: `cloth15`=갑옷 한 벌, `cloth7`=로브 한 벌 — 내부에서 상의/하의가 다시 나뉘지 않음).

파일명의 `_bot`/`_top` 접미사는 **신체 부위(하의/상의)가 아니라 렌더링 순서(z-order)** 구분이다 — hair·cloth·weapon 3개 카테고리 모두 동일 규칙:
- `_bot` = skin(몸통) 레이어보다 **먼저**(몸 뒤쪽) 그려지는 서브레이어
- `_top` = face 레이어보다 **나중**(몸 앞쪽) 그려지는 서브레이어

**근거**: `compose_party.py` 9행 합성순서 주석 `hair_bot -> weapon_bot -> cloth_bot -> skin -> face -> cloth_top -> hair_top -> weapon_top` — `skin`이 정확히 중간에 위치하고 `cloth_bot`은 그 앞, `cloth_top`은 `face` 뒤. 이 구조상 "하의 없이 상의만" 같은 조합은 애초에 존재할 수 없다(bot/top은 같은 옷 한 벌의 앞/뒷면 레이어일 뿐, 별개 장비 슬롯이 아님).

**방증(교차검증 2건 일치)**: cloth 17종 중 11종(`cloth1,2,3,4,5,6,9,10,11,16,17`)은 `_top` 레이어가 완전히 비어 있음(파츠_인벤토리.md §2-2, Pillow 알파 전수스캔) — "하의가 없다"가 아니라 "몸 앞으로 튀어나오는 부분(망토 앞자락 등)이 디자인에 없는 옷"이라는 뜻. **`Resource/data/parts.csv`에서도 정확히 같은 6종(`cloth7,8,12,13,14,15`)만 `AssetPathTop` 컬럼이 채워져 있고 나머지 11종은 공란**임을 직접 대조로 확인 — Pillow 알파스캔과 CSV 데이터 저작, 서로 다른 두 출처가 완전히 일치.

### 1-3. 색상변형(`_c<N>`) 구조

- skin/face: 색상변형만 있고 bot/top 없음(`skin_c1.png`처럼 단일 파일).

`parts.csv`의 `#id_txt` 컬럼이 weapon4·5에 한해 이미 실명을 부여했다: `part_weapon_spear`(weapon4), `part_weapon_wand_c1~c4`(weapon5). weapon1~3은 `part_weapon_w1`/`w2`/`w3`로 **번호만 있고 이름이 없다**. 파츠_인벤토리.md §1-3의 실측(bbox·형태 비교)도 weapon4=SPEAR·weapon5=WAND만 "확정"(프로모 이미지 `weapon_types.gif`와 실루엣 일치), weapon1~3(추정 SWORD/AXE/DAGGER)은 "추정"으로 남겨뒀던 것과 정확히 일치한다. 즉 **게임 데이터 레이어와 아트 실측 레이어가 서로 독립적으로 같은 결론(spear·wand만 확정, 나머지 3종 미확정)에 도달**했다(본 조사, 두 출처 대조).

### 1-5. JobTags·GenderTag 컬럼 사용 현황

`parts.csv`는 `GenderTag`·`JobTags` 두 컬럼을 스키마로 갖고 있으나, 실제 값이 채워진 건 `GenderTag`뿐이고(hair 230행에만, `90200100`=남/`90200200`=여, `genders.csv` 참조) **`JobTags`는 387행 전부 공란**이다(본 조사, 전 카테고리 확인). 즉 "이 파츠는 이 직업 전용"이라는 데이터 레벨 연결은 현재 하나도 없다.

## 질문 3 — 공격 모션과 무기의 관계 (핵심)

### 3-0. 정의

"공격 계열" 행 = **7개**: ATK1(5)·ATK2(6)·ATK3(7)·AIRATK1(8)·AIRATK2(9)·CASTING1(10)·CASTING2(11).

### 3-1. FACT — 무기 종류가 "행 사용 가능 여부"를 제한하지 않는다 (가설 (b) 기각)

5개 무기 전부가 17행 **전부**(공격계열 7행 포함)에 그림을 갖고 있다(weap_ok=5/5, `파츠_인벤토리.md` §3 표, 재현 스크립트 `inventory_row_coverage.py`). "이 무기는 CASTING 행이 없다"거나 "이 무기는 ATK3가 없다" 같은 **구조적 제한은 파일 레벨에 존재하지 않는다** — 어떤 무기를 골라도 17행 전체를 기술적으로 재생할 수 있다.

### 3-2. FACT — 같은 무기 안에도 서로 다른 공격 모션이 이미 여러 개 있다 (가설 (c) 확인)

한 캐릭터(skin_c1+face_c1+hair m5_c1)에 weapon1(추정 SWORD)과 weapon5(확정 WAND)만 바꿔 끼우고 IDLE1/ATK1/ATK2/ATK3/AIRATK1/AIRATK2/CAST1/CAST2 8행을 실제로 재합성해 육안 비교(본 조사, 임시 스크립트 — 세션 스크래치패드에만 출력, 원본·Resource 무변경):

- **weapon1(SWORD 추정)**: ATK1은 전방 하향 슬래시(흰 궤적이 3~4번째 프레임에서 크게 휘두름), ATK2는 궤적 모양이 다른 상향/훅성 스윙, ATK3은 궤적 이펙트 없이 짧은 4프레임 동작 — **3개가 육안으로 서로 다른 스윙**.
- **weapon5(WAND 확정)**: 마찬가지로 ATK1/ATK2/ATK3 궤적(빨간 궤적) 모양이 행마다 다름 — weapon1과 같은 패턴(행마다 별개 동작).
- 즉 **한 무기당 최소 7종(ATK1/2/3 + AIRATK1/2 + CAST1/2)의 서로 구별되는 동작이 이미 리소스에 존재**한다.

### 3-3. FACT — "행"과 "무기 그림"은 서로 다른 두 개의 독립된 축으로 관찰된다

같은 행(예: ATK1)이라도 무기마다 그려지는 그림 자체는 다르다(weapon1은 검 모양 궤적, weapon5는 완드+빨간 궤적). 이건 "무기별 전용 행이 따로 있다"가 아니라 **"행 = 포즈/타이밍 카테고리(무기 무관, 공용 17종) × 무기 텍스처 = 그 포즈에서 손에 들리는 모양(무기별로 상이)"**의 2축 구조로 관찰된다. 질문에 제시된 (a)(전 무기 공통 1개 행)·(b)(무기별 전용 행) 어느 쪽도 정확한 서술이 아니고, 두 축이 독립적으로 겹쳐지는 구조에 가깝다.

### 3-4. FACT — CASTING 행은 무기와 무관하게 거의 동일한 "손 들어올리기" 동작이다

weapon1(검)과 weapon5(완드) 둘 다 CAST1/CAST2에서 유사한 팔 동작(한 손을 들어올리는 자세)을 보이고, 무기 고유의 마법 이펙트(빛·파티클 등)는 관찰되지 않았다(본 조사, 육안 비교) — 검을 든 캐릭터도 CAST1 재생 시 "검을 든 채 손을 드는" 그림이 정상적으로 존재한다(그래픽은 있으나 검의 존재를 정당화하는 별도 연출은 없음).

### 3-5. FACT — 기존 데이터 레이어는 "무기" 단위가 아니라 "스킬" 단위로 행을 매핑한다

`D:\unreal\Resource\data\skills.csv`에 이미 `MotionRow` 컬럼이 존재하며 현재 정의된 스킬 5개가 이렇게 매핑돼 있다(본 조사):

| SkillId(#id_txt) | Kind | MotionRow | 대응 행 |
|---|---|---|---|

## 질문 4 — 현행 파이프라인의 모션 처리

### 4-1. `compose_party.py` — 행 슬라이싱을 하지 않는다

`compose_party.py`는 캐릭터 1명당 **800×680 캔버스 전체(17행 전부)를 합성**해 `T_Party_{ID}.png` 1장으로 저장한다(`CANVAS_SIZE=(800,680)`, 8레이어를 `Image.alpha_composite`로 전체 캔버스에 겹침, 스크립트 123~139행). 특정 행만 잘라내거나 선택하는 로직이 스크립트 내에 **없다** — "어느 모션을 쓸지"는 이 스크립트의 책임 범위 밖이고, 출력은 항상 17행이 다 들어있는 풀시트다.

### 4-2. 행 선택은 UE 머티리얼의 런타임 파라미터가 담당한다(SubUV 방식)

`M_Sprite_Flipbook_Lit`(및 파생 `M_Sprite_PartComposite`) 머티리얼이 `RowIndex`(스칼라, 기본값 0=IDLE1)·`FrameCount`(행마다 다름, 기본값 6)·`GridX=8`·`GridY=17`·`FPS=8`·`TimeOffset`·`FlipX` 파라미터로 UV를 계산해 텍스처의 특정 셀만 샘플링한다(`기본전투무대/raw/S2_임포트머티리얼.md` §2, `A0_합성머티리얼스파이크.md` §8-1 실측). 데모용 MI 8개(`MI_Party_A1~A4`, `MI_Enemy_B1~B4`)는 이 SubUV 파라미터를 전부 오버라이드하지 않고 마스터 기본값을 상속 — **정지 상태에서는 8개 전부 IDLE1(row0)**(`S2_임포트머티리얼.md` §3).

**주의**: 머티리얼 이름과 art-pipeline 직무기술서의 "플립북"이라는 표현과 달리, 실제 구현은 **Paper2D `PaperFlipbook` 애셋이 아니라 `StaticMeshComponent`(`SM_SpriteQuad`) + 커스텀 SubUV 머티리얼**이다(`S2_임포트머티리얼.md` §4 — `SM_SpriteQuad`는 로컬 XY평면 쿼드라 `roll=90` 배치가 필요했다는 실측 기록; `공격버튼데모/raw/D2_구현.md`의 `GetSprite`가 가리키는 컴포넌트도 StaticMeshComponent 계열). 프레임 전환은 텍스처를 여러 장 바꿔 끼우는 방식이 아니라 **하나의 풀시트 텍스처 위에서 UV 좌표만 셰이더에서 이동**시키는 방식이다.

### 4-3. 현재 실제로 행을 전환하는 유일한 코드 경로 — 무기·스킬과 무관한 하드코딩

`BP_BattleSpawnPoint`의 커스텀 이벤트 `PlayAttack`(`공격버튼데모/raw/D2_구현.md`)이 유일하게 런타임에 RowIndex를 바꾸는 코드다:

```
PlayAttack → SetScalar(TimeOffset=GetGameTimeInSeconds) → SetScalar(FrameCount=6.0) → SetScalar(RowIndex=5.0)
  → RetriggerableDelay(0.70초, 핫픽스 이전 0.95) → SetScalar(TimeOffset=0) → SetScalar(FrameCount=6.0) → SetScalar(RowIndex=0.0)
```

`RowIndex=5.0`(ATK1)은 **어느 스폰포인트(=어느 캐릭터·어느 무기)에 있는 인스턴스든 동일하게 적용되는 고정값**이다 — 무기 종류나 스킬 종류에 따라 다른 RowIndex를 대입하는 분기 로직은 현재 이 Blueprint에 없다. `skills.csv`의 `MotionRow` 컬럼(질문3-5)과 이름이 개념적으로 대응되긴 하지만, **`PlayAttack`이 `skills.csv`를 실제로 읽어서 RowIndex를 대입하는 배선은 아직 없다** — 데이터 레이어(계획)와 코드 레이어(구현)가 아직 연결되지 않은 상태다.

### 4-4. 핫픽스 이력 — 가이드 표기와 실측치가 실제로 어긋난 전례가 있다

`frameguide_v2.png`는 "ATTACK 1"을 6프레임으로 표기하고(질문2 표, 재확인) 최초 구현 시에도 `FrameCount=8.0`으로 설정했다가 마지막 2칸이 완전 투명(빈 프레임)이라 깜빡임이 발생 — 오너 실측으로 6프레임임을 확정하고 `FrameCount=6.0`/`Duration=0.70`으로 수정한 이력이 있다(`공격버튼데모/raw/D2_구현.md` 말미 "핫픽스" 섹션). 즉 **가이드 이미지·초기 구현값을 그대로 믿지 않고 실제 알파 데이터로 재확인해야 한다**는 원칙이 이미 한 번 실증된 바 있다.
