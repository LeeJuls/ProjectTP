## 1. 문제의 구조 — 왜 "2축"이 핵심인가 (F4)

E0가 실측으로 확정한 가장 중요한 사실: **모션행과 무기그림은 서로 곱해지는 독립 2축**이다.

```
  최종 렌더 프레임 =  [모션행 = 포즈/타이밍]  ×  [무기텍스처 = 손에 들린 그림]
                      (17종, 무기무관 공용)      (무기 슬롯이 결정, 코스메틱)
                            ↑ 축1                      ↑ 축2
```

- **축1(모션행)**을 누가 정하는가 = "모션 소유권"(결정축 1).
- **축2(무기텍스처)**가 축1·스킬·포지션에 얽매이는가 = "무기 제약"(결정축 2).
- 두 데이터는 **이미 둘 다 존재**한다: 축1은 `skills.csv.MotionRow`(F6), 축2는 `parts.csv` 무기 슬롯(F8). 규칙안은 이 둘의 **연결 방식**을 정하는 것이지 새 데이터 축을 발명하는 게 아니다.

이 2축 관찰에서 자연히 두 갈래가 나온다:
- **축1을 스킬이 소유**(F6 현행 구조 유지) → 무기는 축2 코스메틱. → **안 A·B**
- **축1을 무기가 소유**(무기가 포즈 세트를 규정) → 스킬은 무기에 종속. → **안 C**

## 3. 공통 인프라 — 어느 안을 골라도 필요한 작업

아래 두 가지는 **안 선택과 무관하게 A1 전투완성에서 반드시 해야 하는 공통 구현**이다. 여기서 먼저 못박아 각 안의 비교에서 제외한다.

### 3-1. RowIndex↔MotionRow 배선 (결정축 ④ — A1로 이관)

F7 사실: 현재 `PlayAttack`은 `RowIndex=5.0`·`FrameCount=6.0`을 **어느 캐릭터·어느 스킬이든 고정** 대입한다. `skills.csv.MotionRow`(F6)는 존재하지만 **읽어서 대입하는 코드가 없다.** 어느 안이든 이 배선은 필요하다:

```
[현행 A1 필요작업 — gameplay-engineer 소관]
PlayAttack(SkillId):
  1. skills DataTable에서 SkillId → MotionRow 조회        # 안 C는 여기서 무기+스킬로 행 해석
  2. motions DataTable(§6-1)에서 MotionRow → FrameCount 조회  # ★행마다 프레임수 다름
  3. SetScalar(RowIndex = MotionRow)
     SetScalar(FrameCount = <그 행의 프레임수>)             # 하드코딩 6 제거
     SetScalar(TimeOffset = GetGameTimeInSeconds)
  4. RetriggerableDelay(FrameCount/FPS) → RowIndex=0(IDLE1)·FrameCount=6 복귀
```

- **왜 FrameCount도 배선해야 하나**: 행마다 유효 프레임수가 다르다(ATK1=6, ATK3=4, CAST1=5…). 현재 `FrameCount=6` 하드코딩은 **ATK1에만 맞다.** ATK3(4프레임)를 6으로 재생하면 마지막 2칸 빈 프레임 깜빡임 — E0 §4-4에 **이미 한 번 터진 버그**(ATK1을 8로 재생→투명칸 깜빡임→6으로 핫픽스). 행별 FrameCount는 데이터로 관리해야 재발 방지.
- **결론**: MotionRow→FrameCount 참조 테이블(§6-1 `DT_Motions`)이 공통으로 필요. 이건 밸런스 데이터가 아니라 **스프라이트시트의 기술 사실**(행별 프레임수)이라 스킬마다 중복 저장하면 안 됨(이중진실 금지, [[데이터_서버_규약]]§4).

### 3-2. 30원자 MotionRow 컬럼 (스킬원자 구조와의 정합, 결정축 ①)

방향성2의 30원자(평타10·공용10·직업10)는 전부 `skills.csv`의 행이 된다(G1·G5). **각 원자는 자기 MotionRow를 컬럼으로 갖는다** — 이미 5스킬이 그렇게 돼 있다(F6). 즉 "스킬원자 = 데이터"라는 방향성2 철학과 "MotionRow = 스킬 컬럼"은 **이미 정합**한다. 안 A·B는 이 구조를 그대로 쓰고, 안 C만 이걸 깬다(무기로 소유권 이전).

### 안 B — 느슨한 결합 (스킬 소유 모션 + 무기 외형이 포지션·성향을 생성단계에서 느슨히 준수) 【권장】

**한 줄**: 엔진·전투·배선은 안 A와 **완전 동일**. 단 **1000풀 생성 시** 무기 외형을 캐릭터의 포지션·스킬성향에 **느슨히 맞춰** 뽑는다(기본 준수 + 의도적 예외율). = G3(오너 확정 "느슨한 매핑")을 그대로 구현한 것.

| 항목 | 내용 |
|---|---|
| ① 모션 소유권 | **스킬**(안 A와 동일). 무기는 여전히 축2 코스메틱 — **모션·데미지에 안 닿음.** |
| 평타10 소비 | 안 A와 동일(물리성향→ATK행, 마법성향→CASTING행 분산). 추가로: 평타 원자에 **성향 태그**(PHYS/MAGIC)를 달아 생성단계의 무기 매칭 힌트로 씀(§6-2). |
| ② 무기 제약 | **느슨.** parts.csv 무기행에 **성향/포지션 친화 태그**(§6-3) 부여 → 생성 시 그 캐릭터 포지션·스킬성향에 맞는 무기를 **우선** 뽑되, 정해진 **예외율**(예: 15~20%)로 어긋난 무기도 허용. **런타임 제약 아님**(전투 중엔 무기가 아무 행이나 재생 가능, F3 유지) — 오직 "어느 무기 스프라이트를 그리느냐"의 생성단계 상관관계. |
