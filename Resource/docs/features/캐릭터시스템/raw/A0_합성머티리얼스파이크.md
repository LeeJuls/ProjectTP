## 최종 판정 (한 줄)

**"완전 성공/부분 성공/완전 실패" 3분류 중 어디에도 속하지 않는다 — UE 절차(①신규 머티리얼 생성 ~ ⑦검증) 7단계 중 단 하나도 실행하지 못했다.** MCP 최초 조회(`describe_toolset`/`list_toolsets`)까지는 정상 응답했으나, `MaterialTools.get_expressions`로 기존 `M_Sprite_Flipbook_Lit` 그래프를 조회하려는 시점부터 응답이 멎었고 이후 모든 `unreal-mcp` 호출이 timeout → "not connected"로 전환됐다. 이것은 "합성 머티리얼 기법이 안 된다"는 기술적 결론이 아니라 **도구 접속이 끊겨 시도 자체를 못 한 상태**다. UE 밖에서 가능한 사전작업(파츠 5레이어 텍스처 평탄화)은 완료·육안 검증했다.

## 1. 배경 · 목표

`알파_개발계획.md` §1 S2("렌더 조합 — 레시피→외형, 합성 머티리얼")의 A0 스파이크. 알파의 캐릭터 랜덤 조합(이론상 1,051만 조합, 파츠_인벤토리.md §2-5)은 사전 Pillow 합성이 불가능해 **런타임 머티리얼 합성**이 유일한 경로다. 목표: 5개 텍스처 파라미터(SkinTex/FaceTex/HairTex/ClothTex/WeaponTex)를 알파오버로 겹치면서, 기존 `M_Sprite_Flipbook_Lit`의 SubUV 프레임 애니메이션(GridX/GridY/RowIndex/FrameCount/FPS/FlipX/TimeOffset)과 동기화되는 마스터 머티리얼 `M_Sprite_PartComposite`가 성립하는지 실증.

### 알려진 단순화(지시서 명시)
heroes99 원본은 hair/cloth/weapon 3개 카테고리가 각각 `_bot`/`_top` 서브레이어로 나뉘어 있어 프로덕션 합성 순서는 8레이어(`hair_bot→weapon_bot→cloth_bot→skin→face→cloth_top→hair_top→weapon_top`, `compose_test_combo.py` 기준)다. 이번 스파이크는 지시서가 명시적으로 5개 파라미터만 요구하므로, hair/cloth/weapon 각각의 bot+top을 **카테고리 내부에서만** 먼저 Pillow로 평탄화해 "카테고리당 1장" 텍스처로 만들었다(§2 참조). 카테고리 간 인터리빙(예: hair_bot이 face보다 뒤, hair_top이 face보다 앞)은 이 5레이어 구조로는 재현되지 않는다 — 기술 스파이크의 목적("N장 텍스처 알파오버 + SubUV 동기화가 되는가")에는 영향 없음.

## 3. UE 절차 (미실행 — MCP 차단)

### 3-1. 실행 시도한 것

1. `list_toolsets` — 성공(초기, MCP 정상 응답 확인).
2. `describe_toolset` ×4 (MaterialTools/MaterialInstanceTools/TextureTools/AssetTools) — 성공, 스키마 확보.
3. `AssetTools.find_assets(folder_path="/Game", name="M_Sprite_Flipbook_Lit")` — 첫 시도는 `toolset_name` 누락으로 client-side 에러("Tool not found"), 재시도(toolset_name 명시)는 **성공**: `/Game/Materials/M_Sprite_Flipbook_Lit` 확인.
4. `MaterialInstanceTools.list_parameters(material={refPath:"/Game/Materials/M_Sprite_Flipbook_Lit"})` — 실패: `"...is not a valid object path for property 'material'"`(refPath 형식이 `/Game/Materials/M_Sprite_Flipbook_Lit.M_Sprite_Flipbook_Lit`처럼 패키지+오브젝트명 dot 표기가 필요한 것으로 추정, 이 자체는 사소한 파라미터 형식 문제).
5. `MaterialTools.get_expressions(material_or_function={refPath:"/Game/Materials/M_Sprite_Flipbook_Lit"})` — **여기서 timeout**. 이후 재시도(`list_parameters` dot표기 수정판, `SceneTools.get_current_level`)도 전부 timeout → 결국 `"MCP server unreal-mcp is not connected"`로 고착.

**4~7단계(신규 머티리얼 생성, 5파라미터+블렌드체인 구성, recompile, MI 생성·바인딩, 검증) 전부 착수하지 못했다** — UE에 어떤 애셋도 생성되지 않았고 기존 애셋도 전혀 변조하지 않았다(조회만 시도, 전부 실패).

| 확인 항목 | 방법 | 결과 |
|---|---|---|

| 엔진 로그 상관관계 | `projectTP.log` (`LogModelContextProtocol`) tail 확인 | **§핵심발견 3번 참조** — 마지막 dispatch가 A0③ 세션의 `BlueprintTools.create(UserDefinedStruct)`이고 그 직후 로그가 완전히 멈춤(라인 4994 = 파일 끝) |

**결론(정황 근거, 확정 아님)**: `BlueprintTools.create`를 `asset_type=/Script/CoreUObject.UserDefinedStruct`로 호출하는 것이 게임 스레드의 MCP 처리 큐를 멈춰세우는 트리거일 가능성이 가장 유력하다. 이 스파이크(A0①)의 `get_expressions` 호출은 그 직후(같은 서버, 다른 세션)에 도착했으나 큐가 멈춘 탓에 dispatch조차 로그에 남지 못하고 timeout된 것으로 보인다. 이 서버는 **동시 세션 처리에 안전하지 않을 가능성**이 있다(단일 처리 큐로 추정) — [[A0_CSV파이프라인스파이크]]가 제기한 "동시 세션 충돌" 가설을 이번 조사의 정밀한 로그 라인 대조로 한 단계 더 구체화함(누가·어떤 호출이 정지 시점과 정확히 일치하는지까지 특정).

이 시점에서 프로젝트 규칙(3회 실패 시 중단)에 따라 재시도를 중단했다(list_toolsets/describe_toolset/get_current_level 등 총 10회 이상, 90초 bounded 폴링 1회 포함 약 10분 이상에 걸쳐 시도).

## 4. 그래프 구조 설계안 (미실행 — 재개 시 참고용)

MCP가 막혀 실제로 만들지 못했지만, 재개 시 바로 실행할 수 있도록 설계를 남긴다. **재개 첫 작업은 이 설계안을 그대로 만드는 것이 아니라, `MaterialTools.get_expressions`/`get_property_input`로 `M_Sprite_Flipbook_Lit`을 먼저 조회해 아래 가정(특히 BaseColor vs EmissiveColor 연결, OpacityMaskClipValue, FlipX 구현 방식)을 검증·보정하는 것.**

- **파라미터 그룹 A(텍스처, 5개)**: `SkinTex`/`FaceTex`/`HairTex`/`ClothTex`/`WeaponTex` — `TextureSampleParameter2D` 5개, 전부 같은 계산된 UV를 입력받음(아래 그룹 C).
- **파라미터 그룹 B(SubUV 스칼라, 기존 마스터와 동일 이름 재사용 가정)**: `GridX`(기본8) / `GridY`(기본17) / `RowIndex`(기본0) / `FrameCount`(기본6=IDLE1) / `FPS` / `FlipX` / `TimeOffset` — 기존 `M_Sprite_Flipbook_Lit`과 **이름을 동일하게** 만들어야 "5레이어가 같은 프레임으로 동기화"라는 판정기준을 충족.
- **그룹 C: 공유 UV 계산** (MaterialFunction `MF_SpriteGridUV`로 분리 권장 — 지시서 권고사항):
  - `FrameIndex = floor((Time*FPS + TimeOffset)) % FrameCount`
  - `ColU = (FrameIndex % GridX) / GridX`, FlipX 시 셀 내부 U 미러
  - `RowV = RowIndex / GridY`
  - `FinalUV = TexCoord0 * (1/GridX, 1/GridY) + (ColU, RowV)`
  - 이 한 번의 계산 결과를 5개 TextureSampleParameter2D의 UV 입력에 전부 연결 → "RowIndex/TimeOffset을 바꾸면 5레이어 전부 같은 프레임으로 동기화"의 실제 구현 지점.
- **그룹 D: 알파오버 블렌드 체인**(지시서 순서 skin→face→hair→cloth→weapon, 뒤가 앞을 덮음):
  - `C0 = SkinTex.RGB`
  - `C1 = Lerp(C0, FaceTex.RGB, FaceTex.A)`
  - `C2 = Lerp(C1, HairTex.RGB, HairTex.A)`
  - `C3 = Lerp(C2, ClothTex.RGB, ClothTex.A)`
  - `C4 = Lerp(C3, WeaponTex.RGB, WeaponTex.A)` → `MP_BaseColor`(또는 기존 마스터가 언릿 방식이면 `MP_EmissiveColor` — 재개 시 `get_property_input`으로 확정)
  - `OpacitySum = Max(SkinTex.A, Max(FaceTex.A, Max(HairTex.A, Max(ClothTex.A, WeaponTex.A))))` → `MP_OpacityMask`(5레이어 중 하나라도 불투명이면 픽셀 표시)
  - Blend Mode = Masked, TwoSided = true(기존 마스터와 동일 가정, 재개 시 확인).
- **MI 계획**: `MI_A0_PartComposite_Test1`(parent=`M_Sprite_PartComposite`), 5개 텍스처 파라미터를 §2에서 만든 `T_A0_*` 5장에 바인딩, `RowIndex=0/FrameCount=6`(IDLE1)로 시작.

## 5. 실패 시 대안 판단 (조건부 — 아직 트리거되지 않음)

- **RenderTarget 굽기**: 조합 확정 시 1회성으로 5레이어를 RenderTarget에 구워 단일 텍스처화 — 프레임 애니메이션과는 별개로 "정적 합성"에는 유효하나, RowIndex/TimeOffset이 바뀔 때마다 다시 구워야 해 SubUV 애니메이션과는 근본적으로 상충(굽기 타이밍 설계가 추가로 필요).

이 판단은 **A0①이 실제로 그래프를 만들어보고 컴파일 실패 등 진짜 기술적 벽에 부딪혔을 때** 다시 꺼내는 것이 맞다.

## 8. 재개 실행 결과 (완전성공)

**최종 판정 (한 줄): 완전성공 — 5레이어(Skin/Face/Hair/Cloth/Weapon) 알파오버 합성 + SubUV 애니메이션 동기화가 모두 실제 UE 애셋(머티리얼 컴파일 통과 + MI 파라미터 재조회)으로 실증됨.** MCP는 이번 세션 전체(약 60여 회 호출)에서 단 한 번도 멎지 않았다 — 타임아웃·"not connected" 0건.

### 8-1. 기존 마스터 `M_Sprite_Flipbook_Lit` 그래프 리버스엔지니어링 (재개 첫 작업, §4 가정 검증)

`get_expressions`(28개 노드) → `get_property_input`(BaseColor/OpacityMask/EmissiveColor) → `get_expression_inputs`를 노드별로 역추적 → `get_properties`(각 ScalarParameter의 `parameterName`/`defaultValue`)로 그래프 전체를 재구성했다. **§4 가정과 거의 일치했으나 1가지 부호 차이를 발견**: TimeOffset은 Add가 아니라 **Subtract**(`Time - TimeOffset`)로 연결되어 있었다(기능적으로는 동등, 부호만 반대) — 새 머티리얼은 실측대로 Subtract로 복제.

확정된 파라미터 실명·기본값(8개 스칼라 + 1개 텍스처):

| 파라미터 | 기본값 | 역할 |
|---|---|---|
| `GridX` | 8 | 열 개수 |
| `GridY` | 17 | 행 개수 |
| `RowIndex` | 0 | 현재 애니메이션 행 |
| `FrameCount` | 6 | 현재 행의 프레임 수(IDLE1) |
| `FPS` | 8 | 재생 속도 |
| `FlipX` | 0 | 좌우 반전(0/1) |
| `TimeOffset` | 0 | 재생 시작 오프셋 |
| `EmissiveBoost` | 0 | BaseColor×이 값 = EmissiveColor(기본 0=발광 없음) |
| `SpriteTex` | – | TextureSampleParameter2D |

UV 계산 수식(실측, `MaterialExpression` 체인 순서대로):
```
FrameIndex = Floor((Time - TimeOffset) * FPS) % FrameCount % GridX   // Fmod 2단 체인
LocalU     = TexCoord0.R  (ComponentMask R만 true)
LocalU'    = Lerp(LocalU, 1-LocalU, FlipX)                            // 좌우반전
U          = (LocalU' + FrameIndex) / GridX
LocalV     = TexCoord0.G  (ComponentMask G만 true)
V          = (LocalV + RowIndex) / GridY
FinalUV    = (U, V)   // AppendVector, 이 한 노드를 SpriteTex 등 모든 텍스처 샘플의 UV로 공유
```
출력: `BaseColor = SpriteTex.RGB`, `OpacityMask = SpriteTex.A`, `EmissiveColor = SpriteTex.RGB × EmissiveBoost`. 재질 속성: `BlendMode=BLEND_Masked`, `TwoSided=true`, `ShadingModel=MSM_DefaultLit`, `OpacityMaskClipValue≈0.3333`(UE 기본값).

**이월 과제 #3 해소**: `refPath` 표기법은 `/Package/Path.ObjectName`(패키지명+오브젝트명 dot 표기, 예 `/Game/Materials/M_Sprite_Flipbook_Lit.M_Sprite_Flipbook_Lit`)가 정답이었다 — 이 형식으로 `get_expressions`가 즉시 성공(이전 세션에서 실패했던 지점이 바로 이 호출).

### 8-2. `M_Sprite_PartComposite` 그래프 구성

**도구 선택**: 노드 ~38개 + 연결 ~45개 + 파라미터 설정 ~13개(개별 호출 시 약 100회)를 낱개 MCP 호출로 하면 왕복이 지나치게 많아 위험(안정성 규칙 위반 소지)하다고 판단, `ProgrammaticToolset.execute_tool_script`(서버측 Python 스크립트로 다른 툴들을 오케스트레이션)를 발견해 활용. 먼저 **1노드짜리 스모크 테스트**로 메커니즘을 검증한 뒤, **2개 청크**로 나눠 실행(청크 하나가 멈춰도 이전 청크 상태는 남도록 체크포인트 확보):
- 청크 A: 공유 UV 계산 체인 23노드(TextureCoordinate×1/ComponentMask×2/Time×1/Subtract×1/Multiply×1/Floor×1/Fmod×2/OneMinus×1/LinearInterpolate×1/Add×2/Divide×2/AppendVector×1 = 16개 + ScalarParameter 7개[TimeOffset/FPS/FrameCount/GridX/FlipX/RowIndex/GridY]) — 8-1에서 역산한 수식을 그대로 복제.
- 청크 B+C: TextureSampleParameter2D 5개(SkinTex/FaceTex/HairTex/ClothTex/WeaponTex, 전부 청크A의 AppendVector 출력을 UV로 공유) + Lerp 블렌드체인 4개(skin→face→hair→cloth→weapon 알파오버) + Max 오퍼시티체인 4개 + EmissiveBoost 파라미터(8번째 스칼라)+Multiply 1개 + `connect_to_output`×3(BaseColor/OpacityMask/EmissiveColor) + 재질속성(`BlendMode=Masked`,`TwoSided=true`,`ShadingModel=DefaultLit`) + **`recompile`**.

두 청크 모두 예외 없이 성공 반환 — **특히 `recompile`이 청크 B+C의 마지막 스텝인데 여기서 예외가 안 났다는 것은 셰이더가 실제로 컴파일 통과했다는 뜻**(툴 설명: "raises if the shader fails to compile"). 최종 `get_expressions` 재조회로 노드 수 38개 확인 — 23(공유 UV 계산, TexCoord 포함) + 5(텍스처 샘플) + 4(블렌드 Lerp) + 4(오퍼시티 Max) + 1(EmissiveBoost 파라미터) + 1(Emissive Multiply) = 38, 정확히 일치.

**출력 배선 검증**(`get_property_input`로 재조회, 설정한 그대로 나옴):
- `MP_BaseColor` ← `LinearInterpolate_4`(=weapon까지 다 덮은 최종 블렌드)
- `MP_OpacityMask` ← `Max_3`(=5레이어 알파 중 최댓값)
- 재질 속성 재조회: `blendMode=BLEND_Masked`, `twoSided=true`, `shadingModel=MSM_DefaultLit`, `opacityMaskClipValue=0.3333`(마스터와 완전 일치)

**로직 정합성 추가 검증**(스크린샷 없이 "픽셀이 맞게 나올지" 확인하기 위한 구조적 근거):
- `ComponentMask_0`(U) 재조회 → `{r:true, g:false, b:false, a:false}`, `ComponentMask_1`(V) → `{r:false, g:true, ...}` — 마스터와 채널 배정 완전 일치(U/V 뒤바뀜 없음).
- `ScalarParameter_3`(GridX)=8, `ScalarParameter_6`(GridY)=17 재조회로 heroes99 그리드(8칸×17행)와 일치 확인.
- Lerp 인자 순서 검토: `Lerp(A=아래레이어, B=위레이어, Alpha=위레이어.A)` — 위 레이어 알파가 1이면 위 레이어로 완전 대체, 0이면 아래 레이어 그대로 — "뒤가 앞을 덮는다" 요구사항과 정확히 일치.

§2에서 준비된 5장(`T_A0_{SkinTex_c3,FaceTex_c2,HairTex_f7c6,ClothTex_c8c4,WeaponTex_w3}.png`)을 `/Game/Sprites/A0_spike/`로 임포트(프로젝트 관례상 스프라이트 텍스처는 `/Game/Sprites/`— 기존 `T_Hero_Knight_Sheet` 등과 동일 경로 규칙, `list_folders`/`find_assets`로 확인 후 결정). 5장 전부 개별 호출로 임포트(임포트가 리임포트/팩토리 확인창을 띄울 가능성을 고려해 배치하지 않고 1건씩 확인) — **5건 전부 모달 없이 정상 응답**.
