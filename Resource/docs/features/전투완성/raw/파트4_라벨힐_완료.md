## 1. 구현 요지

### 작업1 — 유닛 라벨(`WBP_SkillMenu:SetActiveUnit`)
FormatText 체인으로 `"{직업} | {이름}"` 형태 라벨을 구성했다. 직업명은 struct 스키마 무변경으로 `BreakStruct_0.jobId`(구조체 필드, 실측 90100100=전사/90100200=마법사 — job_stats.csv 6행 `JobId` 컬럼과 완전 일치, 유휴 핀 재사용)을 그대로 썼다. `BP_BattleSpawnPoint.CharName`(string, Instance Editable) 변수를 신설해 레벨 8기에 세팅했고, `ST_UI`에 4키(`Job.Warrior`=전사 / `Job.Mage`=마법사 / `Battle.UnitLabel`={0} | {1} / `Battle.TurnLabel`=턴)를 추가했다. 기존 `Txt_TurnLabel`의 "턴" 하드코딩도 바인딩으로 해소했다.

### 작업2 — 치유 제자리(`BP_BattleManager:EventGraph`)
`IfThenElse_3(ALLY1).then` → `CallFunction_166`(Delay 0.55) 직결 와이어 1개를 추가하고, 기존 `VariableSet_6`(SetAttackPointOverride)을 고아화했다. ALLY1의 exec 경로가 SELF(막기, 파트3에서 이미 제자리 게이트 PASS)와 동일해져 치유 시전자가 적진으로 걸어나가지 않게 됐다.

### 작업3 — 힐 표시(`SpawnDamageNumber` + `ResolveHit`)
`SpawnDamageNumber` 내부에 판별자 `Amount>0→피해`를 기준으로 `SelectString`(A="-"/B="+")·`Abs`·`SelectColor`(A=빨강 (1.0,0.4196,0.3843) / B=초록 (0.5608,0.8784,0.4157), 형제 항목과 sRGB 값 직접 일치)을 배선하고 `MakeSlateColor` 브릿지를 거쳐 `SetColorandOpacity`에 연결했다. `ResolveHit`에는 힐 전용 콜사이트를 신설했다: 로컬변수 `HpBefore`(SetHp **앞**에서 캡처) → `Delta = Hp재조회 − HpBefore` → `Amount = 0 − Delta`(부호 반전) → 신규 `CallFunction_4`(SpawnDamageNumber, 기존 피해용 `CallFunction_1`과 완전히 별개 노드) → `CallFunction_24`. 만피 치유는 `Delta=0`이므로 `+0` 초록으로 뜬다(오너 D2 결정 — "실제 회복량 표시"). `BattleLog`의 `dmg=−33`(원시 힐량)은 이 작업으로 건드리지 않았다 — 로그는 원시값, 화면은 적용 델타값으로 의도적으로 분리했다.

## 6. spec.md 선명세와의 관계 (오해 방지용 기록)

`ui/battle/WBP_BattleHUD/spec.md` §3-6(`WBP_DamageNumber`)은 이미 다음을 명세하고 있었다:
> `Txt_Amount` 색: 피해=`#FF6B62` / 회복=`#8FE06A`. 텍스트 포맷: 피해="-{amount}", 회복="+{amount}"(숫자 부호는 로컬라이즈 대상 아님, 순수 포맷).

이 명세는 F1(전투완성 MVP) 단계부터 존재했으나, 야간③(데미지 폰트, 2026-07-16)은 **피해(빨강 `-N`) 쪽만 구현**하고 회복(초록 `+N`) 쪽은 남겨뒀었다. 파트4 작업3이 이 spec.md §3-6의 회복 절반을 이번에 완성한 것이다 — **"스펙에 없던 신규 요구"가 아니라 "이미 있던 명세의 미구현분을 채운 것"**임을 다음 사람이 오해하지 않도록 기록한다. 색값(`#8FE06A` ≈ 0.5608,0.8784,0.4157 sRGB)도 spec.md 수치와 일치.
