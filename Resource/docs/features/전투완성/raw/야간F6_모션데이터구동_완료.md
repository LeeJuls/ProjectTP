## 1. 무엇이 됐나 — 스코프 & 구현 체인

**스코프**: `PlayAttack`의 리터럴 3개(RowIndex 5.0/FrameCount 6.0/Delay 0.70)를 `DT_Skills`·`DT_Motions` 조회로 전환. 리터럴 시절과 결과값이 완전히 같아야 하는 **무회귀**가 핵심 제약([[야간큐_TC]] F6-01/F6-02).

**구현 체인**:
1. Manager `PendingSkillId`(신규 멤버) — `EnterExecuting` **진입 직후** `Set=31000000`. **하드코딩 제거 1호**: `ResolveHit`의 `SkillId` 입력 핀이 기존엔 리터럴 `31000000` 직결이었는데, 이번에 `Get PendingSkillId`로 교체됐다. 세팅값 자체는 F6에선 리터럴 고정이 곧 무회귀 정의(F6-06) — F7에서 실제 선택 스킬 ID로 교체될 자리다([[F7_스킬아키텍처_확정]] §8-10).
2. → `GetDataTableRow(DT_Skills)`로 `motionRow` 조회(`RowNotFound` 시 폴백 5).
3. → SpawnPoint의 `PendingMotionRow` 멤버에 스탬프(F6-08).
4. → `PlayAttack`에서 `DT_Motions`를 RowName 산술 `60000000+MotionRow×10000`으로 단건 조회.
5. → `frameCount`(int 컬럼) → `ToFloat` → `PendingFrameCount`(**Float 멤버** — 이 필드를 float로 잡은 것 자체가 int÷int 함정([[야간큐_TC]] F6-05)을 구조적으로 차단).
6. → `Delay=(FC/8)−0.05`.

**설계 편차**: TC 원안이 표현한 "FindItem 체인" 대신, §20에서 이미 검증된 `GetDataTableRow` 패턴을 채택했다 — EventGraph 컨텍스트에서의 재확인으로는 2번째 사례.

## 2. 가짜 GREEN 2건 검출·수정 이력 (핵심 서사)

이번 게이트의 핵심은 기능 구현 자체보다, 검증을 두 번이나 속일 뻔한 가짜 GREEN을 근본원인까지 추적해 잡아낸 과정이다.

### ① FC≥1 가드 분기 반전
- **증상**: `frameCount≥1` 가드의 분기가 뒤바뀌어 있었다 — else(정상 케이스) 분기가 조회값과 무관하게 **6.0을 무조건 재대입**하는 구조.
- **은폐 이유**: Atk1(기본공격 모션)의 실제 FrameCount가 마침 6이라, 버그가 있어도 결과값이 우연히 일치해 겉보기엔 정상으로 보였다 — 함정㊹(§27, "대입문을 그 대입식으로 재검하는 판정은 가짜 GREEN") 계열의 변종: 이번엔 식이 순환한 게 아니라 **특정 테스트값이 우연히 정답과 일치**해 판별력을 잃은 경우다.
- **검출**: verifier의 정적 트레이스(그래프 배선 직접 추적)가 반전된 분기를 검출.
- **수정**: 와이어 스왑으로 분기 교정. 에이전트가 작업 중 스톨되어 Director가 직접 컴파일·저장까지 마무리.

### ② SetPendingSkillId 순서 결함
- **증상**: `SetPendingSkillId`가 판정 경로(약 1.1초 시점)에 배치돼 있었는데, 모션 조회(약 0.4초 시점)는 그보다 먼저 실행돼 **조회 시점엔 `PendingSkillId`가 항상 기본값 0**이었다.
- **연쇄**: `SkillId=0`으로 `DT_Skills` 조회 → `RowNotFound` → 폴백 `motionRow=5` → 그 폴백값 5로 `DT_Motions`를 조회하면 마침 실존하는 유효 행이라 `FrameCount=6.0`이 "정상 성공"처럼 반환됨 — **연쇄 가짜 GREEN**(상류 실패의 폴백값이 하류의 유효 키가 되어 위장).
- **검출**: `motionRow=10`(폴백과 다른 값)을 쓴 관통 테스트가 이 결함을 드러냈다 — 항등 케이스(Atk1, FC=6=폴백)만으론 영원히 못 잡았을 함정.
- **수정**: `SetPendingSkillId`를 이벤트 진입점(`EnterExecuting`) **직후**로 이동 — 모든 소비자보다 시간상 앞서도록 재배치. 수정 후 핀 원문(그래프 실제 연결 상태)으로 증거 확인.

이 2건은 그대로 [[언리얼_MCP_실전노하우]] §30 함정 (59)·(60)의 출처가 됐다.

## 3. 검증

- **RowName 산술 17/17 전수 성립** — `DT_Skills`의 MotionRow 17행 전부 `60000000+MotionRow×10000` 공식이 `DT_Motions` 실제 RowName과 일치, 미스매치 0(F6-03).
- **관통 실측 10/5.0(Casting1)** — `motionRow=10` 디버그 오버라이드로 "Casting1" 모션의 `FrameCount=5.0`이 정확히 반환됨을 확인(폴백이 아닌 실제 조회 경로가 작동함을 증명한 결정적 테스트 — 위 ②를 검출한 바로 그 테스트).
- **원복 후 5/6.0 원상** — 디버그값을 되돌린 뒤 기본공격(Atk1, motionRow=5) 조회가 정확히 `FrameCount=6.0`(기존 리터럴과 동일)으로 복원됨을 재확인.
- **원복 4단계 증거**(`set_rows`→`get_rows`→`save`→`is_dirty false`) — 함정㊴(§25, 디버그값 원복 절차 누락 사고)의 교훈을 DataTable 행 디버그값에도 그대로 적용해, 되돌림을 "실행"이 아니라 "실행+재조회+저장+dirty 플래그 확인"까지 전 과정 증거로 남겼다.
- **무회귀 7/7**(RowIndex5/FrameCount6.0 기준값 불변·`dmg` 대역 불변·런타임 에러 0).
- **③⑤ 회귀 무** — 같은 야간큐의 형제 게이트 ③(데미지폰트)·⑤(로그보강)에 이번 작업이 부정적 영향을 주지 않았음을 확인.
