## 1. 타겟 범위(AoE) 구조 — `Target` 컬럼 확장

### 1-1. 안 비교

현행: `Target` 1컬럼, 토큰 `ENEMY1`/`ALLY1`/`SELF`(전부 단일). F_SkillsRow.Target = String(F2 임포트 완료).

| 축 | **안A: enum 값 확장 (1컬럼 유지)** | 안B: `TargetKind`+`TargetScope` 2컬럼 분리 |
|---|---|---|
| 신규 토큰/컬럼 | 토큰 2개 추가: `ENEMY_ALL`, `ALLY_ALL` | 컬럼 2개로 재편: Kind{ENEMY,ALLY,SELF} × Scope{SINGLE,ALL} |
| struct 변경 | **없음**(String 값만 추가 — DT 재임포트도 신규 스킬 행 추가 시점에만) | F_SkillsRow 필드 추가 = **오너 수동**(F2 §2-3 실측: MCP로 UserDefinedStruct 필드 추가 불가) + DT_Skills 재임포트 |
| BP_BattleManager 분기 | 기존 3분기(F0②/F7-3) **무변경 + 2케이스 추가** | 기존 분기표 전면 재작성(Target 단일 문자열 계약 파기) |
| 유효풀 계산 | 토큰→풀 매핑 표 5행(§1-2) | Kind가 풀, Scope가 개수 — 의미 직교(장점) |
| 무의미 조합 배제 | **구조적 배제** — `SELF_ALL` 토큰을 발급하지 않으면 존재 자체가 불가 | `SELF`+`ALL` 조합이 데이터로 가능 → 검증 룰 필요 |
| 확장성 | 스코프 종류가 늘면(ROW/RANDOM2 등) 토큰 조합 폭발 | 스코프 추가에 강함 |
| "AoE인가?" 질의 | 토큰 나열 비교(`== ENEMY_ALL or == ALLY_ALL`) — 소비처가 늘면 누락 위험 | `Scope == ALL` 단일 비교 |

**권장 = 안A.** 근거: ① struct 무변경(오너 수동 병목·재임포트 리스크 회피), ② 승인된 F0②/F7-3 분기 계약이 "확장"으로 성립(재작성 아님), ③ 무의미 조합이 발급 단계에서 원천 배제, ④ 알파 스코프는 SINGLE/ALL 2종뿐이라 토큰 폭발 미발생. **마이그레이션 예약**: A2 30원자 스킬 재편 때 스코프 종류가 3+로 늘면 그 시점에 안B로 전환(CSV 컬럼 추가 = 하위호환, 데이터규약).

### 1-2. 토큰→유효풀 매핑 (단일 소스 함수 계약)

풀 계산·하이라이트·버튼 활성·커밋이 **각자 재구현하지 않도록**, Manager에 단일 소스 함수를 신설한다(§10 구현 명세):

`ResolveTargetPool(TargetToken: String, Caster: unit) → unit[]` — **TurnQueue 인덱스 오름차순으로 반환(결정론, F9a 재현성)**.

| Target 토큰 | 풀 정의 | 선택 방식 | 커밋 결과(SelectedTargets) |
|---|---|---|---|
| `ENEMY1` | 상대팀(bIsParty ≠ Caster) ∧ bAlive | 풀 하이라이트 → 1기 클릭 | [클릭 유닛] (길이 1) |
| `ALLY1` | 아군(bIsParty = Caster) ∧ bAlive, **자신 포함**(F0②) | 풀 하이라이트 → 1기 클릭 | [클릭 유닛] |
| `SELF` | [자신] | AwaitTarget 스킵(기존) | [자신] |
| `ENEMY_ALL` (신규) | ENEMY1과 동일 풀 | §2 흐름(ALL 모드) | **풀 전체** |
| `ALLY_ALL` (신규) | ALLY1과 동일 풀(자신 포함) | §2 흐름(ALL 모드) | **풀 전체** |

- 슬롯 bEnabled(F7-2) 확장: `ENEMY_ALL` = ENEMY1과 동일 조건(풀 ≥1). `ALLY_ALL` = 항상 true(자기 턴엔 자신이 생존이므로 풀 ≥1 보장 — SELF와 동일 논리).
- `SELF_ALL`·`ENEMY0` 등은 **토큰 미발급 = 존재 불가**(안A의 배제 방식). 미지 토큰이 데이터에 들어오면: DT 조회 후 매핑 실패 → 로그 1줄 + 해당 슬롯 bEnabled=false(방어적 폴백, 크래시 금지).

### 1-3. AoE 실행 모델 — TakeHit 계약 불변, Manager 루프

**TakeHit(Attacker, Target, SkillId)는 단일 대상 원자 계약(§8) 그대로 둔다.** AoE는 상위(Executing)에서 순회한다:

```
Executing A분기(기존: Delay 0.25 → TakeHit 1회):
  Delay 0.25 → ForEach(SelectedTargets[i], 큐 순서) → TakeHit(Attacker, SelectedTargets[i], SkillId)
  (같은 프레임 내 순차 루프. B분기(Delay0.75→WalkBack→EnterTurnEnd)는 무변경)
```

- **로그 = 대상당 1줄**(BattleLog 문법 불변 — F9a 원장이 대상별 행으로 그대로 대조 가능).
- **다중 사망 레이스 안전은 기존 구조가 이미 보장**: TakeHit는 플래그(bAlive/bBattleOver)만 세팅, End 전이는 EnterTurnEnd 단일 관문(F5-2) — 루프 중 몇 기가 죽든 상태 전이 경로는 하나다. bBattleOver는 set-only(후속 반복이 false로 되돌리지 않음).
- **루프 중단 없음**: bBattleOver가 서도 남은 대상에 계속 적용. 단 구조적으로 ENEMY_ALL 풀은 한쪽 파티뿐이라 "마지막 생존자 사망 = 루프의 마지막 죽음"이며, 살아있는 풀 잔여를 남긴 채 전투가 끝나는 경우는 발생 불가(§9 E1). ALLY_ALL(힐)은 사망을 만들지 않음.
- **동시 전멸 불변**: AoE도 적 또는 아군 한쪽만 타격 — §7-4 "한 번의 피해가 양측을 죽일 수 없음" 전제 유지.
- **안무 분기 없음**: F7-3 "균일 step-forward"를 AoE도 그대로(PlayAttack 1회, 대상별 걸음 분기 없음 — A1 범위).
- ⚠ **HURT 리액션 병렬 함정**: F4 §4-1의 `PendingHitTarget` 멤버 경유 + RetriggerableDelay 패턴이 **Manager 단일 인스턴스**면, N회 호출 시 delay가 리셋되어 마지막 대상만 idle 복귀한다. → HURT 재생·복귀 이벤트는 **대상 유닛(BP_BattleSpawnPoint) 소유**로 두어 액터별 delay 인스턴스를 분리할 것(F4 구현 전 확인 항목, §10).

## 2. AoE 선택 UI 흐름

### 2-1. 안 비교

| 축 | 안A: SELF식 즉시커밋 재사용 | **안B: 전체 하이라이트 + 확인 클릭 1회 (권장)** |

| 상태머신 | AwaitTarget 완전 스킵(SELF 경로 재사용) | AwaitTarget 재사용(하이라이트 집합·클릭 유효집합·커밋 결과만 ALL로 치환 — 상태·전이 신설 없음) |
| 구현비 | 0 (분기 1개) | 소 (AwaitTarget에 모드 플래그 1개) |
| 입력 리듬 | 공격 스킬이 클릭 1회로 즉발 — **오클릭 즉시 확정**(취소 부재는 A1 공통이라 되돌릴 수 없음) | ENEMY1과 동일 리듬(스킬 → 하이라이트 확인 → 클릭) — 커밋 전에 "누가 맞는지" 시각 확인 가능 |
| 일관성 | SELF의 즉시커밋은 "자기 대상이라 무해"가 성립 근거 — 공격 스킬로 확장하면 근거 상실 | "대상 있는 스킬은 항상 하이라이트를 보고 클릭 1회로 확정" 규칙이 단일·전체에 균일. SELF만 유일 예외(기존 그대로) |
| WBP 전방호환 | — | HUD 시대에 "전체 확인 버튼"으로 치환하기 쉬움(하이라이트 확인 단계가 이미 존재) |

**권장 = 안B.** SELF 스킵 패턴은 "대상=자신·무해"라는 전제 위의 예외이며, 공격성 AoE에 이식하면 오클릭 방호가 사라진다. 안B는 상태머신 신설 없이 기존 AwaitTarget의 집합만 바꾼다.

### 2-2. 상태·전이 표 (신규 변경분 굵게)

| 상태 | 진입 | 이탈 전이 | 본 설계 변경 |
|---|---|---|---|
| TurnStart | EnterTurnEnd Delay(0.35) 후 | bAlive=false → TurnEnd(기존) / **STUN 스킵 → TurnEnd(신규, §4)** / 정상 → AwaitCommand | **상태이상 스윕 파이프라인 삽입(§3-3)** |

| Executing | SELF 즉시 or AwaitTarget 커밋 | B분기 → TurnEnd(기존) | **A분기 TakeHit × SelectedTargets 루프(§1-3)** |

| End | TurnEnd 관문에서만 | InitBattle 재호출(비파괴) | InitBattle 리셋 목록에 ActiveStatuses·SelectedTargets 추가(§10) |

```

```
버튼은 여전히 `NotifySkillSelected(SkillId)`만 호출(F7-1 매니저 소유 원칙 불변 — WBP_BattleHUD 드롭인 교체 호환).

## 3. 상태이상 구조

### 3-1. 데이터 위치 — 기존 Effect* 컬럼 확장 (권장 안A)

| 축 | **안A: skills.csv 인라인**(Effect* 3컬럼 재사용 + `EffectProcChance` 1컬럼 신설) | 안B: statuses.csv 분리 + skills 2컬럼 참조(StatusId, ProcChance) |
|---|---|---|
| 신규 자산 | 컬럼 1개 + F_SkillsRow 필드 1개(오너 수동 1회) | 신규 테이블·struct·DT·FK 해석 런타임 경로 전부 신설 |
| §8 정합 | step3이 이미 EffectType 분기 — **기존 계약의 자연 연장** | step 8.5에서 2차 DT 조회 필요 |
| 한계 | 스킬당 효과 1개, 효과 파라미터가 스킬마다 중복 | 다중 효과·재사용·상태 NameKey의 정식 거처 |
| 표시명 | 상태 표시명 키는 strings.csv에 직접 발급(§12 — 테이블 없이도 가능) | statuses.csv에 NameKey 컬럼 |

**권장 = 안A(알파).** "간단이 지상명령" + 알파 상태이상은 소수(기절·공약 계열)라 정규화 이득이 비용을 못 넘는다. **balance 카탈로그가 상태이상 종류를 4+종·스킬당 다중 효과로 키우면 그 시점(A2)에 안B로 마이그레이션** — 본 문서의 채널 계약(§3-2)은 양쪽 데이터 위치에 그대로 이식된다. *balance 축이 별도 statuses 테이블을 이미 설계 중이면 Director 병합에서 안B로 뒤집어도 본 문서의 런타임 구조(§3-2~§6)는 무변경 — 바뀌는 것은 step1 해석 단계의 조회 경로뿐.*

**EffectType 토큰 패밀리 계약(신규 핵심)** — 어떤 토큰이 §8의 어느 스텝에서 소비되는가:

| 패밀리 | 토큰(알파) | 판정 스텝 | 확률 | 적용 대상 | 성격 |
|---|---|---|---|---|---|
| SELF_BUFF (캐스트 확정형) | `DMG_REDUCE` | step3 (goto 9) | 무조건 | 시전자 | 스킬 자체가 버프(피해 없음) — 기존 그대로 |
| **ON_HIT (피격 라이더형, 신규)** | `STUN`, `ATK_DOWN` | **step 8.5** | `EffectProcChance` | 피격자 | 데미지 스킬에 얹히는 부가 효과 |
| 기타 | `NONE`, `HEAL`(치유 행의 기존 표기, 불변) | —/step2 | — | — | 기존 데이터 무변경 |

신규 토큰 발급(예: `DEF_DOWN`)은 balance가 하되 **반드시 패밀리 하나에 귀속**시켜야 한다(귀속 = 판정 스텝 결정). 패밀리 미귀속 토큰 = §1-2와 동일한 방어적 폴백(로그+무효과).

### 3-2. 유닛 보유 구조 — 배열 (권장)

| 축 | **배열: `ActiveStatuses: F_ActiveStatus[]` (권장)** | 고정 슬롯(블록 선례: 효과별 멤버 변수 세트) |
|---|---|---|
| 신규 struct | `F_ActiveStatus { StatusToken: String, Value: Float, RemainingTurns: Integer }` — 오너 수동 1종(F2에서 4종 생성한 기검증 경로) | 없음 |
| 상태 추가 비용 | 데이터 행 + (신규 채널일 때만) 훅 1곳 | 상태마다 멤버 2~3개 + 틱/적용/조회/해제 4곳 그래프 수술 반복 |
| 틱/만료 | 루프 1개(§3-3)로 전 상태 균일 처리 | 슬롯별 개별 배선 |
| 오너 의도 정합 | "각종 상태이상" + balance 카탈로그 병행 = 종류 증가 예고 → 배열이 맞는 그릇 | 정확히 2종에서 멈출 때만 우위 |

**권장 = 배열.** 단 **막기(DMG_REDUCE)의 기존 전용 필드(bBlockActive/BlockValue 등, F4 §4-2 스펙)는 배열로 마이그레이션하지 않는다** — 승인된 F4/F7 스펙·TC(H1 지속형·해제 타이밍) 무변경 유지, 두 시스템의 통합은 A2 옵션으로만 예약. 배열은 **신규 ON_HIT 상태 전용**이다.

**채널 계약** — StatusToken이 코드의 어느 훅에서 읽히는가(상태 추가 시 이 표만 갱신):

| StatusToken | 채널(코드 훅) | Value 의미 | 훅 지점 |
|---|---|---|---|
| `STUN` | 턴 스킵 | 미사용(0 고정) | EnterTurnStart 스윕(§3-3 P5) |
| `ATK_DOWN` | 발신 피해 배율 | 감소율 r (0<r<1), 배율 = (1−r) | TakeHit step4(§5) |
| (예약, 알파 비구현) | DEF_DOWN=step5 / HEAL_DOWN=step2 / DOT=TurnStart | — | 채널 신설 = 코드 훅 1곳 추가 |

동일 채널 다중 엔트리(서로 다른 토큰) 공존 시 배율은 **곱 합성** `Π(1−Value_i)`(교환법칙으로 순서 무의미 — 알파 카탈로그는 채널당 1종이라 실발생 없음, 방어적 정의).

### 3-3. 틱 파이프라인 — EnterTurnStart 단일 앵커 (권장 안A)

지시서 제안(H3 쿨다운과 동일 지점) 채택. **EnterTurnStart 파이프라인 고정 순서**:

```
EnterTurnStart(U):
 P1. U.bAlive == false → [기존 사망 스킵: TurnCounter 미증가, 마커·카메라 없음] 즉시 EnterTurnEnd
 P2. TurnCounter +1  (기존 위치 불변 — F1 §2. 기절이어도 여기를 지나므로 +1, §4-3)
 P3. 쿨다운 스윕: U의 전 스킬 쿨 −1, 하한 0  (기존 H3 불변)
     막기 해제: U.bBlockActive 해제  (기존 H1 불변 — ★기절 여부와 무관하게 실행됨, §9 E8)
 P4. [신규] 스킵 판정: bSkipTurn = ActiveStatuses에 (StatusToken=STUN ∧ RemainingTurns>0) 존재
     ★차감 전 값으로 판정 — 차감을 먼저 하면 dur1 기절이 스킵 없이 증발
 P5. [신규] 틱: 전 엔트리 RemainingTurns −1 → ≤0 엔트리 제거(+StatusLog EXPIRE)
 P6. bSkipTurn == true → StatusLog SKIP_TURN 방출 → [사망 스킵과 동일 꼬리: 마커·카메라·행동 없이 EnterTurnEnd]
 P7. 정상 진행(기존 마커·카메라·AwaitCommand)
```

- 앵커 = **피부여자(afflicted)의 자기 턴 시작.** 쿨다운(H3)·막기 해제(H1)와 같은 지점 → 훅 1곳, 신규 상태 전이 없음.
- **기절이 쿨다운 회복·막기 해제를 멈추지 않는다**(P3가 P4보다 앞) — 기절의 이중 페널티 방지 + "기절당한 막기 유닛의 영구 막기" 버그 원천 차단(§9 E8).
- 대안(안B, 비권장): 발신형(ATK_DOWN) 채널만 자기 턴 **종료** 차감으로 분리하면 아래 실효표의 +1 오프셋이 사라지나, 틱 지점이 2개가 되고 관문(EnterTurnEnd)에 신규 로직이 들어간다(F5 관문은 얇게 유지가 원칙). 오프셋은 검증 룰(V3)로 차단 가능하므로 단일 앵커 우선.

**실효 지속(Effective Duration) 표 — balance 필독(카탈로그 값 발급 기준)**:

| 채널 | dur=1 | dur=2 | dur=N | 비고 |
|---|---|---|---|---|
| STUN(스킵) | 자기 턴 1회 스킵 | 2회 스킵 | N회 스킵 | 직관 일치 |
| ATK_DOWN(발신) | **무효과 — 데이터 함정, V3로 금지** | 약화된 행동 1회 | 약화된 행동 N−1회 | 적 턴에 부여→내 턴 시작 P5에서 즉시 −1되므로 +1 오프셋 |
| (참고) DMG_REDUCE 기존 | 다음 자기 턴 시작까지 피격 전부 감소 | — | — | H1 그대로, 배열 미편입 |

### 3-4. 재적용 규칙 — 갱신(refresh), 중첩 금지 (권장)

`ApplyStatus(Unit, Token, Value, Duration)`:

```
동일 Token 엔트리 존재 → Value=신규, RemainingTurns=신규 Duration으로 완전 리셋 (StatusLog REFRESH)
부재                → 엔트리 추가 (StatusLog APPLY)
서로 다른 Token      → 공존(배열이므로 자연)
```

- **중첩(스택) 금지 근거**: ① duration 합산형 스택은 기절 락(§4-4)의 직접 원인, ② Value 합산형 스택은 ATK_DOWN 누적으로 유효 Atk 0 붕괴 경로(min1과 상호작용해 칩뎀 교착 재현), ③ 구현 최소(찾아 덮어쓰기), ④ JRPG 관례.
- 소포크(1줄): "강한 디버프 위에 약한 디버프 = 약한 것으로 덮임(마지막 승리)"이 갱신의 대가 — 대안은 `Remaining=max(잔여,신규)`이나 Value·Duration 소스가 갈라져 오히려 예측 불가. 알파 카탈로그(채널당 1종)에선 차이 실발생 없음 → **완전 리셋 권장**, balance가 겹침을 카탈로그에서 회피 설계.
- 만료·해제 시 **스탯 원복 없음** — ATK_DOWN은 Atk 필드를 mutate하지 않고 step4에서 조회-곱만 하므로(F8 "Atk 스탯 불변" 가드레일과 동일 철학) 엔트리 제거 = 즉시 원상.

## 4. 기절(STUN) ↔ 턴 흐름 통합

### 4-1. 체크 지점 = F5 사망 스킵과 동일 지점, bAlive 뒤

§3-3 파이프라인 그대로: **P1 bAlive(기존) → … → P4 STUN 체크(신규)**. 순서 근거: 시체는 상태이상 틱 자체가 없어야 하고(P1이 먼저 탈출), 기절 스킵의 꼬리(마커 안 켬·카메라 미부착·즉시 EnterTurnEnd)는 사망 스킵과 **동일 코드 경로 재사용** — EnterTurnEnd 안의 bBattleOver 관문(F5-2)도 자동으로 지난다(신규 End 경로 없음).

### 4-2. 스킵 시 소모 — 소모된다 (명시)

기절 스킵은 그 턴에 STUN의 RemainingTurns를 −1 **소모한다**(P4 판정 → P5 차감). dur=1이면 정확히 자기 턴 1회 스킵 후 만료. 스킵이 duration을 소모하지 않는 설계(별도 해제 조건)는 금지 — "스킵은 반드시 지속을 감소시킨다"가 §4-4 무한 락 방지의 기계적 전제다.

### 4-3. TurnCounter 정책 — 기절 스킵도 +1 (권장, balance 확인 1건)

- 기존: 사망 스킵은 미증가(F1 §2 "실제로 진행된 턴만 카운트").
- 기절은 **살아있는 유닛의 턴이 소모된 것** — F1 §2의 anti-stall 의미("막기·치유도 유닛의 턴이므로 증가")와 같은 편이며, 기절 다발 게임일수록 광폭화 램프가 유지되어 종결 보장에 유리하다. 파이프라인상 P2(기존 증가 위치)가 P4(기절 체크)보다 앞이라 **구현상으로도 자연히 +1**.
- F1 문서는 사망 스킵만 다뤘으므로 **balance에 정합 확인 1건**으로 넘긴다(수치 재계산 불요 — F9a even-trade 시나리오엔 기절이 없어 기존 원장 무영향).

### 4-4. 무한 기절 락 검토

**기계적 데드락(상태머신이 멈추는가): 불가능.**
- 불변식: ① 스킵은 반드시 duration을 −1 한다(§4-2). ② 상태이상 부여는 반드시 어떤 유닛의 "행동"을 요구한다(ON_HIT = 스킬 시전의 라이더). ③ 재적용은 갱신뿐, 지속 합산 없음(§3-4).
- 따라서 전원(8기) 동시 기절이어도 각자의 턴마다 duration이 엄격 감소 → 유한 턴 내 누군가 행동 가능. 큐는 스킵을 그냥 지나가므로(H2 길이 불변, % 순환) 크래시 지점도 없다.

**경제적 무한(끝나지 않는 게임): 조건부 가능 — 가드레일 필요.**
- 위험 시나리오: **순수 기절 스킬(PowerRate=0)** 을 양측이 교대 시전 → 피해 0 → HP 불변 → 영원히 종료 조건 미충족. 광폭화도 무력(0에 배율을 곱해도 0). F1 §6 "순수-수동 홀(공격 0=피해 0=∞)"과 동일 계열의 구멍이 **데이터로 성립 가능**해지는 것이 핵심 차이다.
- **가드레일(종류 제시 — 값·채택은 balance 소관)**:

| # | 가드레일 | 성격 | 본 설계의 처리 |
|---|---|---|---|
| G1 | **ON_HIT 상태이상은 PowerRate>0 스킬에만 부착**(순수 디버프 금지) | 구조(데이터 검증 룰 V1) | **본 문서가 요구** — 락이 성립해도 매 시전이 피해를 누적(+광폭화 스케일) → 종결 보장 |
| G2 | 중첩 금지 + 갱신만 | 구조(§3-4) | **본 문서가 요구** — duration이 쌓여 늘어나는 경로 차단 |
| G3 | 기절 스킬 `CooldownTurns ≥ Duration + 1` | 수치 권고식 | 단일 시전자가 한 대상을 영구 락 불가하게 하는 산식 형태만 제시, 값은 balance |
| G4 | 재기절 면역/디미니싱(깬 직후 N턴 면역) | 카탈로그 옵션 | 필요 시 balance가 채택 — 구조상 훅은 ApplyStatus 안 1분기로 수용 가능(알파 기본 미구현) |
| G5 | 절대 턴상한 무승부 룰 | 베타 예약 | F1 §6 기존 플래그와 동일 지점에 병합(신규 아님) |

- G1+G2가 있으면 "게임이 안 끝나는" 기절 락은 구조적으로 불가(피해가 반드시 누적). G3~G4는 "재미(한쪽이 계속 스턴당하는 체감)" 영역 — balance 판단.

### 4-5. 기절의 표시 (알파)

A1 임시물 규율(thin) 준수 — 기절 시각 표현(아이콘·포즈·부동 텍스트)은 **신설하지 않는다**(비범위: 데미지 부동숫자와 동급). 스킵은 StatusLog로만 기록, 시각 피드백은 WBP_BattleHUD(#1) 시대에 로컬라이제이션 키(§12)로 표시. 기절 유닛은 idle 유지(모션 변경 없음).

## 5. 공격력약화(ATK_DOWN) ↔ TakeHit step4 합성 계약

F8이 확정한 step4 국소 원칙([[광폭화_재검증]] §7)에 **같은 슬롯·같은 방식**으로 인자 1개를 추가한다:

```
step4 (최종형, F8 병합 후):
  base = floor( Attacker.Atk × PowerRate × BerserkMult(TurnCounter) × OutgoingAtkMult(Attacker) )

  OutgoingAtkMult(U) = Π(1 − Value_i)   # U.ActiveStatuses 중 ATK_DOWN 채널 엔트리. 없으면 1.0
```

| 계약 항목 | 내용 |
|---|---|
| 적용 스텝 | **step4 국소, 단일 floor**(F8과 동일 — 중간 floor 금지) |
| 곱 순서 | Atk × PR × Berserk × StatusMult — 전부 곱이라 교환법칙상 순서 무의미. 표기는 이 순서로 고정(로그·문서 일관성) |
| Atk 스탯 | **불변(mutate 금지)** — F8 가드레일과 동일. 만료 시 원복 로직 자체가 불필요해짐(§3-4) |
| HEAL 비적용 | step2(HEAL)는 step4 미경유 → **치유량은 시전자의 ATK_DOWN에 영향받지 않음**(F8 TC-F8-05 "힐 미스케일"과 동일 가드 — 별도 채널 HEAL_DOWN은 알파 밖 예약) |
| DMG_REDUCE와 독립 | 막기는 step6(수신 배율), ATK_DOWN은 step4(발신 배율) — 겹쳐도 각자 스텝에서 곱, 상호 간섭 없음 |
| min1 | step7 max(1,·)이 최종 방어 — ATK_DOWN으로 0 이하가 되어도 1 보장(기존 그대로) |
| 값 범위 | Value = 감소율 r, **0 < r < 1** (r=1은 유효 Atk 0 = 순수-수동 홀 재현이므로 상한 exclusive — 스키마 제약 V6, 정확한 값은 balance) |

## 6. TakeHit §8 계약 확장 — step 8.5 삽입 (기존 번호 불변)

기존 0~9의 의미·번호를 바꾸지 않고(문서·TC 상호참조 보존) **8과 9 사이에 8.5를 신설**한다. HEAL(step2)·DMG_REDUCE(step3)는 goto 9로 **8.5를 구조적으로 우회** — 라이더 확률판정과 겹칠 수 없다.

```
0. 쿨 체크(기존)
1. 스킬 해석(기존) + EffectProcChance 로드 [신규 컬럼]
2. Kind==HEAL → 회복 처리; goto 9                                  (기존 — 8.5 미경유)
3. EffectType==DMG_REDUCE → 자기버프 부여; goto 9                   (기존 — 8.5 미경유)
4. base = floor(Atk × PR × BerserkMult × OutgoingAtkMult)           (§5 확장)
5. dmg = base − Def                                                 (기존)
6. 막기 배율(기존)
7. max(1, floor)(기존)
8. HP 적용 + 사망 플래그/승패 플래그(기존 F4/F5 — bAlive=false 시 Target.ActiveStatuses 전체 Clear 추가, §9 E3)
8.5 [신규] 상태이상 확률판정 — EffectType ∈ ON_HIT 패밀리({STUN, ATK_DOWN})일 때만:
    a. Target.bAlive == false → 미적용(로그 effectApplied=false) → 9   # 시체에 상태이상 없음
    b. roll = RandomFloat01()                                          # 전투 최초의 난수 — §7-3 결정론 절 필수 참고
    c. roll < EffectProcChance → ApplyStatus(Target, EffectType, EffectValue, EffectDurationTurns)  # §3-4 갱신 룰
       else → 미적용
    d. BattleLog effect 필드 세팅(§7-1)
9. 쿨다운 세팅 + 애니 이벤트 + 로그 방출 + return                    (기존 — 8.5 결과와 무관하게 항상 실행)
```

**삽입 위치 근거**: ① 사망 판정(8) **뒤** — 이 타격으로 죽은 대상에 상태이상이 붙는 모순 차단(8.5a), ② 쿨 세팅(9) **앞** — 발동 실패해도 쿨은 소모(단일 출구 유지), ③ 2·3의 goto 9와 무교차 — 힐/자기버프 스킬은 라이더를 가질 수 없음이 구조로 보장(+데이터 검증 룰 V2로 이중 방어), ④ AoE는 대상별 TakeHit 호출이므로 8.5도 **대상별 독립 굴림**(자연 획득).

**순수 디버프(PowerRate=0 + ON_HIT) 분기 — 포크**:
- **안A(권장): 알파 금지** — 데이터 검증 룰 V1으로 차단. 근거: §4-4 G1(무한 락의 구조 가드) + step7 min1 때문에 PowerRate 0이어도 1뎀이 나가는 어색함 자체를 회피.
- 안B: 허용하려면 step4 앞에 가드 분기 신설 — `PowerRate==0 ∧ EffectType∈ON_HIT → dmg=0 확정(4~8의 HP 변경·min1·사망체크 전부 스킵) → 8.5 직행`. balance가 순수 디버프를 꼭 원할 때만 채택(그 경우 G3/G4 가드레일 채택이 사실상 필수가 됨을 명시).

## 7. 로그 스키마 확장

### 7-1. BattleLog 확장 — 옵션 필드 3개 (기존 6필드·순서 불변)

```
BattleLog|turn=|attacker=|target=|action=|dmg=|hp=[|berserk=][|effect=<EffectType토큰>|effectRoll=<0~1 float>|effectApplied=<true/false>]
```

- **effect 3필드는 ON_HIT 스킬일 때만 존재** — 기존 스킬의 로그 라인은 바이트 단위로 불변 → F9a 기존 원장·`extract_battle_log.py` 파서(라인 통과형) 자동 호환.
- 필드 상대 순서 고정: `hp=` → (`berserk=`, F8 후속 확장과 공존) → `effect=` → `effectRoll=` → `effectApplied=`.
- `effectRoll`을 **항상 기록**(발동 여부와 무관) — `roll < ProcChance ⇔ applied` 를 사후 감사 가능하게 하여, 난수 도입 후에도 verifier가 판정 로직 자체를 결정론적으로 검증할 수 있다(굴림값은 랜덤이되 "굴림→결과 매핑"은 로그로 완전 감사).
- effectApplied=false의 두 원인(굴림 실패 vs 대상 사망)은 roll·hp 필드 조합으로 구분 가능(hp=0 ∧ applied=false = 사망 미적용) — 별도 reason 필드는 두지 않음(스키마 최소).
- AoE: 대상당 1줄(§1-3) — 같은 turn·attacker·action에 target만 다른 N줄.

### 7-2. StatusLog 신설 — 상태 라이프사이클 (별도 프리픽스)

```
StatusLog|turn=<TurnCounter>|unit=<GetDisplayName>|status=<토큰>|event=<APPLY|REFRESH|EXPIRE|SKIP_TURN|CLEAR>|remaining=<차감후 잔여턴>
```

| event | 방출 지점 |
|---|---|
| APPLY / REFRESH | §6 step 8.5c(ApplyStatus 내부) |
| EXPIRE | §3-3 P5(잔여 0 제거) |
| SKIP_TURN | §3-3 P6(기절 스킵 확정) |
| CLEAR | 사망 시 일괄 제거(§9 E3) |

- BattleLog 문법을 오염시키지 않기 위한 별도 프리픽스. 매 턴 틱(TICK) 이벤트는 **기록하지 않음**(로그 소음 방지 — APPLY turn과 EXPIRE turn의 차로 지속 검증 가능).
- ⚠ verifier 확인 1건: `extract_battle_log.py`의 라인 필터가 `BattleLog|` 프리픽스 한정이면 StatusLog는 별도 grep으로 수집(스크립트 수정 아님, 검증 커맨드 추가).

### 7-3. F9a 결정론 유지 전략 (필수 계약)

§8의 "전 과정 난수 없음" 전제가 8.5b에서 처음 깨진다. 결정론 검증을 지키는 3원칙:

1. **기존 F9a 시나리오(기본공격 even-trade) 무영향** — 기본공격은 EffectType=NONE, 8.5 미진입, 로그 라인 불변 → 기존 원장(23 유닛턴) 그대로 유효.
2. **상태이상 TC는 ProcChance를 1.0/0.0으로 강제한 스캐폴드 데이터로 실행**(확정 발동/확정 미발동 양극단 = 결정론 복원). 중간 확률의 통계 검증은 알파 비수행(스모크만 옵션).
3. 난수원: BP `RandomFloat`(비시드) 알파 허용. 완전 재현이 필요해지면 `RandomStream` 시드 고정 옵션(verifier 플래그) — PvP 서버권위 난수는 베타 예약.
