# 야간③b 완료 — 데미지 폰트 위치수정 게이트 통과

> 대상: [[야간③_데미지폰트_완료]] · [[언리얼_MCP_실전노하우]] §31
> 오너가 실사용 중 리포트한 데미지 숫자 오표시 위치(HP바 기준에서 벗어난 엉뚱한 위치) 버그를 진단·수정한 결과를 기록한 문서. 1차 가설(DPI 미보정) 조치로는 미해소였고, 카메라 ON/OFF 대조 진단으로 액션캠 좌표 1회 고정·재투영 부재를 근본원인으로 확정한 뒤 위젯 자가 재투영으로 해결했다.

## 1. 무엇이 됐나 — 증상 & 스코프

**증상(오너 리포트)**: 데미지 숫자가 HP바 위가 아닌 **엉뚱한 위치**에 표시된다.

**스코프**: [[야간③_데미지폰트_완료]]에서 구현한 데미지 숫자 위젯(`WBP_DamageNumber`)의 스폰 위치 계산 로직을 진단·수정.

## 2. 진단 서사 — 카메라 ON/OFF 대조로 근본원인 확정 (핵심 서사)

### 1차 가설 — DPI 미보정 (조치했으나 미해소)
DPI 미보정을 1차 가설로 세우고 좌표 계산 함수를 `ProjectWorldLocationToWidgetPosition`으로 교체 + (0,24) 픽셀 바이어스를 추가했다. **결과: 미해소** — 324px 오차가 그대로 잔존.

### 진단 매트릭스 — 카메라 ON/OFF 대조
1차 조치가 실패하자 카메라 상태를 변수로 격리하는 대조 테스트를 실행했다: 액션캠 **OFF 시 27px**(정상 범위) vs **ON 시 324px**(비정상). 카메라 상태가 오차 크기를 결정하는 변수임이 드러났다.

### H1 확정 — 액션캠 좌표 1회 고정, CamBack 후 재투영 없음
데미지 숫자 위젯은 **스폰되는 순간의 액션캠 시점 좌표를 1회만 계산해 고정**한다. 이후 CamBack(기본 카메라 복귀)이 일어나도 이 좌표는 재투영되지 않는다. 반면 HP바는 **월드 WidgetComponent**라 매 프레임 자동 추적한다 — 이 비대칭이 근본원인(H1)으로 확정됐다.

## 3. 수정 — 위젯 자가 재투영

- `WBP_DamageNumber`에 `TargetWorldLoc` 변수 신설.
- `SetTargetWorldLoc` **래퍼 함수** 신설 — 외부 BP의 raw 멤버변수는 Set 노드를 직접 생성할 수 없다는 툴 함정([[언리얼_MCP_실전노하우]] §31 함정(63))을 우회하는 구조.
- **Event Tick 체인**: `GetOwningPlayer` → `ProjectWorldLocationToWidgetPosition` → (0,24) 픽셀 바이어스 → `SlotAsCanvasSlot` → `SetPosition`. 실패 시 무동작(no-op)으로 안전하게 스킵.
- Manager(`BP_BattleManager`) 측 배선은 `CreateWidget` 직후 `SetTargetWorldLoc` 함수호출 노드 1개 추가로 끝남.
- 2.4s짜리 **일시 인스턴스**라 매 틱 재투영 비용은 무시 가능 — 상시(항상 떠있는) 위젯에 한해 PWTS(Project World To Screen) 매틱 갱신을 폐기하기로 한 기존 원칙과 충돌하지 않는다(데미지 숫자는 애초에 상시 위젯이 아니므로 그 원칙의 적용 대상 밖).

## 4. 검증

- **픽셀 색상 매칭 객관 측정** — 액션캠 프레임 **31.0px**, 기본캠 복귀 **37.4px**(기준 ≤50px → **PASS**).
- 위젯 소멸(수명 종료) 정상 확인.
- **DPI 보정 유지** — 기본캠 정확도에 계속 기여함을 확인(1차 조치가 무효화되지 않았음).
