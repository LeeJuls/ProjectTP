# F7 스킬 아키텍처 확정 (SSOT)

> 이 문서는 F7 스킬 시스템(스킬/마법/버프/디버프/힐 통합 판정)의 **설계 SSOT**다 — system-ui(구조축)+balance(수치축) 설계 위에서 Director가 내린 병합판정이며, **F7b 구현의 유일 참조 문서**다. 여기에 gameplay-engineer의 라이브 BP 실측 피드백, qa-critic BLOCKER 게이트, 그에 대한 Director 최종판정(§8), umg 실측·메뉴 판정(§11)까지 포함한다. TC(테스트케이스) 상세는 별도 [[F7_TC]](qa-critic 작성)가 SSOT이며 본 문서는 다루지 않는다.

## 1. 문서 위상

F7 구현은 **F7a**(메뉴 배선+기존 5스킬 실전, struct 신설 불요)와 **F7b**(3층 데이터 이주+인터프리터, struct 부트스트랩 후)로 분할된다(§7-2). F7a는 [[plan]] §F7 7-1~7-4의 기존 계약이 그대로 유효하며 이 문서의 대상이 아니다 — **본 문서는 F7b가 참조할 유일 설계 문서**다.

설계(§2~§7)는 2026-07-15 주간에 system-ui+balance 병합판정으로 확정됐고, 같은 날 야간에 gameplay-engineer가 실제 BP 그래프(`ResolveHit` 149노드 등)를 실측해 구현 레벨 세부사항을 보고했으며, 이어 qa-critic이 BLOCKER 게이트로 그 구현안 일부를 재정정했다(§8). **§8과 §2~§7이 상충하는 서술은 §8이 우선**한다(실측+게이트 기반 정정이므로). §8 내부에서도 후속 게이트가 선행안을 대체한 경우 해당 절에 명시했다(§8-7).

2026-07-16 취침 직전, 오너가 struct 부트스트랩의 수행 주체를 추가 승인했다(§8-13) — F7a/F7b의 단계 구조 자체는 유지되나, F7b 착수 시점이 "아침"에서 "야간 후반 시도"로 앞당겨질 수 있다.

## 2. 배경

오너 요구 7건:

| # | 요구 |
|---|---|
| ① | 스킬/마법 별도 메뉴 + 하이브리드(예: 성기사=스킬 내려찍기+마법 힐, 최소 1카테고리는 혼합 보유) |
| ② | 타격 없이 버프/디버프/힐만 하는 스킬(예: 신의 축복=아군전체 버프+적군전체 디버프) |
| ③ | status 발동 타이밍이 스킬 전/후를 오갈 수 있어야 함(예: 자기 공격력버프를 걸고 나서 타격) |
| ④ | 효과별 발동턴+확률이 필수 |
| ⑤ | 조건부 효과(예: elf면 무조건 스턴 / HP 50% 이상이면 데미지 2배) |
| ⑥ | **"하드코딩 최대한 회피, 레고처럼 조립"** — governing principle. 이후 추가·수정이 계속된다는 전제 |
| ⑦ | MVP-first — 기본 구조를 먼저 짜고 간단한 스킬부터, 확장은 데이터로 |

**기존 인라인 단일효과 모델**(`skills.csv`의 `EffectType`/`EffectValue`/`EffectDurationTurns`/`EffectChance` 4컬럼, [[상태이상_확정]] 채택분)은 스킬당 효과 1개만 표현 가능해 ②(다중 그룹)·③(타이밍)·⑤(조건)을 표현할 수 없다. → [[상태이상_확정]]이 이미 예고했던 정규화(**안B 마이그레이션**)를 F7에서 실행한다.

### 3-4. 분리 원칙

**상태정의(DT_StatusEffects) = "뭘 하는지"(재사용) / 스킬 효과엔트리(DT_SkillEffects) = "언제·확률·조건으로 어떤 상태를 거는지"(스킬 소유).** 강도·지속은 정의가 소유한다 — 재사용이 성립하는 이유. 커스텀 강도가 필요하면 **신규 상태행을 발급**한다(인라인 오버라이드 금지).

## 4. 판정엔진 (제네릭 인터프리터)

```
ApplySkillEffects(Caster, SkillId, SelectedTargets[]):   ← EnterExecuting이 호출
  엔트리 = DT_SkillEffects[SkillId] 정렬(Timing PRE→ON→POST, Order) 순회:
    targets = ResolveEntryTargets(entry.TargetGroup, Caster, SelectedTargets)  # SELF→[caster]/ALL→풀/SINGLE→SelectedTargets
    ForEach target (bAlive 선체크 — 시체 라이더 방지):
      (bApply, mult) = EvaluateCondition(Caster, target, entry.Cond)   # GATE=적용여부/MULT=값배율, 조건없음=ALWAYS
      if not bApply: continue
      if RandomFloat01() >= EffectResolveChance(entry.Chance): continue  # DebugForceEffectChance 훅 승격
      ApplyEffectEntry(Caster, target, entry, mult)   # switch(EffectType)
```

위는 **개념 레벨** 의사코드다 — 실제 BP 구현 세부(3-패스 버킷·쿨다운/로그 위치·`ResolveHit` 시그니처 등)는 §8을 우선 참고.

### 4-1. 애플리케이터

| EffectType | 구현 |
|---|---|
| DAMAGE | `ResolveHit`(TakeHit §8 step0~9 **통짜 1브릭** — 마이크로 분해 금지, Def감산·막기곱·min1·사망판정은 내부 안정 구현. **정확한 통짜 범위·신규 시그니처는 §8-5 참고**) |
| HEAL | Def·min1 미경유, MaxHp 클램프 |
| STATUS | `ApplyStatus`(channel 캐시 없이 토큰 기록. **막기 채널 케이스의 알파 실제 경로는 §8-7 참고**) |

### 4-2. 채널 훅

| Channel | 훅 지점 |
|---|---|
| SKIP_TURN | `EnterTurnStart` TS4 |
| OUT_MULT | step4(발신) `GetStatMult` — Π(1+V) |
| IN_MULT | step6(수신) |
| DOT | TurnStart 틱(**STUB**) |
| δ차감 | `TickStatusesAtTurnEnd`(기존 그대로) |

### 4-3. Timing이 오너 예시를 성립시킨다

`[(SELF,STATUS:ATK_UP,PRE),(SINGLE_ENEMY,DAMAGE,ON)]` → PRE에서 ATK_UP이 `ActiveStatuses`에 먼저 들어가고, ON의 step4 `GetStatMult`이 실시간 조회 → 증가된 Atk로 타격한다(`GetStatMult`=조회-곱·mutate 없음이라 자연 성립 — 별도 특수 로직 불요). 이 PRE→ON 순서가 §8-7(b)의 "ATK_UP은 skip 마커 없이도 자연 해소"의 근거이기도 하다.

### 4-4. 불변 보존 ([[상태이상_확정]] 승계)

- `bBattleOver`는 `ResolveHit` 내부에서만 세팅(레이스 차단, H-5)
- `EnterTurnEnd` 단일관문
- δ틱(자기 턴 시작 체크 / 자기 턴 종료 차감)
- min1
- step4 곱순서 단일 floor
- 기본공격 even-trade 원장 무손상

### 4-5. ExecPattern과 효과엔트리는 직교

연출축(ExecPattern)과 판정축(효과엔트리)은 분리된 축이다:

```
WalkForward → [PRE 적용] → 모션(HitCount 반복) → [임팩트마다 ON_HIT] → [POST] → WalkBack
```

VFX 훅 정렬: `OnCast`=PRE · `OnProjectile`=모션중 · `OnImpact`=ON_HIT (1:1)

## 5. 레고 원칙 (#1 설계기준, 오너 governing principle)

### 5-1. Zero per-skill 코드

판정엔진에 `SkillId==리터럴` 비교 노드 **0개**(qa 정적 TC). 유일 허용 = 유니버설 폴백 상수(`BasicAttackSkillId` 등, 데이터 멤버).

### 5-2. 개방-폐쇄 심 (축별 1행 표)

새 항목을 추가해도 기존 로직은 무편집:

| 신규 축 | 편집 지점 |
|---|---|
| 새 EffectType | `ApplyEffectEntry` case 1개 |
| 새 Channel | 해당 훅 case 1개 |
| 새 ScaleMode | `ComputeBase` case 1개 |
| 새 TargetGroup | `ResolveEntryTargets` case 1개 |
| 새 Timing | 순회 앵커 1곳 |
| 새 CondProperty | `EvaluateCondition` case 1개 |

**새 스킬 = 코드 0(DT 행만) · 새 상태 = 코드 0(상태 행만).**

### 5-3. 브릭 카탈로그 (IMPL=알파 실배선 / STUB=어휘만+switch default 폴백 로그+무효과)

| 축 | IMPL | STUB |
|---|---|---|
| EffectType | DAMAGE·HEAL·STATUS | STATUS_CLEANSE·SHIELD·DISPEL·SUMMON·TAUNT·REVIVE |
| Channel | SKIP_TURN·OUT_MULT·IN_MULT | DOT·HOT·DEF_MULT·SHIELD_ABSORB·REFLECT·IMMUNE 등 |
| ScaleMode | ATK | FLAT·MAXHP_PCT_TARGET·MAXHP_PCT_CASTER·MISSING_HP_PCT 등 |
| Timing | PRE·ON·POST | ON_KILL·ON_TURN_START·ON_DEATH 등 |
| TargetGroup | SELF·SINGLE_* | ALL_*(A2)·RANDOM_ENEMY·LOWEST_HP_ALLY·DEAD_ALLY 등 |
| Cond | ALWAYS·HP_PCT(+Subject CASTER/TARGET·Op EQ/GE/LT·Result GATE/MULT/NONE) | RACE·STATUS_HAS·TURN_COUNTER 등 |

### 5-4. 미래 무차단

- (a) 다절 조건(AND/OR) → 향후 `DT_Conditions` 분리 + `CondGroupId`(`EvaluateCondition`만 확장)
- (b) 다중그룹(신의 축복) → **이미 성립** — 엔트리 2행(ALL_ALLY 버프 + ALL_ENEMY 디버프), 데이터만 추가
- (c) 신규 채널 → 행 + 훅 1곳

### 5-5. 기존 코드 갭 (실측 발견, F7b 필수 해소)

`TS4`/`GetOutgoingAtkMult`가 현재 상태토큰을 **문자열 리터럴로 비교**하는 구현이라 "새 상태(기존 채널)=코드 0" 약속이 아직 실현되지 않은 상태다. 상세·해결책은 §8-8.

## 8. BP 실측·구현 확정 (gameplay-engineer 실측 + qa-critic BLOCKER 게이트, 2026-07-15 야간 — Director 최종판정)

> §2~§7의 설계를 실제 BP로 구현하기 전, gameplay-engineer가 현재 라이브 그래프(`ResolveHit` 149노드 등)를 실측해 구현안을 제시했고, qa-critic이 BLOCKER/MAJOR 게이트로 이를 재검증해 일부(특히 §8-7)를 대체 정정했다. 아래는 그 **최종 병합판정**이며 **F7b 착수 전 반드시 반영**해야 한다. §2~§7과 상충하는 서술은 이 절이 우선한다. TC 상세는 별도 [[F7_TC]](qa-critic)가 SSOT다.

### 8-1. 엔트리 저장·조회 = RowName 규칙 방식 (신기술 0)

- `DT_SkillEffects` RowName 규칙: `{SkillId}_{N}`(예: `31001000_0`) — 조회는 `GetDataTableRowNames` 열거 + 접두어 매칭 + `GetDataTableColumnAsString` 컬럼 read. **現 `ResolveHit`이 `DT_Skills` 조회에 이미 쓰는 검증된 패턴을 재사용**(신기술 0).
- `DT_StatusEffects`는 `StatusId` 단일행 exact match(더 쉬움).
- **기각안**: FK 배열(TArray 임포트 미검증) · JSON 임포트(CSV 파이프라인 이탈).

### 8-2. DT 데이터 갱신 표준 = `DataTableTools.set_rows` (신규 발견)

`import_file` 재호출의 함정㉙(delete-recreate, 하드레퍼런스 GUID 위험)을 완전 회피하는 신규 표준. 향후 `DT_Skills`/`DT_SkillEffects`/`DT_StatusEffects` 갱신은 전부 이 경로를 쓴다.

### 8-3. Timing 순회 = 3-패스 버킷 + within-ON 저작 규약

PRE→ON→POST 각각 별도 `ForEach` + `EqualExactly(String)` 필터 — 전부 기검증 노드이며 정렬 노드는 쓰지 않는다. 동일 Timing 내 `Order` 서브정렬은 **STUB**(정렬 로직을 만들지 않고 CSV 작성 시 원하는 순서로 행을 배치하는 관례로 커버).

**within-ON 저작 규약(V-rule, qa-critic)**: 동일 Timing(특히 ON) 내에서 RowName `_N` 오름차순 = **DAMAGE 엔트리가 STATUS 엔트리보다 앞**에 오도록 CSV를 작성한다. [[상태이상_확정]] E4(킬링블로우 라이더 미발동 — 대상이 죽었으면 후속 STATUS는 미적용)를 신모델에서도 그대로 유지하기 위한 데이터 규약이며, 인터프리터의 `bAlive` 선체크와 결합해 성립한다.

### 8-4. Condition 컨벤션 + `EvaluateCondition` 계약 (fail-closed)

- `CondProperty` 공란 또는 `ALWAYS` → **선행 명시 분기**로 `(true, 1.0)` 반환(switch의 default 도달이 원천 불가능한 구조로 작성 — 우연한 폴백 의존 금지).
- **미지의 Property/Op**(스키마에 없는 값) → 로그 1줄 + **해당 엔트리 스킵**(fail-closed — 이해 못 하는 조건은 "적용 안 함"으로 안전측 처리, 무조건 적용되는 fail-open 금지).
- `HP_PCT` 스케일 = 0~100 정수(예: `"GE 50"`).
- **MULT 삽입점** = step4 곱 내부(pre-Def, §4-4의 "단일 floor" 불변식과 동일 위치 — Def 감산 전, floor는 곱 전체에 1회만).

### 8-5. DAMAGE 통짜 브릭 범위 정정 — damage core(step4~8) + `ResolveHit` 시그니처

**§6 Director 확정 판정 ③("DAMAGE=TakeHit §8 통짜 브릭")의 정확한 경계.** 現 `ResolveHit`(149노드)은 다음 4조각을 내장하고 있다:
- step1 Effect*4컬럼 read
- step2 HEAL 숏컷
- step3 막기 숏컷
- step8.5 상태롤

F7b에서 이 4조각은 **인터프리터(§4)로 이관·제거**하고, **step4~8(수식·floor·HP적용·사망판정·`bBattleOver` 내부세팅=H-5 유지)만** 통짜로 보존한다.

**신규 시그니처**: `ResolveHit(Attacker, Target, DamageCoeff)` — 내부 DT 조회를 제거하고, 엔트리의 `Value × CondMult`(인터프리터가 계산)를 `DamageCoeff` 인자로 수취한다. `DT_Skills`가 슬림화되며 `PowerRate` 컬럼 자체가 사라지므로 인자화가 필수다(qa BL-3).

★**작업 순서 경고**: `F_SkillsRow` 컬럼 제거와 `ResolveHit` 수술은 **같은 세션에서 연속 실행**해야 한다. 중간 상태(컬럼은 지웠는데 `ResolveHit`은 구식 조회 그대로)로 두면 `GetDataTableColumnAsString`이 빈 문자열을 반환 → `StringToFloat`이 0으로 캐스팅 → 전 상태이상이 무음전멸(발동 자체가 조용히 사라짐)한다. **(2026-07-16 갱신: 이 경고는 §8-13의 additive-first 4단계 순서로 구조적으로 해소됐다 — 구컬럼을 인터프리터 검증 후까지 병존시키므로 이 위험 중간상태 자체가 발생하지 않는다. 아래 서술은 "왜 순서가 중요한지"의 근거로 유지한다.)**

### 8-6. 쿨다운 게이트·세팅 + BattleLog 방출 = `ApplySkillEffects` 캐스트 레벨

step0 게이트(쿨다운 체크)·step9 세팅(쿨다운 갱신)과 **BattleLog 방출**을 `ApplySkillEffects` **최상위(캐스트당 1회)로 이동**한다. 이유:
1. 멀티히트(`HitCount`≥2) 시 엔트리마다 반복 실행되어 자충돌하는 것을 예방.
2. 막기/치유처럼 DAMAGE 엔트리가 없는 캐스트는 舊 구조(step9가 `ResolveHit` 내부)에선 쿨다운도 로그도 아예 발생하지 않았다 — **캐스트당 BattleLog 1줄 원칙**(무DAMAGE 캐스트도 원장 라인을 남긴다)으로 이를 방지.

STATUS 엔트리별 롤을 로그에 어떻게 기록할지(필드 스키마)는 **아직 미확정**이다 — §8-9 참고, F7b 착수 시 확정.

### 8-7. 막기 = 레거시 유지 + ATK_UP 캐스트한정 수명 (qa-critic BLOCKER 게이트 최종 정정)

이 항목은 gameplay-engineer의 최초 실측안("SELF_BUFF 차감skip = `bSkipNextDecrement` 플래그 신설")을 **qa-critic BLOCKER 게이트가 대체**한 최종판정이다.

**(a) 막기(DMG_REDUCE) = 레거시 그대로 유지(BL-6/MJ-4)**: IN_MULT 채널의 알파 구현은 배열(`ActiveStatuses`)에 태우지 않는다. `ApplyStatus`의 **채널 케이스**가 직접 `bBlockActive=true`·`BlockValue=−DefaultValue`를 세팅(StatusId는 리터럴 `0` 취급)하고, 해제는 기존 TS3 로직 **무변경**이다. δ 배열에는 미편입. 다중 IN_MULT(예: DEF_DOWN이 막기와 동시에 걸리는 경우)의 합성은 **베타**로 이월(배열 이주와 함께 처리).

**(b) ATK_UP 수명 = 캐스트 한정, skip 마커 불요**: `Duration=1`, 별도 skip 마커 없이 자연 해소된다 — PRE에서 부여 → **같은 턴** step4가 이를 반영(Timing 순서상 ON이 PRE 다음이므로 만료 전에 이미 소비, §4-3) → 자기 턴 종료 δ 차감으로 만료. 이는 "dur=N → N회" 불변식과 정확히 일치하며, **"다음 자기 턴 공격이 증폭되지 않는 것"이 정답지**다(부여 턴 자체의 공격만 증폭이 의도). 舊 Director 확정 ⑥(§6)의 "당턴 종료 즉시만료 방지" 문구는 **캐스트 한정 버프에는 해당 사항 없음**으로 정정한다 — PRE→ON이 TE2(턴종료 차감)보다 먼저 실행되므로 별도 방지 장치 없이도 이미 안전하다.

**지속형 SELF_BUFF**(여러 턴 유지되는 버프)는 이번 정정의 범위 밖이다 — skip 마커가 필요한 것은 이 케이스뿐이며, **베타 심으로 문서화만** 해둔다(알파 미구현).

**결론**: `F_ActiveStatus`는 (a)(b) 모두 배열 미경유/자연해소이므로 **완전 무변경**이다. `bSkipNextDecrement` 도입안은 보류하며 struct 부트스트랩 목록에서 제외한다(§3-6·§7-2 반영 완료).

### 8-8. 신규 함정 — 채널 조회 리터럴 비교 (F7b 필수 해소)

`TS4`(`EnterTurnStart`)와 `GetOutgoingAtkMult`가 현재 `"STUN"`/`"ATK_DOWN"` **문자열 리터럴 비교**로 구현되어 있다. 이대로 두면 §5(레고 원칙)의 "새 상태(기존 채널)=코드 0" 약속이 거짓이 된다. → **`LookupChannel(StatusToken)`**(`StatusToken` → `DT_StatusEffects.Channel` 조회) 함수로 제네릭화가 필요하다. F7b 필수 작업. (§8-7(a)의 막기 레거시 경로는 배열을 안 쓰므로 이 리팩터 대상이 아니다 — `bBlockActive` 직접 조회는 그대로 둔다.)

### 8-9. 로그 스키마 — 미결 (F7b 착수 전 확정 필요)

[[상태이상_확정]] §8의 `BattleLog` `effect=`/`effectRoll=`/`effectApplied=` 필드 + `effectRoll=-1` sentinel(M3)은 **舊 "1콜=데미지+상태롤" 전제**(`TakeHit` 한 번 호출이 데미지와 상태롤을 동시 처리)로 설계됐다. 신모델(DAMAGE/STATUS가 분리된 엔트리, §8-6의 "캐스트당 1줄" 원칙)에서 **STATUS 엔트리별 롤을 누가 어떻게 이 필드에 기록할지는 아직 미확정**이다 — qa-critic TC([[F7_TC]]) 제작과 함께 F7b 착수 전 확정 예정. ⚠ 이 항목은 "확정"이 아니라 "미결"로 취급한다.

### 8-10. `EnterExecuting` 재배선 = 1노드 스왑 + 캐스트당 1회 호출

실측: `ResolveHit` 콜노드의 `SkillId` 핀이 리터럴 `31000000`으로 미연결 상태. → `ApplySkillEffects` 콜로 교체 + `PendingSkillId` 연결(**1노드 스왑**). `delete_node` 절차 엄수: `find_nodes` 선확인 → uasset 백업 → 1수정1컴파일(함정㉜·㉟ 이력 그래프 참고).

### 8-11. 부호규약 전환 경고 (신규, qa-critic BL-2)

§6 Director 확정 판정 ②(부호규약 통일: 증가+/감소− → ×(1+V))의 데이터 전환 시 위험. 저장값(예: `−0.25`)과 소비 공식(舊 `1−V` → 신 `1+V`)을 **반쪽만 전환**하면 부호가 꼬여 **증폭이 조용히 역전**된다 — 예: 막기가 ×0.5 대신 ×1.5(피해 감소가 증가로), 공격력약화가 ×0.75 대신 ×1.25(디버프가 버프로)가 되며 **에러 없이 조용히** 발생한다. §8-5의 작업순서경고와 동일하게 **데이터 마이그레이션과 공식 변경을 동일 세션에서 연속 실행**해야 한다. **(2026-07-16 갱신: §8-13의 additive-first 순서를 따르면 신규 컬럼·신규 공식이 3단계에서 함께 배선되므로 이 경고도 구조적으로 완화된다 — 단, 3단계 자체(신규 컬럼 값 기입 ↔ 신규 공식 배선)는 여전히 짝을 맞춰야 한다.)**

### 8-13. 부트스트랩 주체·순서 확정 (최종 델타, 오너 승인 2026-07-16 취침 직전)

**주체 변경**: struct 부트스트랩(`F_SkillEffectRow`·`F_StatusEffectRow` 신설, `F_SkillsRow` 컬럼 편집)은 기존 "오너 수동" 원칙(§7-2·§7-1의 F7a/F7b 분할 사유)에서 **Claude 직접 수행(야간 시도)**으로 오너가 승인·변경했다. **실패 시(3회) 폴백 = 아침 오너 수동**(기존 절차 그대로 유지 — 전역 CLAUDE.md "3회 실패 시 중단→오너 호출" 원칙과 정합).

**실행 순서 = additive 먼저 · destructive 마지막** (4단계 — §8-5 작업순서경고를 이 순서로 대체·구체화):
1. 신규 struct 2종(`F_SkillEffectRow`·`F_StatusEffectRow`) + DT 2종(`DT_SkillEffects`·`DT_StatusEffects`) + CSV 3종(§3-6) — **순수 추가(무파괴)**.
2. `F_SkillsRow`에 **컬럼 추가만**(Category·HitCount·PrimaryTargetGroup·FX/SFX 등, §3-1) — **기존 `Effect*`/`PowerRate` 컬럼은 잔존·병존**시킨다(제거하지 않는다). §8-5가 경고한 무음전멸은 "제거" 자체가 유발하므로, 병존시키면 이 단계에서 위험이 발생하지 않는다.
3. 인터프리터(§4) + `ResolveHit` 수술(§8-5) — 신규 컬럼/신규 DT만 소비하도록 재배선. **[[F7_TC]] 게이트 통과 필수.** 이 단계 전에는 구컬럼이 살아있어 舊 판정 경로가 계속 동작한다(회귀 안전망).
4. 구컬럼(`Effect*`/`PowerRate`) 제거 — **3단계 인터프리터 검증 통과 후에만** 실행하는 최종 정리 단계.

이 순서에서는 §8-5가 경고한 위험 중간상태("컬럼은 지웠는데 `ResolveHit`은 구식 조회 그대로")가 **구조적으로 발생하지 않는다**(구컬럼이 4단계까지 살아있으므로). §8-5·§8-11의 "동세션 연속 실행" 경고는 이제 **"4단계를 3단계 검증 전에 앞당기지 말 것"**으로 재해석한다.

**일정 변화**: F7a/F7b **분할 구조 자체는 유지**(§7-1·§7-2)하되, 분할 사유였던 "struct=오너 수동"(→아침 대기) 제약이 해제되어 **F7b가 "아침"에서 "야간 후반 시도"로 이동**한다(부트스트랩이 야간에 성공하는 경우에 한함 — 실패 폴백 시 원래대로 아침). Category 탭(§11-2)도 같은 이유로 **야간 가능 목록**에 포함된다.

**착수 전 필수 선조사 — 함정㉓ 재발 방지**: struct 편집 수단(MCP struct 전용 도구 / Python 스크립트 / 데스크톱 자동화)을 착수 전에 먼저 조사해 확정한다. [[언리얼_MCP_실전노하우]] §19 함정㉓ 이력: `BlueprintTools.create(asset_type=UserDefinedStruct)`(비-Blueprintable 경로)를 호출하면 **블로킹 Slate 모달이 뜨고 300초간 무응답 → 공유 MCP 브릿지 전체(다른 세션 포함)가 마비**되며, Win32/UI Automation 어느 쪽으로도 내용을 읽을 수 없어 오너가 직접 팝업을 닫아야만 복구된다(당시 실측: `/Script/CoreUObject.UserDefinedStruct` 경로 한정 — `/Script/UMG.UserWidget`류는 모달 없이 정상 생성됨). 이 경로는 **재시도 금지** — 대안 수단을 먼저 확보하지 못하면 야간 시도 자체를 보류하고 아침 폴백으로 넘어간다.
