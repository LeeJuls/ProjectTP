## 1. 무엇이 됐나 — 스코프 & 실측 갭 4건 → 보강 4종

**스코프**: 로그만으로 원장(사망·승패·재전투·턴스킵) 재구성. 실측으로 확인된 갭 4건에 보강 4종을 매칭했다.

| # | 갭 | 보강 |
|---|---|---|
| ① | 전투 종료·승패가 로그에 없음 | `State\|turn=<T>\|event=BATTLE_END\|winner=<A/B>` — `EnterEnd`에서 방출 |
| ② | 재전투인지 최초 시작인지 로그로 구분 불가 | `State\|event=INIT\|mode=<FRESH/RESTART>` — `InitBattle` 초입, 舊 `bBattleOver` 값으로 판별 |
| ③ | 어느 타격이 막타(사망)였는지 로그로 특정 불가 | `BattleLog` **사망 라인에만** `\|died=true` append |
| ④ | 사망 유닛의 턴 스킵이 STUN 스킵과 로그상 구분 불가 | `StatusLog\|...\|event=SKIP_TURN_DEATH`(TS1, 기존 STUN 계열 TS5와 구분) |

## 2. 구현 세부 — winner 매핑 · died 무접촉 append

- **winner 매핑**(①): `EnterEnd`에서 `WinningTeam = GetbIsParty(Attacker)` — 마지막 타격의 `Attacker`가 파티 소속이면 `true`(파티 승리 → `winner=A`), 아니면 `winner=B`. 핀 실측으로 확정한 매핑.
- **died 무접촉 append**(③): `SelectString`으로 사망 여부에 따라 `"|died=true"` 또는 **빈 문자열**을 고른 뒤 그대로 Append하는 방식. 비사망 라인은 빈 문자열이 항등적으로 붙으므로 **바이트가 조금도 바뀌지 않는다** — [[상태이상_확정]] §8-1이 정한 "기존 스킬 로그 라인은 바이트 단위 불변" 원칙을 died 필드에도 그대로 적용한 구현.

## 4. 실측 라인 원문 6종

```
State|event=INIT|mode=FRESH
BattleLog|turn=7|attacker=SpawnPoint_Party_A3|target=SpawnPoint_Enemy_B1|action=SLASH|dmg=45|hp=0|died=true
StatusLog|turn=15|unit=SpawnPoint_Enemy_B1|status=DEATH|event=SKIP_TURN_DEATH|remaining=0
State|turn=30|event=BATTLE_END|winner=A
State|event=INIT|mode=RESTART
BattleLog|turn=3|attacker=SpawnPoint_Enemy_B2|target=SpawnPoint_Party_A1|action=ATTACK1|dmg=18|hp=62
```

1줄: 최초 전투 시작(`FRESH`). 2줄: turn7에 B1이 사망 막타를 맞아 `died=true`가 붙음. 3줄: 8유닛 순환 주기(turn7+8=15)로 B1의 턴이 다시 돌아오자 `SKIP_TURN_DEATH` 방출. 4줄: turn30에 B팀 전멸로 `BATTLE_END|winner=A`. 5줄: End 클릭 후 재시작이 `RESTART`로 정확히 표기. 6줄: 재시작 후 정상 타격 라인은 `died` 필드 자체가 **부재**(비사망 무접촉의 실물 증거).

## 5. 이월

- **F9a 기대원장 갱신**: 사망 라인 8줄에 `died=true` 반영(기지 항목 — [[야간큐_TC]] LOG-04 PLAUSIBLE 플래그로 이미 지정돼 있던 작업).
