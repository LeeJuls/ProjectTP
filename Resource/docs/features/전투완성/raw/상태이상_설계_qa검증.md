## 1. BLOCKER (치명 — 확정 전 구현 착수 불가)

### B1. ATK_DOWN 틱 앵커/Duration 의미 정면충돌 — 두 설계자가 서로 상대안을 명시 기각

- **위치**: balance §3-2 틱 타이밍 명세 ↔ system-ui §3-3 D5 실효표.
- **문제**: balance = "피부여자 **자기 턴 종료 시** 잔여 −1. Duration=2 = 자기 턴 **정확히 2회의 행동**이 영향권. 단일화하려고 시작 틱을 쓰면 'Duration=2가 실제 1행동'이 되는 데이터 왜곡(**기각 근거**)". system-ui = "단일 앵커: 자기 턴 **시작**, 체크 후 차감(D5). 실효표: ATK_DOWN dur2 = **약화된 행동 1회**(+1 오프셋)". 즉 **balance가 명시 기각한 바로 그 시작 틱을 system-ui가 채택**했고, system-ui가 기각한 종료 차감(안B)을 balance가 전제했다. 컬럼명 불일치와 달리 이건 병합 지시에서 안 잡혔다.
- **재현 시나리오**: system-ui 구조대로 구현 + balance 데이터(파볼 ATK_DOWN dur2) 임포트 → 부여(적 턴 step8.5) → 피부여자 턴1 시작 P5에서 2→1(약화 행동 1회) → 턴2 시작 P5에서 1→0 제거 → 턴2 행동은 정상. **실효 = 약화 행동 1회** = balance가 "기대 절감 ~10~13, 체감 미달"로 **명시 기각한 dur1 상당의 실효**. 결과: §5-3 킷 EV 등가 설계 붕괴(마법사 킷 EV 반토막 → 킷 비대칭), TC-SE03("자기 턴 2회, 3회째 원복")은 구현 직후 FAIL.
- **파생 2차 피해**: balance D6-안A의 카운터플레이("약화 턴 치유로 흘리기")도 시작 틱에서는 "흘리기"가 아니라 **완전 회피**(약화 행동 0회)가 됨 — 스킬 선택 유의미화라는 설계 의도까지 변질.
- **V3(dur≥2)는 이 충돌을 못 막는다**: V3는 dur1 무효과의 하한 가드일 뿐, "dur2의 의미가 문서마다 2배 다름"은 통과시킨다.
- **해소 옵션** (채택 = Director):
  - **δ(권장): 전 채널 균일 "시작 체크·종료 차감"** — 자기 턴 시작 P4에서 활성 판정만(STUN 스킵·ATK_DOWN 배율은 잔여>0 기준), 차감·EXPIRE는 자기 턴 종료(EnterTurnEnd, ActiveUnit 한정·인덱스++ 이전)에서. 검산: STUN dur1=1스킵 ✓(스킵 꼬리도 EnterTurnEnd 경유라 차감 실행), ATK_DOWN dur2=행동 2회 ✓ = **balance §3-2 문면("2회째 자기 턴 종료 시 해제") 그대로**. system-ui가 기각한 "채널별 앵커 2종"이 아니라 채널 무관 단일 규칙 — 기각 근거(채널 분기)가 적용 안 됨. 비용: 관문(F5-2)에 무분기 차감 스윕 1개(관문 얇게 원칙과의 트레이드는 designer 재가 필요). 부수 이득: 실효표가 직관(dur N=N회)이 되고 V3가 함정 가드에서 밸런스 취향으로 강등.
  - β: system-ui 시작 틱 유지 + 데이터를 dur3으로 상향(실효 2행동) — balance 문서 §3-2 근거·매트릭스·TC 전면 재작성 + "Duration 숫자≠실효"라는 영구 데이터 왜곡. 비권장.
  - γ: system-ui 안B(발신 채널만 종료 차감) — 채널별 앵커 2종. δ가 같은 효과를 균일 규칙로 얻으므로 열등.
- **담당**: Director 확정 → balance(수치 문서 정합 확인)·system-ui(§3-3 파이프라인 개정)·gameplay(구현).

### B2. EffectValue 의미 충돌 — 같은 컬럼 값 0.75가 ×0.75(balance) vs ×0.25(system-ui 공식)

- **위치**: balance §3 카탈로그(ATK_DOWN EffectValue=**0.75** = 배율 그 자체, §6-2 "AtkDownMult = 0.75") + §7-2 CSV diff(파볼 행 `ATK_DOWN,0.75,2,0.35`) ↔ system-ui §3-2 채널 계약("Value 의미: **감소율 r**, 배율=(1−r)") + §5(`OutgoingAtkMult = Π(1−Value)`) + §11-1 V6("r=1은 피해 0 홀이므로 상한 exclusive" — 감소율 해석에서만 성립하는 근거).
- **재현 시나리오**: balance CSV(0.75)를 그대로 임포트 + system-ui 공식(×(1−V)) 구현 → 약화 전사 기본공격 = floor(40×(1−0.75)×1.0) = 10, −Def10 = 0 → min1 = **1**. 의도(30→20, −33%)가 아니라 **−97% 붕괴 + 칩뎀 교착 재현**(balance가 ×0.5조차 "게임 결정 크기"로 기각했던 것보다 심함). 손계산 정답지(약화 매트릭스 43/47 등) 전 셀 미스매치 → L1 TC 전멸.
- **V6가 못 잡는다**: V6 "ATK_DOWN ∈ (0,1)"은 0.75도 0.25도 통과 — 의미(감소율/배율) 명시 없는 범위 검사는 이 오류에 무력.
- **판정 근거(어느 해석이 기존 계약인가)**: v1 §8 step6 기존 막기 공식 = `dmg × (1 − BuffValue)`, EffectValue=0.5 — **기존 컬럼 관례는 감소율**. system-ui가 관례 정합, balance가 이탈.
- **권장 처리**: **감소율 통일 + 파이어볼 행 EffectValue = 0.25로 정정**(효과는 동일한 ×0.75). float 안전도 동일(1−0.25=0.75 이진 정확). balance 문서 §3 표·§6-2 식·§7-2 CSV diff 3곳 정정 + V6에 "의미=감소율" 1줄 명시. 대안(배율 통일)은 막기 0.5가 우연히 대칭값이라 데이터는 무변경이나 v1 §8 공식 문면 개정이 필요해 접촉면이 더 큼.
- **담당**: Director 확정 → balance(문서·CSV diff 정정).

### M2. step4 곱 순서 두 문서 불일치 — float-floor 방어책의 자기 무효화

- **위치**: balance §6-2 `Atk × AtkDownMult × PowerRate × BerserkMult`(+"곱 순서 고정" 명시, §6-4 "전 셀 float 검산 완료"는 이 순서 기준) ↔ system-ui §5 `Atk × PowerRate × BerserkMult × OutgoingAtkMult`.
- **문제**: 수학적으론 교환법칙으로 동일하나 IEEE754 곱은 순서에 따라 중간 반올림이 달라질 수 있고, balance 스스로 "곱 순서 고정"을 float 함정(§6-4, 0.7 실함정) 방어책으로 선언했다. 두 문서가 **서로 다른 순서를 각자 "고정"** — 방어책이 계약 차원에서 무효.
- **권장**: **balance 순서 채택**(검산이 완료된 순서 + AtkDownMult(=1−0.25=0.75, 이진 정확)를 정수 Atk 직후에 곱해 오차 유입 최소). system-ui §5·F8 문안(F1 §7 표기)에 반영. 구현 후 TC-QA05로 노드 순서 실증.

### M3. 로그 계약 자기모순 — "effectRoll 항상 기록" vs "킬링 블로우는 롤 자체를 생략"

- **위치**: system-ui §7-1("effectRoll을 **항상 기록**") ↔ §6 step8.5a + balance §6-2("사망 시 **롤 생략**").
- **문제**: 대상 사망(8.5a)이면 굴리지 않으므로 기록할 roll 값이 없다 — 구현자가 임의 결정(0? −1? 필드 생략?)하게 되고 verifier 파서가 라인마다 다른 형태를 만난다. §7-1의 사망 구분법(hp=0 ∧ applied=false) 자체는 성립하나 roll 필드 표기가 미정.
- **권장**: "8.5a 생략 시 `effectRoll=-1`(불가능 값) 기록"으로 확정 — 필드 유무 분기 없이 파서 균일 + `-1`은 (roll<Chance) 감사에서 자동 제외 가능. 대안(필드 생략)도 가하나 옵션 필드가 또 조건부가 되어 파서 분기 증가.

## 3. 종결 판정 2건 (순환참조 해소 — 추가결정 불요)

### 3-1. [#4] 기절 스킵 TurnCounter = +1 확정 (CONFIRMED)

두 설계안이 서로에게 "확인 1건"으로 미룬 것을 본 검증이 종결한다. 근거 3축:
1. **의미 일관**: F1 §2의 카운트 기준을 "살아있는 유닛에게 턴이 배분·소비되면 +1, 죽어서 배분 자체가 없으면 미증가"로 읽으면 — 막기·치유(+1, 수비 행동도 소모)와 기절(행동 강제 낭비)은 같은 편, 사망(큐의 유령 슬롯)만 반대편. 기절 미증가 시 스턴 다용 게임의 광폭화 램프가 늦어져 anti-stall 역행.
2. **구현 자연**: system-ui P2(+1)가 P4(기절 판정)보다 앞 = 기존 `EnterTurnStart` IsValid 경로 증가 지점 그대로 — 추가 코드 0.
3. **원장 무영향**: F9a 23턴 even-trade는 기본공격 전용(EffectType=NONE, 8.5 미진입) → 프록 0 → 기절 0. balance §6-3 종결 재검(임계 ≤45 유한)도 완료 — 기절은 수비자가 공격해야만 나오므로 스톨 조력 불가.
- **후속(문서만)**: F1 §2에 "기절 스킵은 사망 스킵과 달리 +1" 1줄 추가(balance). TC-SE07의 "(D8 채택 시)" 조건부 삭제 → 무조건 TC로 승격.
- **파생 발견(m8)**: 기절 스킵 턴은 카운터를 먹되 BattleLog 라인이 없다 → **로그 원장에 turn 번호 갭 발생**(기존 사망 스킵은 번호 자체를 안 먹어 갭이 없었음 — 신규 현상). F9a 대조 로직이 턴 연속성을 가정하면 FAIL → TC-QA07로 명시 검증.

### 3-2. [#6] 무한 기절 락 — G1+G2 구조 차단 성립 (CONFIRMED)

- **논증**: 기절 부여는 ON_HIT 라이더 = 반드시 스킬 시전(행동) 필요(불변식②) + G1(ON_HIT⇒PR>0)로 매 부여가 min1 이상 피해 동반 + G2(중첩 금지)로 duration 축적 경로 차단 + "스킵은 반드시 −1"(§4-2) → 락 유지 행위 자체가 HP를 단조 소모 + TurnCounter는 계속 증가(§3-1)해 광폭화 램프 가속 → **유한 종결**. 락을 안 걸면(공격 0) 기절도 0 — 즉 기절 락은 기존 순수-수동 홀(F1 §6, A1 무해 판정)의 **특수형으로 환원**되며 신규 구멍이 아니다.
- **데이터 실확인**: 베기 PR1.3>0 ✓ 파볼 PR1.7>0 ✓ 기본공격 EffectType=NONE(cd0 부착 금지) ✓ — 알파 5행은 G1 기만족.
- **잔여 명시 2건(MINOR)**: ① G3(cd≥dur+1)는 베기(cd1, dur1)가 **불만족**인데 어느 문서도 "미채택"을 선언 안 함 — balance §5-4의 확률 게이팅(연속 P≤6.3%)으로 갈음함을 병합 문서에 1줄 명시(m11). ② Chance=1.0 강제 테스트 중에는 확률 게이트가 사라져 베기 반복 = 인위 영구 락 가능 — TC 시나리오 통제 필요(TC-QA12).

## 검증 축과 통과 근거 (빈손 방지 조항)
