## 버그 발견 — lat 부호반전 로직이 반전 케이스(적 턴)에서 0벡터를 낳음

### 증상
V2 구축 직후 1차 PIE 실측에서 CT3-02(camLoc 공식 일치 ±1cm) 검증 중 **아군 턴(A1)은 공식과 정확히 일치(오차 0.0004cm)하지만 적 턴(B1)은 242cm 오차**가 재현됨. Z값은 정확히 일치하는데 X/Y만 크게 어긋나는 특징 — CamHeight 항은 정상, lat 관련 항만 이상 신호.

### 진단 (§4 방법론 — 가설 반증 실험)
로그의 실측 camLoc에서 역산(`lat*CamLateral = camLoc - A + axis*CamBack - (0,0,CamHeight)`)한 결과 **implied lat의 norm이 2.86e-6**(사실상 완전한 0벡터)로 확정 — 부호가 반대로 적용된 게 아니라 lat 자체가 소거되고 있음을 확인. 임시 진단 스캐폴드(`select_lat.ReturnValue → ToString(Vector) → PrintString`, §9 노하우 "임시 스캐폴드 심고 검증 후 제거" 정석 패턴)를 `Branch(len_less).else`와 `SetActorLocation` 사이에 삽입해 실측한 결과: 아군 턴(반전 불필요 케이스) `X=0.251 Y=-0.968 Z=0.000`(정상) / 적 턴(반전 필요 케이스) `X=-0.000 Y=0.000 Z=0.000`(완전 0벡터, 확정).

### 근본 원인 ①(1차 수정 시도, 부분 효과) — `gt_zero`(latY>0) 프로모터블 비교 노드의 B핀 리터럴 미확정
`유틸리티|연산자|보다큼(>)`을 승격시킨 `gt_zero` 노드의 B핀(비교 대상 `0`)에 `set_pin_value`를 호출하지 않아 값이 확정되지 않은 채 방치됐던 것을 발견 → `0.0`으로 명시 세팅. 이 수정은 근본 원인이 아니었음이 재검증(값 불변)으로 확인됨 — 그러나 리터럴 미확정 자체는 잠재 버그이므로 수정을 유지.

### 근본 원인 ②(확정, 실제 버그) — 스칼라곱 의도의 `set_pin_value(B, '-1.0')`이 A핀(Vector) 연결 이후 호출되며 Vector×Vector로 승격, 스칼라 문자열이 Vector로 파싱 실패해 `(0,0,0)`으로 소실
`lat_flip_mul = normalize_lat * -1.0`(반전 벡터 계산)을 프로모터블 곱셈 노드로 만들 때, **A핀(Vector, `normalize_lat`)을 먼저 연결한 뒤 B핀에 `set_pin_value(-1.0)`을 호출**했다. 이 시점 노드는 이미 A핀 타입(Vector)에 맞춰 `수학|벡터|vector*vector`로 승격되어 있었고, B핀도 Vector 타입이 되어 `-1.0`이라는 스칼라 문자열이 Vector 파싱에 실패, 값이 소실되어 사실상 `(0,0,0)`으로 남았다. `get_node_infos` 재조회로 `PromotableOperator_17`의 `type_id`가 `수학|벡터|vector*vector`(스칼라곱이 아님)임을 확인해 확정.

**해법**: 스칼라곱 대신 **뺄셈으로 반전을 표현**(`flipped = ZeroVector - lat`, Vector-Vector 연산 — 이미 다른 곳(`mul_axis_back` 등)에서 검증된 안전한 승격 패턴과 동일 계열). 기존 곱셈 노드 삭제 → `유틸리티|연산자|빼기` 신규 생성 → `MakeVector(0,0,0)`을 A, `normalize_lat`을 B로 연결 → `수학|벡터|vector-vector`로 정상 승격 확인(재조회) → `select_lat.B`에 연결.

### 검증(재계산)
수정 후 실측: 아군 턴 `cam=X=-1296.068 Y=-7709.821 Z=885.508`(변화 없음, 이미 정상), 적 턴 `cam=X=1380.676 Y=-7015.988 Z=869.273`(수정 전 `X=1317.947 Y=-6773.986`에서 변경). Python 재계산 대조: 아군 턴 오차 0.0004cm, 적 턴 오차 0.0008cm — 둘 다 ±1cm 이내로 확정 통과.

### 진단 스캐폴드 정리
`break_pins(Branch.else, diag_print.execute)` → `connect_pins(Branch.else, SetActorLocation.execute)`(직결 복원) → `delete_node`×2(diag_print, diag_tostr_lat). 재컴파일 0 에러/0 경고 확인.

## 신규 함정 — `언리얼_MCP_실전노하우.md`에 추가 권장 (함정⑬ 후보)
**프로모터블 오퍼레이터의 A핀을 먼저 연결한 뒤 B핀에 `set_pin_value`로 스칼라 리터럴을 세팅하면, 노드가 이미 A핀 타입(예: Vector)으로 승격되어 있어 B핀도 같은 타입으로 승격되고, 스칼라 문자열이 그 타입으로 파싱 실패해 조용히 소실(값 없음 → 사실상 0)될 수 있다.** 에러도 경고도 발생하지 않고 `set_pin_value` 자체는 `returnValue: null`(성공)을 반환하므로 이 실패는 **`get_node_infos` 재조회로 B핀의 실제 값과 타입을 확인하지 않으면 감지 불가**하다. "Vector×Float 스칼라곱" 의도라면, A핀 연결 전에 B핀을 먼저 float 리터럴로 세팅하거나(타입 아직 미확정 상태), 혹은 안전하게 **Vector-Vector 뺄셈(`ZeroVector - v`)으로 부호반전을 표현**하는 편이 이 함정을 원천 회피한다. 이번 세션에서 이 버그가 "정상처럼 보이는 케이스(아군 턴)"와 "실패하는 케이스(적 턴)"를 모두 만들어냈다는 점이 특히 위험함(양쪽 다 컴파일 0 에러, 실행도 크래시 없이 조용히 틀린 좌표만 냄) — **동일 로직의 대칭 케이스(양쪽 팀·양쪽 부호)를 반드시 함께 실측 대조**하는 것이 이런 함정의 유일한 실전 발견법이다(이번 세션의 CT3-02가 정확히 이 대조 덕에 잡음).
