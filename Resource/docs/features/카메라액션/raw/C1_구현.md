## 5. EnterExecuting 배선 (카메라 컷)

### 실그래프 확정 조회
`find_nodes(entry_points_only=true)` + `get_node_infos`로 `EnterExecuting`(K2Node_CustomEvent_2) 확인. `get_connected_subgraph`로 전체 체인 추적(exec `connected_pins` 기반, type_id 표시 문자열이 아닌 실제 배선으로 추적) — plan이 지시한 삽입 지점("HideAll 직후, WalkForward 앞")은 정확히 `HideAll(23).then → PrintString_169("ExecWalkPhase").then → WalkForward(165).execute` 구간으로 확정.

### 배선 체인
```
PrintString_169.then
  → IsValid(PC)
     [Is Not Valid] → WalkForward(스킵①)
     [Is Valid] → IsValid(SelectObject(A=GetActionCamParty, B=GetActionCamEnemy, bSelectA=GetIsParty(ActiveUnit)))
        [Is Not Valid] → WalkForward(스킵②)
        [Is Valid] → CastToActor(Object=SelectObject.ReturnValue)
           [CastFailed] → WalkForward(스킵③, 안전망)
           [then] → SetViewTargetWithBlend(self=PC, NewViewTarget=CastToActor.AsActor, BlendTime=0.4, BlendFunc=EaseInOut)
                    → PrintString("CamCut:{GetDisplayName(선택캠)}:t={GameTime}", bPrintToScreen=false, bPrintToLog=true)
                    → WalkForward(정상)
```

### 함정 발견 — `유틸리티|SelectObject`의 `ReturnValue`는 A/B 입력이 Actor 타입으로 연결돼도 자동 승격되지 않음
§9/§14 노하우의 "프로모터블 오퍼레이터는 데이터 연결 후 자동 재해석"이 이번엔 **부분적으로만** 성립했다. `SelectObject`(비교 오퍼레이터가 아니라 Object 조건 선택 노드)는 A/B 입력 핀은 연결된 데이터(Actor 타입)에 맞춰 정확히 `오브젝트 레퍼런스`→구체 타입으로 갱신되지만, **출력(ReturnValue)은 제네릭 `오브젝트 레퍼런스`(UObject)로 고정**되어 있어 `SetViewTargetWithBlend.NewViewTarget`(Actor 특화 타입)에 직결 시 `"Could not connect pin ReturnValue to NewViewTarget. The pins may be incompatible types."` 에러 발생.
**해법**: `Utilities|Casting|CastToActor`(impure, exec 포함)를 `SelectObject.ReturnValue → CastToActor.Object → CastToActor.AsActor → NewViewTarget` 형태로 삽입. `CastFailed` 분기는 WalkForward로 안전 폴백(이론상 도달 불가 — Actor를 다시 Actor로 캐스트하는 것이므로 항상 성공하지만, 안전망으로 유지). 이 함정은 이후 §스캐폴드에서 `ScaffoldTarget`(Actor 타입 변수)을 `NotifyUnitClicked.ClickedUnit`(BP_BattleSpawnPoint 특화 타입)에 연결할 때도 동일하게 재현 — `Utilities|Casting|CastToBP_BattleSpawnPoint`로 동일 해법 적용.

### `FormatText` 로그 조립
`GetDisplayName(선택캠)` + `GetGameTimeInSeconds()` → `FormatText("CamCut:{0}:t={1}")` → `ToString(Text)` → `PrintString.InString`. `create_node`로 막 생성한 FormatText는 인자 핀 2개(idx1="0", idx2="1")가 기본으로 존재(이번 세션에서 실측 확인 — 문서에 따라 0개로 시작하는 경우도 있었으므로(EnterTurnEnd 쪽 사례) 매번 재조회 후 필요 개수만큼 `add_node_pin`/`remove_node_pin`으로 맞추는 것이 안전).

## 6. EnterTurnEnd 배선 (카메라 복귀)

### 실그래프 확정 조회
`EnterTurnEnd`(K2Node_CustomEvent_1)의 `then`이 `SetBattleState(VariableSet_3)`에 직결됨을 `get_connected_subgraph`로 확인 — plan이 지시한 "진입 액션, 마커 OFF 앞이든 뒤든 첫머리"는 정확히 이 지점(EnterTurnEnd 자체의 최초 진입부).

### 배선 체인
```
EnterTurnEnd.then
  → IsValid(PC)
     [Is Not Valid] → SetBattleState(스킵)
     [Is Valid] → SetViewTargetWithBlend(self=PC, NewViewTarget=GetDefaultCamera, BlendTime=0.3, BlendFunc=EaseInOut)
                  → PrintString("CamBack:t={GameTime}", bPrintToScreen=false, bPrintToLog=true)
                  → SetBattleState(정상)
```
`DefaultCamera`는 이미 Actor 타입 멤버 변수 게터라 `NewViewTarget`에 직결 성공(CastToActor 불필요 — §5의 SelectObject 특유 함정과 무관, 순수 변수 게터는 타입이 정확).

## 7. 미니 TC 스캐폴드 (2턴, W2/W3fix 패턴 재사용)

```
BeginPlay → Delay(3.0) → NotifyAttackButtonClicked() → Delay(0.3)
  → [ForEachLoopWithBreak(GetTurnQueue()) + Branch(GetIsParty(Element)≠GetIsParty(GetActiveUnit)) → SetScaffoldTarget(Element)→Break]
  → CastToBP_BattleSpawnPoint(ScaffoldTarget) → NotifyUnitClicked(ClickedUnit=캐스트결과)
  → Delay(4.0) → NotifyAttackButtonClicked() → Delay(0.3) → [동일 탐색 2회차] → CastToBP_BattleSpawnPoint → NotifyUnitClicked(...)
```

### 함정 발견 — 2턴차 Delay 값 최초 시도(2.4초)가 실제 1턴 소요시간(약 3초)보다 짧아 `bInputLocked` 가드에 막힘
최초 `Delay(2.4)`로 시도하자 로그에 `NotifyAttackButtonClicked: BLOCKED (bInputLocked=true)`가 재현됨(에러 없이 조용히 무시되는 패턴이라 처음엔 "게임 클럭이 정지했다"고 오판할 뻔함 — §9 함정⑦ 재현 정황과 표면적으로 유사해 혼동 위험 있었음). 근본 원인은 `PlayAttack` 내부 애니메이션 타이머(W2 문서 기존 발견 사항, 이번 세션이 신규 발견한 게 아님)로 인해 1턴 전체(Executing 시작~AwaitCommand 재진입)가 plan 예상보다 오래 걸리기 때문(실측 약 3.0초, W2가 이미 문서화한 "PlayAttack 완료 지연" 그대로).
**해법**: `Delay`를 2.4→4.0초로 여유있게 증가. 재시도 시 정확히 재현됨(`CamCut:ActionCam_Party:t=3.4` → `CamCut:ActionCam_Enemy:t=7.733` 순서로 2턴 모두 정상 진행).

### 함정 재확인 — MCP 클라이언트가 에디터와 반복 통신하는 동안 실제 OS 전경 창이 백그라운드(Claude 창)로 밀려 게임 틱이 사실상 정지
초기 진단 시도(창 전면화 전)에서 40초+ 순수 대기에도 게임 시간이 t=6.4에서 전혀 진행되지 않는 현상 관찰. PowerShell `user32.dll`(`GetForegroundWindow`)로 확인한 결과 전경 창이 "Claude"(에이전트 작업 창)였음(§9 함정⑦의 원인 후보 재확인). `ShowWindow`+`SetForegroundWindow`로 `projectTP RUNNING - DO NOT CLOSE`(실제 PIE 게임 창)를 전면화했으나 이번 사례에서는 이것만으로는 진행이 재개되지 않았고, **위 "Delay 부족" 원인이 실제 근본원인**이었음(Delay 4.0 수정 후에는 창 전면화 없이도 30초 백그라운드 대기만으로 2턴 모두 정상 완주). 즉 이번 세션에서는 두 현상(창 전경 여부, Delay 부족)이 동시에 관찰돼 혼동 위험이 있었으나, 최종적으로는 **Delay 부족이 유일한 근본원인**으로 확정(창 전경 여부는 무관했던 것으로 재현 실험상 판명 — 정정 사항으로 §9 문서에 참고 부기 권장, 이번 세션 범위 밖).

## 9. 스캐폴드 정리

`BeginPlay.then → Delay(3.0)` 연결을 `break_pins`로 절단 후 나머지 스캐폴드 노드 28개(1턴차 14개+2턴차 14개) 전부 `delete_node`로 제거, 멤버 변수 `ScaffoldTarget`도 `remove_variable`로 제거. 재조회로 `BeginPlay.then.connected_pins:[]`(완전 원복, W3fix 종료 시점과 동일 상태) 확인. 재컴파일(`warnings_as_errors=true`) 0 에러/0 경고.

핵심 배선(EnterExecuting/EnterTurnEnd 카메라 컷·복귀)은 `get_connected_subgraph` 재조회로 스캐폴드 제거 후에도 정확히 보존됨을 최종 확인(총 59개 노드, 3개 폴백 경로 전부 WalkForward로 정상 합류).

## 저장
`save_assets([])` → `map_battle_octopath` is_dirty=false, `BP_BattleManager` is_dirty=false 확인.

## 알려진 이슈/제약
