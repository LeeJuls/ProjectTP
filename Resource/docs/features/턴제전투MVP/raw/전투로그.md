## 목적·설계

BP-only 제약(BP엔 파일쓰기 노드가 없음) 하에서 전투 로그를 남기는 방식으로 **"UE 로그에 구조화 라인 + 추출 스크립트"** 를 채택했다. BP_BattleManager가 매 공격 실행 시 `BattleLog|` 프리픽스가 붙은 파싱 친화적 라인을 로그 전용(Print to Screen=false, Log=true)으로 남기고, 별도 파이썬 스크립트가 UE 엔진 로그에서 이 라인만 걸러내 전용 파일로 저장한다.

## 라인 포맷

```
BattleLog|turn=<TurnCounter>|attacker=<ActiveUnit 라벨>|target=<SelectedTarget 라벨>|action=ATTACK1
```

- `turn`: `TurnCounter`(int, BP_BattleManager 신규 변수). `InitBattle`에서 0으로 리셋, 실제로 턴이 열리는 경우(`EnterTurnStart`의 `IsValid(ActiveUnit)` 분기 중 **Is Valid** 경로)에만 +1. Is Not Valid(null 스킵) 경로는 증가하지 않음 — "실제로 진행된 턴"만 카운트.
- `attacker`/`target`: `유틸리티|GetDisplayName(Object=...)` 결과(액터 `GetDisplayName()`, 예: `SpawnPoint_Party_A1`). 라벨은 실측 SpawnPoint 액터 라벨과 완전히 일치함(검증 로그 참고).
- `action`: 현재 MVP 범위(공격 1종, 데미지·스킬 분화 없음)라 항상 `ATTACK1` 고정.
- `|` 구분자로 필드를 나누는 이유는 파싱 용이성(추출 스크립트가 라인을 그대로 통과시키므로, 후속 분석 스크립트에서 `line.split('|')`로 바로 key=value 딕셔너리화 가능).

## 구현 위치 (BP_BattleManager)

- **`EnterExecuting`**(Custom Event, EventGraph — Delay 필요해 함수 그래프 불가, 기존 [[E2_상태머신]] 결정 계승): 진입 직후(`SetBattleState(4)` 이전)에 BattleLog PrintString을 삽입. 즉 상태 전이보다 로그 기록이 먼저 실행됨.
  - 흐름: `CustomEvent(EnterExecuting) → PrintString(BattleLog) → SetBattleState(4) → ...`(기존 로직 그대로 유지).
  - PrintString 구성: `GetActiveUnit`→`GetDisplayName` / `GetSelectedTarget`→`GetDisplayName` / `GetTurnCounter` 3개 값을 `Utilities|Text|포맷텍스트`(FormatText)의 와일드카드 인자 핀 `{0}`(turn, Integer 자동승격)·`{1}`(attacker, String)·`{2}`(target, String)에 연결 → `ToString(Text)` → `PrintString(bPrintToScreen=false, bPrintToLog=true)`.
- **`InitBattle`**(Function Graph): `SetCurrentIndex(0)` 직후, `SetbInputLocked(false)` 이전에 `SetTurnCounter(0)` 삽입.
- **`EnterTurnStart`**(Function Graph): `IsValid(ActiveUnit)`의 **Is Valid** 분기 진입 직후, `MarkerOn(self=ActiveUnit)` 이전에 `SetTurnCounter(GetTurnCounter + 1)` 삽입(`수학|인티저|int+int` 프로모터블 연산자, 리터럴 B=1).

컴파일 `warnings_as_errors=true` 확인 — 에러 0.

## 검증 (임시 스캐폴드 + PIE)

지시서 규정대로 **유닛 리터럴 금지**(§9 함정⑥ — `set_pin_value`로 세팅한 리터럴 오브젝트 레퍼런스는 PIE 중 에디터 원본 "그림자" 액터를 계속 가리켜 가짜 회귀를 낳음) 원칙을 지켜, 스캐폴드의 클릭 대상은 전부 `GetTurnQueue()` → `Utilities|Array|Get(사본)`으로 배열 인덱스 접근했다.

**스캐폴드**: `BeginPlay → Delay(3.0) → NotifyAttackButtonClicked → Delay(0.3) → NotifyUnitClicked(TurnQueue[1]=B1) → Delay(2.5) → NotifyAttackButtonClicked → Delay(0.3) → NotifyUnitClicked(TurnQueue[0]=A1)`(2턴).

**절차**: `IsPIERunning` 확인(false) → `bThrottleCPUWhenNotForeground`를 진단용으로 `false`로 임시 변경([[언리얼_MCP_실전노하우]] §9 함정⑦ 대응, 켜진 채로는 Delay가 사실상 정지) → `StartPIE(warmup=2s)` → 약 9초 대기 → `GetLogEntries(category="LogBlueprintUserMessages", pattern="BattleLog\\|")`로 확인 → `StopPIE` → `bThrottleCPUWhenNotForeground` 원복(`true`) → 스캐폴드 12개 노드 전부 `delete_node`로 제거, BeginPlay exec 미연결 원상복구 → 재컴파일 에러 0.

## 확장 여지 (향후 데미지/스킬 붙을 때)

- `action` 필드가 현재 `ATTACK1` 고정인데, 스킬 다양화 시 `EnterExecuting`이 아는 스킬 식별자(스킬 ID/이름)를 FormatText에 추가 인자로 붙이면 된다.
- 데미지/힐 적용 로직이 붙으면 `|damage=<값>` 또는 `|heal=<값>` 필드를 같은 FormatText 포맷 문자열에 이어 붙이는 방식으로 확장 — 추출 스크립트는 라인 전체를 그대로 통과시키므로 수정 불필요.
- 상태이상(버프/디버프)이 생기면 `|status=<이름>` 필드 추가 고려.
- 분석 스크립트가 필요해지면(단순 추출이 아니라 파싱·집계) `line.split('|')`로 `key=value` 쌍을 딕셔너리화하는 헬퍼를 추출 스크립트나 별도 분석 스크립트에 추가할 것.
