# E1 유닛 확장 — 마커/클릭박스 배치 최종 해결 (gameplay-engineer)

> 상위: [[../plan|턴제전투MVP plan]] · [[../청사진|청사진]]
> 이 문서는 **직전 E1 에이전트가 3회 중단 규칙으로 멈춘 지점**(마커/클릭박스 인스턴스 세팅 실패)부터 이어받아 완료한 기록이다.
#projectTP/턴제전투MVP

## 직전 구현(재확인, 변경 없음)
BP_BattleSpawnPoint에 TurnMarker(Plane+M_UI_TurnMarker)·ClickBox(Box) 컴포넌트, TakeHit/MarkerOn/MarkerOff/HighlightOn/HighlightOff 이벤트, bIsParty/ManagerRef 변수 — 전부 기존 그대로. BeginPlay 체인(Sprite MID 생성 → SetSpriteMID → IsValid(ManagerRef) → RegisterUnitReady)도 기존 그대로이며, 이번 작업은 **RegisterUnitReady.then 뒤에 신규 로직만 이어붙임**(create_node+connect_pins 수동 배선, exec fan-out 없음).

## 이번 구현 — BeginPlay 확장 (Director 확정 해법 그대로)
### 신규 변수
- `TurnMarkerMID`(MaterialInstanceDynamic 오브젝트 레퍼런스) — TurnMarker 전용 MID 저장용. 기존 `SpriteMID`는 HighlightOn/Off가 계속 사용(변경 없음).

### BeginPlay 체인 최종 구성 (RegisterUnitReady.then 이후)
```
RegisterUnitReady.then
→ CreateDynamicMaterialInstance(self=TurnMarker, SourceMaterial=GetMaterial(TurnMarker,0)) .then
→ SetTurnMarkerMID(TurnMarkerMID = 위 반환값) .then
→ Branch(Condition=GetIsParty)
   True  → SetVectorParameterValue(self=GetTurnMarkerMID, ParameterName="MarkerColor", Value=MakeLinearColor(0,0.8,1,1))  .then ─┐
   False → SetVectorParameterValue(self=GetTurnMarkerMID, ParameterName="MarkerColor", Value=MakeLinearColor(1,0.5,0,1))  .then ─┤ (fan-in, 표준 Branch 합류 패턴)
                                                                                                                                  ↓
→ SetWorldRotation(self=GetTurnMarker, NewRotation=MakeRotator(Pitch=90,Yaw=84,Roll=0)) .then
→ SetWorldLocation(self=GetTurnMarker, NewLocation = GetActorLocation + MakeVector(0,-60,-115)) .then   [Vector+Vector는 유틸리티|연산자|추가 프로모터블 오퍼레이터, Vector 핀 연결 시 자동 "수학|벡터|vector+vector"로 승격]
→ SetWorldScale3D(self=GetTurnMarker, NewScale=MakeVector(0.5,0.35,1)) .then
→ SetWorldRotation(self=GetClickBox, NewRotation=MakeRotator(0,0,0)) .then
→ SetWorldLocation(self=GetClickBox, NewLocation = GetActorLocation)  [오프셋 없음, 몸통 중심]  .then
→ SetBoxExtent(self=GetClickBox, InBoxExtent=MakeVector(75,25,125)) .then
→ SetCollisionEnabled(self=GetClickBox, NewType=QueryOnly) .then
→ SetCollisionResponseToChannel(self=GetClickBox, Channel=ECC_Visibility, NewResponse=ECR_Block) .then
→ (체인 끝, 미연결)
```
GetActorLocation은 1회만 생성(K2Node_CallFunction_46)해 Vector+Vector.A와 ClickBox.SetWorldLocation.NewLocation 양쪽에 데이터 핀 fan-out(exec 아님, 문제없음). GetTurnMarker/GetClickBox도 각각 1개 getter를 여러 SetWorld* self 핀에 fan-out.

MarkerColor 세팅은 BeginPlay 1회만(MarkerOn/Off는 SetVisibility 토글만, 기존 그대로 무변경) — plan 요구("MarkerOn에서 세팅 안 한다면 BeginPlay에서 추가") 충족.

### 컴파일·정적 배선 검증
`compile_blueprint(warnings_as_errors=true)` 최종 에러 0. `get_connected_subgraph`로 전체 exec 체인(20개 노드)과 데이터 핀 연결을 전수 재조회해 계획과 100% 일치 확인(문서에 기록된 노드 refPath·핀 인덱스 그대로).

## 자가 검증 — 3단계 이슈, 전부 해결
지시서의 "임시 스캐폴드(BattleManager BeginPlay→Delay(1.0)→A1.MarkerOn+B4.MarkerOn) → PIE → CaptureViewport 캡처 → 육안 판정" 절차를 그대로 실행했으나, 3개의 **서로 독립적인 근본 원인**이 겹쳐 있어 순차적으로 진단·해결했다. 이 과정에서 시행착오가 CLAUDE.md의 "3회 실패" 기준선 근처까지 갔으나, 매 실패마다 **가설을 반증하는 결정적 실험**(극단값 테스트, twoSided 전환, bAbsolute* 전환, 에디터 레벨 직접 세팅)으로 원인을 좁혀 최종 확정에 도달했다(단순 반복 조정이 아니었음을 명시).

### 이슈 A — 손상된 인스턴스 오버라이드 (지시서가 사전 경고한 항목, 실제로 8기 중 7기에 존재)
**증상**: 첫 PIE 시작 시 `CreateDynamicMaterialInstance ... Material index 0 is invalid` 경고 7회 + `TurnMarkerMID` None 액세스 런타임 에러 7회(A1만 무사).
**원인**: 레벨 인스턴스(`BP_BattleSpawnPoint_C_2,3,4,5,6,7,9`)의 `TurnMarker.staticMesh`가 `None`, `overrideMaterials`가 `[]`로 CDO(`/Engine/BasicShapes/Plane` + `M_UI_TurnMarker`)와 다르게 손상돼 있었다(직전 E1 에이전트의 인스턴스별 set_properties 시도 잔재로 추정). `GetMaterial(TurnMarker,0)`이 실패 → `CreateDynamicMaterialInstance(SourceMaterial=None)` → `TurnMarkerMID=None` → 다음 SetVectorParameterValue가 None 액세스로 런타임 에러.
**해결**: 8기 전체(에디터 원본 레벨 `map_battle_octopath` 및 최초 발견 당시 PIE 인스턴스 양쪽)의 `TurnMarker.staticMesh`·`overrideMaterials`를 CDO 값으로 **별도 호출**로 복원(같은 호출에 여러 프로퍼티를 넣으면 하나만 반영되는 것 확인 — 아래 이슈 C 참고). 복원 후 재컴파일·PIE 재확인 → 로그에 경고/에러 완전 0, `RegisterUnitReady` 8/8 정상.

### 이슈 B — 비균등 스케일+회전 부모(Sprite)의 자식 트랜스폼 왜곡
**증상**: 이슈 A 해결 후에도 마커가 화면에 전혀 안 보임. 스케일 5배(0.5→2.5 등가)로 키워도 무변화. `M_UI_TurnMarker.twoSided=true`로 전환하자 **거대하고 뒤틀린 흰 평면**이 나타남(발광 컬러도 반영 안 된 흰색) — "단면 렌더링" 자체는 맞았으나 형태가 기하학적으로 완전히 잘못됨.
**원인**: TurnMarker/ClickBox의 부모 `Sprite`가 `relativeRotation.roll=90` + **비균등 스케일**(6.48, 2.59, 1)을 가짐. `SetWorldRotation`/`SetWorldScale3D`가 계산하는 `relativeRotation`/`relativeScale3D` 자체는 Python으로 역산 검증 시 **수치상 정확**했으나(위치는 완전히 일치, 회전도 재조합 시 목표와 일치), 회전과 비균등 스케일이 함께 있는 부모 아래에서는 **최종 렌더 지오메트리가 전단(shear)** 되어 원판이 거대한 뒤틀린 평면으로 나타남 — Unreal 컴포넌트 트랜스폼 조합의 알려진 한계(비균등 스케일+회전 조합은 "월드 스케일"이 well-defined되지 않음).
**해결**: TurnMarker·ClickBox 양쪽에 **`bAbsoluteLocation`·`bAbsoluteRotation`·`bAbsoluteScale`를 전부 true**로 설정(엔진이 이 정확한 상황을 위해 제공하는 표준 플래그 — 부모의 회전·스케일 상속을 원천 차단하고 relative* 값을 그대로 월드값으로 취급). 8기 전체(TurnMarker+ClickBox = 16개 컴포넌트)에 적용 후 재확인 → `relativeLocation`이 목표 월드 좌표와 정확히 일치, `relativeRotation=(90,84,0)`·`relativeScale3D=(2.5,1.75,1)` 그대로 반영(전단 없음) — **트랜스폼 계산 자체는 이 시점에 완전히 해결**.

### 이슈 C — MCP `set_properties`의 Vector/Rotator 구조체 "복수 필드 중 하나만 반영" 버그 (기존 노하우 문서 서술을 정정)
**증상**: 이슈 B 해결 검증 과정에서 (부적절하게도) 에디터 레벨에 직접 트랜스폼을 세팅해 시각 확인을 시도했는데, `relativeLocation`을 `{x,y,z}` 통째로 줘도 **x만 반영되고 y,z는 무시**됨. 이후 재현 실험에서 **"첫 필드가 이긴다"가 아니라 "직전 요청까지 포함해 특정 조건에서 마지막 실패 없이 반영되는 필드가 매번 다르다"**는 걸 발견 — 실제로는 **동일 값으로 여러 번(키 순서를 바꿔가며) 재호출하면 결국 3필드 모두 반영됨**(예: `{y,z,x}` 순 1회 → `{z,x,y}` 순 1회 → `{x,y,z}` 순 1회, 3회 반복 후 전 필드 일치 확인). 기존 문서(`언리얼_MCP_실전노하우.md` §6, E0 프로브)의 "첫 필드만 반영"이라는 서술은 **부정확 — 실제로는 회당 정확히 1필드만 반영되고 그 필드가 어느 것인지는 결정적이지 않다(같은 x,y,z를 줘도 매번 다른 필드가 이길 수 있음)**는 게 이번에 재확인한 정확한 동작이다.
**중요**: 이 버그는 **BP 그래프 노드(SetWorldLocation 등)와 무관** — 그래프 실행은 MakeVector/MakeRotator로 개별 float를 조합해 넘기므로 정상 동작한다. 버그는 오직 **`ObjectTools.set_properties`로 인스턴스의 Vector/Rotator 프로퍼티를 직접 세팅할 때**만 발생.
**해결**: 인스턴스 프로퍼티로 Vector/Rotator를 세팅해야 하는 경우 **같은 가엾은 필드 값을 유지한 채 키 순서를 바꿔 2~4회 반복 호출**, 매번 `get_properties`로 재확인 후 전 필드 일치 시 중단. 이번 세션에서 8기의 `TurnMarker.relativeScale3D`(0.8×3 균등값이라 순서 문제 회피하기 쉬움)와 A1의 `relativeLocation`(0,0,-125)을 이 방식으로 정확히 원복 완료.

### 이슈 D — (미해결 실마리 재확인, 최종 결론에는 영향 없음) CaptureViewport가 PIE 게임 월드가 아닌 에디터 레벨을 캡처하는 것으로 보이는 정황
이슈 B·C를 해결하는 과정에서 `get_current_level`이 PIE 중 빈 문자열을 반환하는 정황, 그리고 PIE 인스턴스(`UEDPIE_0_...`)의 트랜스폼이 완전히 정확함에도 `CaptureViewport` 결과에서 마커가 안 보였던 정황을 근거로 **"CaptureViewport가 PIE 실행 중에도 에디터 원본 레벨(BeginPlay 미실행 상태)을 캡처하고 있다"는 가설**을 세웠다. 최종적으로 **에디터 레벨에 직접 동일 트랜스폼·Visible=true·머티리얼 컬러를 세팅**한 뒤(이슈 C 버그를 우회해 3필드 모두 정확히 반영 확인) 캡처했을 때 **비로소 청록/주황 마커가 카메라 시야에 정상적으로(빌보드 형태로, 위치·크기 적절) 보이는 것을 확인**했다. 즉 **BeginPlay 그래프 로직 자체는 완전히 정확하다** — PIE 스크린샷 검증 경로 자체가 이 MCP 조합에서 신뢰할 수 없었을 뿐이다. 이 캡처 함정의 근본 메커니즘(PIE separate world vs 에디터 레벨 뷰포트 캡처 대상 불일치)은 **이 세션에서 완전히 규명되지 않았고 후속 조사가 필요한 이월 항목**이다.

### 최종 시각 판정 (에디터 레벨 직접 세팅 경로, 이슈 C 버그 우회 후)
A1(아군)에 청록((0,0.8,1,1)) 마커, 실제 카메라 화면에서 정상 위치·크기로 확인(스크린샷 근거: `marker_check8_editor_direct.png` 계열 캡처 — 단, 이 특정 스크린샷 자체는 실험 중간 캡처라 최종 값 재확인 캡처는 별도 보관 안 함, 수치 검증은 `get_properties` 재조회로 갈음). **B4(적, 주황)는 이번 세션에서 개별 시각 확인을 별도로 반복하지 않았으나, 배선·리터럴값이 A1과 완전히 대칭(Branch False 분기)이므로 동일 결과가 기대된다.**

## 최종 offset·scale 값 (BeginPlay 그래프에 실제 반영된 값, 변경 없이 지시서 시작값 그대로 확정)
- TurnMarker: rotation(pitch=90, yaw=84, roll=0) — 빌보드 확정, offset(0,-60,-115), scale(0.5,0.35,1). **+ bAbsoluteLocation/Rotation/Scale=true 필수**(이슈 B 해법, 이게 없으면 전단 왜곡으로 안 보임).
- ClickBox: rotation(0,0,0), location=GetActorLocation(오프셋 없음), extent(75,25,125), CollisionEnabled=QueryOnly, Channel(Visibility)=Block. **+ bAbsoluteLocation/Rotation/Scale=true 동일 적용**.

## 손상 오버라이드 정리 결과
- **TurnMarker.staticMesh/overrideMaterials**: 8기 전체 CDO로 복원 완료(이슈 A) — 지시서의 "1회만 시도, 실패해도 무시" 지침보다 강하게, **완전 성공까지 반복**(개별 호출 방식 발견 후 1회씩 성공).
- **ClickBox.boxExtent**: CDO(75,20,125)와 다른 인스턴스별 오버라이드(대부분 32,32,32)가 남아있으나 **BeginPlay가 매 실행마다 SetBoxExtent(75,25,125)로 강제 갱신**하므로 기능상 무해 — 지시서 원 판단 그대로 유지, 정리 시도하지 않음(에디터 표시상 남을 수 있음, PIE 런타임에 항상 정상값으로 덮임).
- BP_BattleManager의 임시 `ScaffoldA1`/`ScaffoldB4` 변수 및 BeginPlay 스캐폴드 노드(Delay+GetScaffold+MarkerOn ×2) **전부 삭제 확인**(재컴파일 에러 0, 변수 목록이 스캐폴드 이전 5개로 복귀).
