# E0 파일럿 프로브 결과 (gameplay-engineer)

## 프로브 ① — Function Graph + add_object_function_param + compile (GE-C2 우회안)

**목적**: 커스텀 이벤트에 오브젝트 파라미터를 붙일 수 없다는 GE-C2 제약의 우회안(Function Graph로 대체)이 실제로 동작하는지 실증.

**절차·결과**: **성공**
1. `BlueprintTools.create(folder_path="/Game/Blueprints", asset_name="BP_BattleManager", asset_type={refPath: "/Script/Engine.Actor"})` → `/Game/Blueprints/BP_BattleManager.BP_BattleManager` 생성됨.
2. `BlueprintTools.add_function_graph(blueprint=BP_BattleManager, graph_name="NotifyUnitClicked")` → 그래프 `/Game/Blueprints/BP_BattleManager.BP_BattleManager:NotifyUnitClicked` 생성됨.
3. `BlueprintTools.add_object_function_param(graph=NotifyUnitClicked, param_name="ClickedUnit", object_class={refPath: "/Game/Blueprints/BP_BattleSpawnPoint.BP_BattleSpawnPoint_C"}, input_param=true)` → 성공. 반환값 `{"direction":"EGPD_Output","index_id":1,"node":{"refPath":".../NotifyUnitClicked.K2Node_FunctionEntry_0"}}` (direction이 EGPD_Output인 것은 FunctionEntry 노드 입장에서 "이 파라미터 값을 내보내는" output pin이라 정상 — 함수 호출자 입장에서는 input).
4. `BlueprintTools.compile_blueprint(blueprint=BP_BattleManager, warnings_as_errors=true)` → `returnValue: null`(에러 없음, 정상 완료).
5. 재조회 `get_node_infos([NotifyUnitClicked.K2Node_FunctionEntry_0])` → output_pins에 `{"name":"ClickedUnit","type_id":"BP Battle Spawn Point 오브젝트 레퍼런스"}` 확인됨.

**결론**: GE-C2 우회안(NotifyUnitClicked를 Function Graph로 구현하고 add_object_function_param으로 오브젝트 파라미터 추가)이 **완전히 실증됨**. 폴백(이벤트 디스패처 등) 검토 불필요.

## 프로브 ② — byte 변수 + 관련 변수 4종

**목적**: enum 생성 불가(GE-C1) 대안인 byte 타입 변수가 `add_variable`로 정상 생성되는지 확인.

**절차·결과**: **성공**
- `add_variable(BP_BattleManager, name="BattleState", type_name="byte")` → 성공
- `add_variable(BP_BattleManager, name="bInputLocked", type_name="bool")` → 성공
- `add_variable(BP_BattleManager, name="CurrentIndex", type_name="int")` → 성공
- `add_variable(BP_BattleManager, name="RegisteredCount", type_name="int")` → 성공

**결론**: byte 변수 포함 4종 전부 정상 생성·컴파일 확인.

## 프로브 ③ — 오브젝트 배열 변수(TurnQueue) 원소 세팅

**목적**: `TurnQueue`(BP_BattleSpawnPoint ref ARRAY, instance-editable)를 레벨 배치 인스턴스에서 인덱스별로 채울 수 있는지 확인.

**절차**:
1. `add_object_variable(BP_BattleManager, name="TurnQueue", object_class={refPath:"/Game/Blueprints/BP_BattleSpawnPoint.BP_BattleSpawnPoint_C"}, container_type="ARRAY")` → 성공.
2. `set_variable_instance_editable(BP_BattleManager, variable_name="TurnQueue", instance_editable=true)` → 성공.
3. `compile_blueprint(warnings_as_errors=true)` → 에러 0.
4. 레벨 `map_battle_octopath`(이미 로드됨)의 `BattleStage/SpawnPoints` 폴더에서 8기 refPath 확보(`get_actors_in_folder`) 후 `get_label`로 라벨 매핑 재확인 — [[../../옥토패스대치/raw/배치_1|배치_1]] 문서 표와 100% 일치:
   - C_0=SpawnPoint_Party_A1, C_9=SpawnPoint_Party_A2, C_2=SpawnPoint_Party_A3, C_3=SpawnPoint_Party_A4
   - C_4=SpawnPoint_Enemy_B1, C_5=SpawnPoint_Enemy_B2, C_6=SpawnPoint_Enemy_B3, C_7=SpawnPoint_Enemy_B4
5. `SceneTools.add_to_scene_from_asset(asset_path="/Game/Blueprints/BP_BattleManager", name="BattleManager", xform={location:{x:0,y:-7100,z:630}, rotation:{pitch:0,yaw:0,roll:0}, scale:{x:1,y:1,z:1}})` → 액터 `BP_BattleManager_C_0` 생성, 폴더 `BattleStage/Logic`으로 이동, 라벨 "BattleManager" 확인(`get_label` 재조회).
6. **원소 세팅 시도 — GE-H3 원가정과 실측이 다름을 발견**:
   - 시도 A: `set_properties(values='{"TurnQueue[0]": {refPath: "...C_0"}}')` → **실패**: `"the following properties could not be set: TurnQueue[0]"`
   - `list_properties(BattleManager)`로 실제 프로퍼티명이 카멜케이스 `turnQueue`(대문자 `TurnQueue`가 아님)임을 확인.
   - 시도 B: `set_properties(values='{"turnQueue[1]": {...}}')` → **여전히 실패**: `"could not be set: turnQueue[1]"`
   - 시도 C: `set_properties(values='{"turnQueue.1": {...}}')` → **실패**: `"could not be set: turnQueue.1"`
   - 시도 D(성공): `set_properties(values='{"turnQueue": [{refPath:"...C_0"}]}')` — **배열 전체를 리스트값으로 재할당** → 성공(`returnValue: true`). 재조회 `get_properties(["turnQueue"])` → `{"turnQueue":[{"refPath":".../BP_BattleSpawnPoint_C_0"}]}` 확인.
   - 이 방식으로 2원소(`[A1,B1]`) 먼저 검증 후, 8원소 전체를 한 번의 `set_properties` 호출로 순서대로(`[A1,B1,A2,B2,A3,B3,A4,B4]` = `[C_0,C_4,C_9,C_5,C_2,C_6,C_3,C_7]`) 세팅 완료.
   - 최종 재조회(ProgrammaticToolset으로 각 refPath→라벨 변환): `{"count": 8, "labels_in_order": ["SpawnPoint_Party_A1","SpawnPoint_Enemy_B1","SpawnPoint_Party_A2","SpawnPoint_Enemy_B2","SpawnPoint_Party_A3","SpawnPoint_Enemy_B3","SpawnPoint_Party_A4","SpawnPoint_Enemy_B4"]}` → **plan이 요구한 순서와 정확히 일치**.
7. `compile_blueprint(BP_BattleManager, warnings_as_errors=true)` → 에러 0(최종 확인).

**결론 — plan과의 차이 명시**:
- **목표(오브젝트 배열에 8개 SpawnPoint 참조를 정확한 순서로 세팅)는 100% 달성**. plan의 폴백 조건("원소 세팅이 안 되면 그 사실을 보고, 폴백은 실행하지 말고 보고만")에 해당하는 **완전한 실패는 아니었음** — 개별 변수(Unit0..7) 폴백으로 갈 필요 없다고 판단해 계속 진행함.
- 다만 GE-H3가 가정한 정확한 문법("인덱스별 개별 set_properties" = `TurnQueue[i]` 같은 경로 접근으로 원소 하나씩 세팅)은 **3가지 표기(`TurnQueue[0]`, `turnQueue[1]`, `turnQueue.1`) 전부 거부됨**. 실제로 동작하는 유일한 방법은 **매 호출마다 배열 전체를 리스트로 통째로 재할당**하는 것. 이는 "인덱스별 개별 접근"이 아니라 "배열 전체 재할당을 여러 번 누적 호출"이라는 점에서 원 설계 문서(GE-H3)의 절차 서술과 다르다.
- **Director/qa-critic 확인 요망**: 이 차이가 이후 단계(E1~E3에서 유닛 register 로직이 런타임에 배열에 원소를 추가/제거해야 하는 경우, 예: null 슬롯 compact-제외 로직)에 영향을 주는지 검토 필요. 런타임 BP 그래프 내부(Add to Array 노드 등)에서의 배열 조작은 이 MCP `set_properties` 제약과 무관 — 이건 **에디터 MCP를 통한 인스턴스 프로퍼티 세팅**에만 해당하는 제약이다. 런타임 큐 빌드(Init 상태, null 슬롯 compact)는 Blueprint 그래프 노드(Add/Remove from Array)로 구현되므로 이 제약의 영향을 받지 않을 것으로 판단되나, E2 구현 시 재확인 필요.
