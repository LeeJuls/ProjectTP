# E3 폴리시·엣지+풀 회귀 — 게이트 실행 기록 (gameplay-engineer)

> 상위: [[../plan|턴제전투MVP plan]] · [[../청사진|청사진]] · 선행: [[E2_상태머신]]
> **이 실행은 직전 E3 에이전트가 240초 단일 대기로 워치독 강제 종료된 직후의 재실행**이다. §0에서 잔여 상태를 전수 복구한 뒤 본 테스트를 진행했다.

## 0. 잔여 상태 복구 (재실행 최우선 작업)

직전 에이전트가 BP_BattleManager BeginPlay에 심어놓은 스캐폴드가 그대로 남아있었다.

1. **IsPIERunning=true 확인 → StopPIE**.
2. **BeginPlay 오염 발견**: `ReceiveBeginPlay → Delay(1.0) → ScaffoldTurnStep(self)` 체인이 BeginPlay에 직결되어 있었다(원래는 exec 미연결이 정상 — E2 문서 기록과 대조 확인). `ScaffoldTurnStep`이라는 신규 Custom Event(46개 노드 규모, 재귀 8턴 루프 + `ScaffoldTurnCounter` 변수)가 EventGraph에 잔존.
3. **전수 삭제**: `find_nodes`(entry_points_only)로 진입점 4개 확인(BeginPlay/EnterExecuting/EnterTurnEnd/ScaffoldTurnStep) → `get_connected_subgraph`로 ScaffoldTurnStep 연결 46개 노드 + BeginPlay 체인의 Delay/호출 노드 2개 = 총 48개 노드를 `ProgrammaticToolset`으로 일괄 `delete_node`(48/48 성공) → `remove_variable("ScaffoldTurnCounter")`.
4. 재확인: 진입점 3개(BeginPlay/EnterExecuting/EnterTurnEnd)로 복귀, 변수 8개(BattleState·bInputLocked·CurrentIndex·RegisteredCount·TurnQueue·ActiveUnit·SelectedTarget·ButtonRef, plan.md 명세와 완전 일치)로 복귀. BeginPlay `then` 핀 미연결(원래 상태) 확인. `compile_blueprint(warnings_as_errors=true)` 에러 0.
5. **TurnQueue 8원소 재조회**: `[A1,B1,A2,B2,A3,B3,A4,B4]`(C_0,C_4,C_9,C_5,C_2,C_6,C_3,C_7) — 이미 정상 상태, 재할당 불필요했음.
6. **`bThrottleCPUWhenNotForeground` 확인**: 조회 결과 `false`(직전 에이전트가 진단용으로 끈 뒤 워치독으로 강제 종료되어 원복 못 한 상태) — 테스트 진행 동안 그대로 유지, §6에서 최종 `true` 복원.
7. BP_AttackButton/BP_BattleSpawnPoint도 함께 점검: 양쪽 다 진입점 개수·exec 연결 정상(BP_AttackButton에 `ReceiveTick`/`ActorBeginOverlap이벤트` dangling stub 2개가 exec 미연결 상태로 존재하나, 이는 **E2 세션 이전부터 있던 것으로 추정되는 무해 스텁**이며 지시서 범위(BP_BattleManager BeginPlay)를 벗어나 손대지 않음 — Director 보고용 이월 항목).

## 1. 스캐폴드 테스트 슈트 (BeginPlay 임시 체인, 유닛 참조=TurnQueue Get만)

### 설계
`ScaffoldE3Run` Custom Event 1개(선형 flat 시퀀스, 재귀 없음) — BeginPlay→Delay(2.0)→ScaffoldE3Run 순차 호출. 각 블록은 `create_node`+`connect_pins`+`set_pin_value`로 배선(직전 에이전트의 재귀 구조 대신 **선형 체인**으로 재설계 — 재귀 구조가 240초 대기의 근본 원인이었을 가능성을 배제하기 위한 설계 변경).

### 핵심 이슈 발견 및 수정 — 스캐폴드 자체의 타이밍 설계 결함 (게임 로직 버그 아님)
1차 실행에서 8턴 중 3번째 턴부터 `NotifyUnitClicked: ignored (same team or self)` / `ignored (not in AwaitTarget state)`가 반복되며 정상 순환이 깨졌다. **근본 원인 진단**: 실측 1턴 사이클이 약 2.0초(AwaitTarget 재진입 간격)인데, 스캐폴드의 클릭 간격을 0.1+1.6=1.7초로 짧게 잡아 클릭이 이전 턴의 Executing/TurnEnd 잔여 상태에 도달 → 3번째 버튼 클릭이 `bInputLocked=true`로 씹힘 → 이후 클릭들이 게임 상태와 계속 어긋남. **정적 배선 검증**(`NotifyUnitClicked`/`EnterTurnEnd`/`InitBattle` 전수 재조회)으로 게임 로직 자체엔 결함이 없음을 먼저 확정한 뒤, 스캐폴드의 턴 블록 후행 Delay를 1.6→**2.0초**로 조정해 재실행 → 8턴 완전 정상 순환 확인. **이 삽질은 게임 로직 결함이 아니라 스캐폴드 타이밍 설계의 실수였음을 명시한다.**

같은 원인 계열로 TC-23 스캐폴드의 target 인덱스도 1차 실행에서 오설계(TC-22가 4연타 왕복만 하고 턴을 소모하지 않아 ActiveUnit이 여전히 B1인데 target도 B1로 지정 → "same team" 오탐)를 발견·수정(target index 1→0)해 재실행, 정상 확인.

## 2. null 슬롯 [TC-17/18]

에디터 원본 레벨의 `TurnQueue[2]`(A2)를 `set_properties`로 `null`로 강제 설정 → 짧은 2턴 스캐폴드(BeginPlay→Delay(2.0)→ScaffoldNullTest: Attack→Delay(0.1)→Unit(B1)→Delay(2.0)→Attack→Delay(0.1)→Unit(A1)→Delay(2.0)→Print(DONE), 총 소요 약 6.4초) PIE 실행.

### 핵심 발견 — InitBattle의 compact 로직 실증 확인
PIE 인스턴스(`UEDPIE_0_...`)의 `TurnQueue`를 직접 `get_properties`로 재조회한 결과 **7원소**(`[A1,B1,B2,A3,B3,A4,B4]`, null이던 A2 슬롯이 완전히 사라짐)로 확인됨. `InitBattle` 함수 본문을 `get_connected_subgraph`로 전수 재조회해 원인 확정: `ReverseforEachLoop(TurnQueue)` → 각 원소 `IsValid` 체크 → **Is Not Valid → `Array|RemoveIndex`**로 역순회 방식 정확히 compact(인덱스 밀림 없음) → `TurnQueue.Length()==0` 가드(에러 로그+정지) → 정상 진행. 이는 plan.md 명세("Init: 큐 빌드(null 슬롯 compact 제외)", "큐 길이 0이면 에러 로그·정지")를 정확히 구현한 것으로, **버그가 아니라 설계대로 동작하는 정상 로직**임을 확정.

### TC 판정
| TC | 판정 | 근거 |
|---|---|---|
| TC-17(modulo=compact 후 실길이, null 강제 시 정상 순환) | **PASS** | PIE 인스턴스 TurnQueue가 7원소로 정확히 compact됨(RemoveIndex 확인), `CurrentIndex=2`가 compact된 배열의 유효 원소(B2)를 정확히 가리킴(get_properties 직접 재조회) |
| TC-18(null 슬롯 skip, 무한루프/크래시 없음) | **PASS** | compact 자체가 null을 원천 제거하므로 런타임 skip 자체가 불필요해짐(더 견고한 구현) — 2턴 스캐폴드 정상 완주, 크래시 없음. `EnterTurnStart`의 `IsValid(ActiveUnit)→Is Not Valid→EnterTurnEnd()`(즉시 skip) 경로도 정적 확인(compact 실패 잔여 대비 이중 방어) |

에디터 원본 TurnQueue는 즉시 8원소(null 없이)로 원상 복구(`set_properties`, 재조회로 확인). 스캐폴드(ScaffoldNullTest 이벤트 + BeginPlay 연결, 17개 노드)는 `ProgrammaticToolset` 일괄 삭제(17/17 성공), 진입점 3개·BeginPlay 미연결 상태로 원복, 컴파일 에러 0.

## 신규 노하우 (실전노하우 추가 후보 — Director 승인 후 §10으로 등재 권장)

### 스캐폴드 배선 실패 시 connect_pins 실패가 같은 트랜잭션 내 앞선 create_node 결과를 롤백시킬 수 있음
`ProgrammaticToolset.execute_tool_script` 안에서 `connect(beginplay, ..., n_delay, ...)` 성공 → `connect(n_delay, ..., call_scaffold, ...)`가 실패(대상 노드가 CustomEvent라 exec input이 없어서 실패)했을 때, **스크립트 실행 전체가 실패로 처리되며 그 안에서 앞서 성공했던 `create_node`(n_delay)의 결과까지 그래프에서 사라진 것으로 관찰됨**(노드 번호가 118을 건너뛰어 119로 이어짐 — 118이 생성됐다가 롤백된 것으로 추정). **대응**: 배선이 실패할 위험이 있는 복잡한 스크립트는 노드 생성과 연결을 분리된 여러 개의 작은 스크립트/개별 tool 호출로 나눠 실행하고, 각 단계마다 `get_node_infos`로 실제 반영 여부를 재확인할 것. 특히 Custom Event 노드로 직접 exec 연결을 시도하면 반드시 실패한다 — Custom Event는 input_pins가 없으므로, 호출하려면 반드시 별도로 `함수호출|<이벤트명>` type_id의 CallFunction 노드를 만들어 그 노드로 연결해야 한다.

### 스캐폴드로 다턴 시나리오를 검증할 때 클릭 간격은 반드시 실측 1턴 사이클보다 여유있게 잡을 것
고정 Delay 값(예: 0.1+1.6=1.7초)으로 여러 턴을 연속 클릭하는 스캐폴드를 짤 때, 설계상 예상 사이클(1.10초, plan.md 명세치)이 아니라 **직전 게이트에서 실측된 값(예: E2가 보고한 "+0.25~0.32s 편차" 반영 시 약 2.0초)** 을 기준으로 여유를 둬야 한다. 여유가 부족하면 클릭이 이전 턴의 Executing/TurnEnd 잔여 상태에 도달해 `bInputLocked` 가드에 씹히고, 이후 모든 클릭이 게임 상태와 어긋나 "same team"/"not in AwaitTarget" 같은 오탐 로그가 연쇄적으로 발생한다 — 이것은 게임 로직의 결함이 아니라 스캐폴드 자체의 타이밍 설계 실수이므로, 오탐 발견 시 먼저 정적 배선을 재확인해 로직 결백을 확정한 뒤 스캐폴드 타이밍부터 의심할 것.
