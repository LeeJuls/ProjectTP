# E2 Manager 상태머신 — BattleManager 구현·자가검증 (gameplay-engineer)

> 상위: [[../plan|턴제전투MVP plan]] · [[../청사진|청사진]] · 선행: [[E1_유닛확장]]
#projectTP/턴제전투MVP

## 구현 요약

BP_BattleManager에 상태 진입 함수 5개(EnterTurnStart/EnterAwaitCommand/EnterAwaitTarget = Function Graph, EnterExecuting/EnterTurnEnd = Custom Event) + InitBattle + NotifyUnitClicked(본문 교체) + NotifyAttackButtonClicked(신규) 배선. BP_AttackButton에 ShowAttack/ShowCancel/HideAll 3종 + 유령공격 배선 제거 + Manager 호출로 교체. ST_UI에 `Battle.Cancel` 추가. 레벨에서 Manager↔AttackButton 상호 참조 세팅.

## 설계와 다르게 구현한 부분 (중요 — qa-critic 대조 필수)

### ① EnterExecuting·EnterTurnEnd는 Function Graph가 아니라 Custom Event(EventGraph)로 구현
**사유**: UE 표준 제약 — **Delay/RetriggerableDelay 같은 latent 노드는 Function Graph에서 사용 불가**(함수는 동기 실행이어야 함). `find_node_types`로 함수 그래프 컨텍스트에서 `유틸리티|플로컨트롤|Delay`를 검색하면 결과가 비어 있고, EventGraph 컨텍스트에서만 나옴을 실측 확인. Executing(Delay 0.25/0.75 필요)과 TurnEnd(Delay 0.35 필요)는 어쩔 수 없이 Custom Event로 전환. 다른 곳에서의 "호출"은 커스텀 이벤트도 `함수호출|EnterExecuting` 형태로 CallFunction 노드처럼 호출 가능해 명세가 요구한 "이름 붙은 상태 진입 지점"이라는 정신은 그대로 유지됨. 나머지 3개(TurnStart/AwaitCommand/AwaitTarget)는 Delay가 필요 없어 Function Graph 그대로 유지.

### ② NotifyAttackButtonClicked의 "Switch(BattleState)"를 Branch 2개(Equal 비교)로 구현
**사유**: BattleState가 byte라 `SwitchOnByte`류 전용 노드가 `find_node_types`에서 발견되지 않았고(byte enum이 아니라 순수 byte라 Switch 노드 자체가 없을 가능성), 분기 대상이 AwaitCommand(2)/AwaitTarget(3) 단 2개뿐이라 `Branch(BattleState==2)` 단일 분기로 충분히 동일 로직 구현. 기능적으로 명세의 분기 결과와 100% 동일.

## MCP 실전 함정 (신규 발견 — §7 추가 예정)

### `add_event`를 병렬(같은 메시지 내 여러 tool call)로 호출하면 `event_name`이 무시되고 기본 이름("커스텀이벤트")으로 생성될 수 있음
EnterExecuting/EnterTurnEnd 두 Custom Event를 동시에 `add_event` 호출했을 때, 첫 번째(EnterExecuting)가 실제로는 이름 미반영(`AddEvent|Custom|커스텀이벤트`)으로 생성됨 — 로그에 `LogBlueprint: Warning: User provided name was invalid 이미 사용 중인 이름입니다. - node named 커스텀이벤트` 확인. **해법**: 같은 이벤트명을 다시 단독으로 `add_event` 재호출하면 신규 노드가 정확한 이름으로 생성됨(리네임이 아니라 별도 신규 노드) → 기존 배선을 새 노드로 옮기고 잘못된 이름의 노드를 `delete_node`. **결론**: 여러 Custom Event를 만들 땐 순차 호출(1개씩) 또는 사후 `list_events`로 이름 검증 필수.

### `set_pin_value`로 오브젝트 레퍼런스 리터럴을 직접 지정 가능(신규 확인 노하우)
함수 호출 노드의 오브젝트 파라미터 핀에 `set_pin_value`로 레벨 인스턴스 경로 문자열(예: `/Game/Stages/map_battle_octopath.map_battle_octopath:PersistentLevel.BP_BattleSpawnPoint_C_4`)을 직접 주면 정확히 반영됨(재조회로 확인). 스캐폴드에서 `NotifyUnitClicked(ClickedUnit=하드참조)` 구성에 활용 — 별도 "리터럴 오브젝트 레퍼런스" 노드 불필요.

### `find_node_types`의 카테고리가 노드마다 영문/한글 혼용이라 검색어를 여러 형태로 시도해야 함
`Utilities|FlowControl|Branch`(영문)와 `유틸리티|플로컨트롤|Delay`(한글)가 같은 논리적 카테고리(FlowControl) 아래 혼재. `Utilities|FlowControl|시퀀스`(Sequence는 카테고리 영문+노드명 한글)처럼 부분 혼용도 있음. 실패 시 영/한 양쪽으로 재시도 필요.

### TakeHit 재측정 결론 — 새로운 이월 이슈 발견
TakeHit 이벤트 진입 로그("TakeHit:{DisplayName}")가 이번 세션에서 **단 한 번도 발생하지 않음**. 진단 프로브(IsValid(SpriteMID).Is Not Valid 분기에 임시 PrintString)를 추가해 확인한 결과, **`[BP_BattleSpawnPoint_C_4] TakeHit: SpriteMID IS NOT VALID`**, **`[BP_BattleSpawnPoint_C_0] TakeHit: SpriteMID IS NOT VALID`**가 확인됨 — **양쪽 유닛 모두 SpriteMID가 None**이라 TakeHit 로직 전체(SetScalar/RetriggerableDelay/revert)가 조용히 스킵되고 있음.

**중요 소견**: PlayAttack 이벤트도 정확히 동일한 `IsValid(SpriteMID)` 게이트를 사용하나 로그 스텁이 없어 확인 불가 — 같은 원인으로 **PlayAttack 애니메이션도 재생되지 않고 있을 가능성이 매우 높음**(정적 배선 확인, PlayAttack의 Is Valid 분기 첫 노드가 SetScalarParameterValue라 로그가 원천적으로 없음).

**진단 경과**: 정적 배선 전수 재검증(BeginPlay→CreateDynamicMaterialInstance→SetSpriteMID→...) 완료, 100% 계획대로 정확. Sprite 컴포넌트의 staticMesh/overrideMaterials도 정상(MI_Party_A1 등 유효). CreateDynamicMaterialInstance 실패 관련 엔진 경고도 로그에 없음. **가설 3종(머티리얼 손상/add_event 레이스 잔재/GetMaterial 배선 오류) 모두 기각** — 근본 원인 미규명 상태로 CLAUDE.md "3회 실패 시 중단" 원칙에 따라 이 이상 조사하지 않고 이월.

**스코프 판단**: 이 문제는 BP_BattleManager(E2 담당 범위)의 TakeHit **호출 자체는 정상**(로그로 TakeHit 이벤트가 실제 호출됨을 확인)이며, 실패 지점은 순수하게 BP_BattleSpawnPoint(E1 담당 범위) 내부의 SpriteMID 런타임 무효화에 있음. E1 문서의 "TakeHit 실측 0.33 vs 설계 0.45 편차" 미해결 이슈, "CaptureViewport 대상 불일치" 미해결 이슈와 함께 **BP_BattleSpawnPoint의 MID 초기화/유지 관련 세 번째 미해결 이슈**로 등록.

## E2-후속: SpriteMID 무효화 재수사 — 원인은 BeginPlay가 아니라 자가검증 스캐폴드였다 (gameplay-engineer, 2026-07-07)

> 지시서: "PIE에서 유닛 SpriteMID가 무효(IsValid=false)라 TakeHit/PlayAttack 가드에 막혀 애니메이션이 재생되지 않는 회귀" 수사. **결론: 회귀는 존재하지 않았다.** BP_BattleSpawnPoint/BP_BattleManager의 실제 게임 로직은 처음부터 끝까지 정상이었고, E2가 관찰한 "SpriteMID IS NOT VALID"는 **E2 자신의 자가검증 스캐폴드가 안고 있던 MCP 함정**의 산물이었다.

### 수사 방법 — "진단 프린트 사다리"로 단계별 확정
1. **정적 배선 전수 재확인**(`get_connected_subgraph`): BeginPlay 체인 38개 노드 전부 재조회, E1 문서의 기록과 100% 일치. `CreateDynamicMaterialInstance(Sprite)`→`SetSpriteMID`의 exec/데이터 연결에 "고아 Set"·혼동 없음. TakeHit의 `IsValid(SpriteMID)` 게이트 배선도 정확.
2. **진단 PrintString 3종 삽입**(BeginPlay의 (a)CreateDMI 직후 (b)SetSpriteMID 직후 재Get (c)BeginPlay 최말단) → PIE → **8기 전원 3지점 전부 VALID**. BeginPlay 로직 자체는 완전 결백 확정.
3. **TakeHit을 실제로 발화시켜야 함** — 그러나 MCP엔 PIE 콘솔 명령 주입 수단이 없어 임시 스캐폴드(BP_BattleManager.BeginPlay→Delay(4.0)→TakeHit(self=하드참조) 직접 호출) 필요.
4. 스캐폴드의 `self` 파라미터를 `set_pin_value`로 **리터럴 오브젝트 레퍼런스**(`/Game/Stages/map_battle_octopath.map_battle_octopath:PersistentLevel.BP_BattleSpawnPoint_C_4`, PIE 접두사 없음) 지정 — 이 패턴은 E2 자신이 §8 노하우에 "성공 실증"으로 기록한 것과 동일.
5. PIE 실행 → TakeHit은 호출됐으나(`DIAG_D: TakeHit ENTERED` 확인) **SpriteMID가 여전히 Not Valid**(`GetDisplayName(SpriteMID)`가 빈 문자열 반환 — None 확정). 그런데 같은 시각 `ObjectTools.get_properties`로 PIE 인스턴스(`UEDPIE_0_map_battle_octopath...BP_BattleSpawnPoint_C_4`)의 SpriteMID를 조회하면 **항상 유효한 MID**(`MID_MI_Enemy_B1_flip_0`)였다 — 모순.
6. **모순의 해소**: 에디터 원본(비-PIE) 액터(`/Game/Stages/map_battle_octopath.map_battle_octopath:...BP_BattleSpawnPoint_C_4`, PIE 접두사 없음)의 SpriteMID를 조회하니 **`None`**(BeginPlay 미실행이므로 당연). 즉 **스캐폴드가 세팅한 리터럴 레퍼런스는 PIE 시작 후에도 PIE 카피로 리매핑되지 않고 에디터 원본("그림자" 액터, BeginPlay 미실행·SpriteMID=None)을 계속 가리켰다.** 로그 prefix(`[BP_BattleSpawnPoint_C_4]`)는 `GetName()` 결과만 표시해 PIE/에디터 구분(`UEDPIE_0_` 접두사)이 드러나지 않아 착시를 유발했다.
7. **결정적 반증 실험**: 스캐폴드의 `self`를 리터럴 대신 **런타임 배열 인덱스 접근**(`GetTurnQueue()` → `Array|Get(사본)` Index=1, 즉 실제 게임 로직이 8/8 등록 시 채운 PIE 인스턴스 배열)으로 교체 → 재실행 → **`TakeHit:SpawnPoint_Enemy_B1` / `TakeHitRevert:SpawnPoint_Enemy_B1` 정상 발화, SpriteMID VALID 확정**. `NotifyUnitClicked`(실제 게임 흐름)도 `SelectedTarget`을 파라미터로 받아 멤버변수에 Set할 뿐 리터럴을 쓰지 않으므로 **이 함정과 무관 — 실제 오너 플레이(마우스 클릭 경로)에서는 처음부터 문제가 없었다.**

### 근본 원인 확정
**BP_BattleSpawnPoint/BP_BattleManager의 게임 로직에는 결함이 없다.** E2가 관찰한 회귀는 **E2 자신의 자가검증 스캐폴드**가 `set_pin_value`로 세팅한 **PIE 접두사 없는 에디터 원본 레벨 액터 레퍼런스**를 그대로 self로 사용했기 때문에 발생한 **가짜 신호**였다. 이 패턴(§8 "set_pin_value로 함수 호출 노드의 오브젝트 파라미터에 리터럴 레퍼런스를 직접 지정 가능")은 `NotifyUnitClicked` 테스트처럼 **함수 호출 직후 즉시(같은 PIE 틱 내) 결과만 관찰하는 경우엔 문제가 드러나지 않았을 수 있으나**, 이번처럼 **BeginPlay가 채운 런타임 상태(SpriteMID)에 의존하는 하위 로직을 검증**할 때는 에디터 원본 "그림자" 액터를 잘못 가리켜 완전히 그릇된 결론(가짜 회귀)을 낳는다. E1 이슈 D(CaptureViewport가 PIE 게임 월드가 아닌 에디터 레벨을 캡처하는 정황)와 **동일 계열의 "MCP PIE/에디터 레벨 혼동" 함정**이다.

### 스로틀링 가설 판정 — 부분 확인 (일부 기여, 완전한 설명 아님)
- `EditorPerformanceSettings.bThrottleCPUWhenNotForeground`(현재 프로젝트 기본값 `true`) 상태에서는 스캐폴드의 `Delay(4.0)`가 **30초 이상 대기해도 전혀 발화하지 않음**(게임 틱이 사실상 정지) — 강한 정황 증거.
- `false`로 끈 뒤에는 정상 발화했으나, 두 차례 재현 모두 **실측 약 3.37초**(설정 4.0초 대비 -16%, 설계보다 오히려 **빠르게** 발화) — E1/E2가 원래 보고한 "+0.25~0.32s 느려짐" 편차와는 **부호·크기가 다름**.
- **판정**: 스로틀링 켜짐 상태는 게임 틱을 극단적으로(사실상 무한대로) 지연시킬 수 있어 **PIE 자가검증 세션에선 반드시 꺼야 하는 전제조건**임을 확인했으나, E1/E2가 보고한 수백ms 단위의 편차 자체를 스로틀링만으로 완전히 설명할 수는 없다 — **별도의 PIE 게임시간 계산 오차 요인이 남아있을 가능성, 후속 조사 필요**(이번 세션 범위 밖).
- 세팅은 진단 종료 후 **원래 값(`true`)으로 복원**했다(에디터 로컬 설정이라 변경 자체는 허용됐으나, 프로젝트 기본값을 영구 변경할 의도가 아니었으므로).
