## 6. ⚠ 스크린샷 실패 + 스케일 초과 경고 (Director/hd2d-art-director 보고 필요)

### (a) CaptureViewport 실패 (기술적 한계, P1에서 이미 경고됨)
- `EditorAppToolset.CaptureViewport`(3파라미터: captureTransform/annotations/bShowUI 모두 명시) 호출 시 결과가 **약 480만 문자(base64 PNG, 원본 약 3.6MB)** 로 MCP 파이프라인의 토큰 한도를 초과, 이미지 열람 불가.
- `Editor.General.EditorPerformanceSettings`의 `bDisplayHighDPIViewports=false` + `maxViewportRenderingResolution=1280` 오버라이드 시도했으나 결과 크기 변화 없음(원인: 뷰포트 캡처가 별도 렌더패스를 사용하거나 데스크톱 실제 해상도에 연동되는 것으로 추정) → 설정 원복 완료, 3회 미만 시도 후 중단(전역 규칙 준수).
- **결론: 이 환경에서 CaptureViewport 스크린샷 자체가 현재 불가능**. 대신 `WorldPosToScreenCoords`로 8개 지점(발밑/머리 좌표) 정밀 실측 완료, 아래 표 근거.

## 7. S3 수정 — 세로 왜곡 버그 + A2 지형 튐 해결 (2026-07-05)

### (a) 회전축 실측 — 근본 원인 재진단 (기존 진단 정정)

기존 로그(§6)는 "화면 세로=로컬Y, 화면 가로=로컬X"를 전제로 스케일만 문제 삼았으나, **실측 결과 전제 자체가 틀렸음**을 확인했다.

- `SM_SpriteQuad` 로컬 바운드(`StaticMeshTools.get_bounds`) 실측: X∈[-50,50], Y∈[-50,50], Z≈0 (법선=로컬 Z, 평면=로컬 X-Y).
- 기존 회전 `(pitch=0, yaw=90, roll=90)` 적용 시 `get_actor_bounds` 실측(A1, 스케일 X=6.48/Y=2.59/Z=1로 축소 적용 후에도 동일 패턴):
  - world X 범위 폭=0 (두께 없음), world Y 범위 폭=648(=100×6.48, 로컬X가 매핑), world Z 범위 폭=259(=100×2.59, 로컬Y가 매핑).
  - 즉 **스프라이트 평면의 법선(로컬Z)이 world X를 향함** → 카메라(위치(0,-7550,750), yaw=90, forward=world+Y)에서 보면 이 평면은 카메라를 정면으로 보는 게 아니라 **옆면(카메라 forward축과 평행)** 으로 서 있었다. `WorldPosToScreenCoords`로 world X 이동 시 화면X가 크게 변함을 확인(가로=world X 축)했고, 액터의 world X 폭이 0에 가까웠으므로 화면상 "얇은 막대"로 보인 것이 근본 원인.
  - 계산 검증: Roll=90(X축 회전)만으로 로컬Z(0,0,1)→world(0,-1,0), 로컬Y(0,1,0)→world(0,0,1), 로컬X 불변. 여기에 Yaw=90(Z축 회전)이 추가로 걸리면 이 결과가 다시 90도 회전해 로컬Z가 world+X를 향하게 됨 → **yaw=90이 불필요·유해했음**.
- **정정된 회전값: `(pitch=0, yaw=0, roll=90)`** — 로컬X→world X(가로), 로컬Y→world Z(세로), 로컬Z(법선)→world -Y(카메라 방향) 매핑을 실측으로 확인:
  - `get_actor_bounds` 재실측(A1, yaw=0/roll=90/scale X=6.48,Y=2.59,Z=1): world X폭=648(가로), world Y폭=0(법선, 카메라향), world Z폭=259(세로). 가로:세로=648:259=**2.503:1**로 목표 2.5:1 정확히 일치.
  - `WorldPosToScreenCoords`로 발-머리(세로)/좌-우(가로) 스크린 변화량 실측: 세로Δy=0.1469(정규화), 가로Δx=0.2629(정규화) — 화면 픽셀 환산(1920×1080) 시 가로 504.8px : 세로 158.7px ≈ 3.18:1로 world비율(2.5:1)보다 화면상 다소 넓게 보이나(카메라 원근 투영 특성, pitch=-6 및 캐릭터별 카메라 거리차 영향), **"세로로 늘어난 막대" 문제는 완전히 해소**되고 정상적인 인체 비율(가로<세로 아님, 가로가 넓지 않고 오히려 자연스러운 원근 폭)로 전환됨.

### (b) 최종 적용값 — 8기 전체

전원 동일 회전 `(pitch=0, yaw=0, roll=90)`, 동일 스케일 `(X=6.48, Y=2.59, Z=1.0)`로 갱신(기존 X=6.48/Y=6.48/Z=2.59에서 변경). `set_actor_transform`은 미지정 필드도 identity로 덮어씀을 실측으로 확인했으므로 **location/rotation/scale 전체를 매번 명시**해서 호출함(스키마 설명과 실제 동작 불일치, 주의 필요).

### (c) A2 지형 튐 — 원인 재확인 및 해결

- A2(X=-450) 위치의 콜리전을 배제하고(액터를 Z=5000으로 임시 격리) `trace_world`로 순수 지형만 재트레이스: X=-450, Y=-6600 지점 지형고도=847.32로, 인접 지점(X=-600:523.32, -300:504.58, 0:486.61, 600:485.83)보다 **실제로 320~360cm 높은 국지 둔덕이 지형에 존재**함을 확인(자기 콜리전 간섭이 아니라 실제 지형 기복임을 확정 — 최초엔 자기간섭 가설을 세웠으나 A1/A3/B4 등 타 지점도 액터 유무와 무관하게 트레이스 결과가 일관됨을 교차검증하여 기각).

### (e) 스크린샷 시도 결과 — 환경 한계 재확인

- `CaptureViewport`(annotations 명시적으로 전부 0/비활성화, bShowUI=false)로 1회 재시도 → 결과 약 480만 문자(base64 PNG)로 MCP 파이프라인 토큰 한도 초과, 파일로 자동 저장됐으나 해당 파일도 단일 라인이 너무 길어 Read 도구로 청크 분할 불가(offset/limit이 라인 단위라 무력화됨).

- 전역 규칙(3회 미만 시도 후 중단)에 따라 스크린샷 확보는 중단하고, 대신 `WorldPosToScreenCoords` 8기 전원 발/머리 실측(위 표)으로 화면 안착을 검증 완료. **스크린샷 파이프라인 자체는 verifier 단계에서 별도 방법(예: 프로젝트에 콘솔 명령 실행 MCP 툴 추가, PIE 내 스크린샷 API 등) 재검토 필요 — scene-builder 권한 밖, Director/오너 보고 사항.**
