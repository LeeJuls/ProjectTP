## 1. 증상 재확인

`raw/S3_비율수정.png`를 Read로 확인. 8기 스프라이트 중:
- **글리치 있음**: 왼쪽 검은 캐릭터, 중앙 초록 캐릭터, 오른쪽 갑옷 캐릭터들 — 가로 스캔라인/찢김
- **선명함**: 회색 궁수, 빨강 도적, 분홍머리 캐릭터

패턴: 글리치가 발생한 캐릭터는 공통적으로 **저채도·어두운 톤**, 선명한 캐릭터는 **밝고 고대비**. 비율·배치는 정상(변경 안 함).

## 2. 원인 진단

### 가설 검증 순서
1. **TAA/TSR smear (검증 결과: 유력)** — `SearchCVars`로 `r.AntiAliasingMethod` 조회 → 값 `4`(TSR, Temporal Super-Resolution, UE5 기본값).
2. 관련 cvar 추가 조회:
   - `r.TSR.ThinGeometryDetection` = 0 (Nanite Foliage 비활성 시 기본값)
   - `r.TSR.ThinGeometryDetection.Coverage.ShadingRange` = 3
   - `r.TSR.ShadingRejection.Mode` = 1, `r.TSR.ShadingRejection.Flickering` = 1
   - `r.TemporalAAFilterSize`=1, `r.TemporalAACurrentFrameWeight`=0.04 등 TAA 계열 파라미터도 확인(TSR이 내부적으로 이 인프라 일부 공유)

### 결론
TSR의 히스토리 리젝션/씬 리젝션 휴리스틱(`ShadingRejection`, `ThinGeometryDetection`)이 **Masked + TwoSided 얇은 쿼드**(HD-2D 픽셀 스프라이트의 전형적 구조, S2에서 확인된 `M_Sprite_Flipbook_Lit`)에서 프레임 간 히스토리 누적이 불안정해지며, 특히 저휘도/저대비 영역에서 이 불안정성이 가로 스캔라인 형태로 시각화되는 것으로 판단. 고대비·밝은 색상 캐릭터는 TSR의 shading rejection 임계값을 넘지 않아 상대적으로 안정적으로 보임 — 증상 패턴과 일치.

가설 2(카메라 각도), 가설 3(Masked/TwoSided Z-fighting)은 배치·비율 정상 확인 및 원인 패턴(색상별 선택적 발생)과 불일치하여 우선순위 하향.

## 3. 적용한 수정

### 변경 방법
cvar를 직접 SET하는 툴이 MCP 세트에 없어(`SearchCVars`는 조회 전용), 프로젝트 세팅을 통해 변경:

```
ConfigSettingsToolset.SetSectionProperties(
  containerName="Project", categoryName="Engine", sectionName="Rendering",
  propertiesJson={"defaultFeatureAntiAliasing": "AAM_FXAA"}
)
```

### 원래값 (되돌릴 때 사용)
```json
{"defaultFeatureAntiAliasing": "AAM_TSR"}
```

### 반영 검증
`SearchCVars("AntiAliasingMethod")`로 즉시 확인:
- 변경 전: `r.AntiAliasingMethod` = `4` (TSR)
- 변경 후: `r.AntiAliasingMethod` = `1` (FXAA)

프로젝트 세팅 변경이 에디터 뷰포트 cvar에 **즉시(에디터 재시작 없이) 반영**됨을 확인. FXAA는 non-temporal 방식이라 TSR/TAA의 히스토리 누적 아티팩트(스캔라인 글리치)가 구조적으로 발생하지 않음 — 다만 FXAA는 TSR 대비 안티에일리어싱 품질(엣지 스무딩)이 낮아지는 트레이드오프 있음.

## 4. ⚠ 미완 사항: 스크린샷 before/after 검증 불가 (도구 제약)

`CaptureViewport` 결과(base64 PNG)가 약 430~480만자로, MCP 반환 토큰 한도를 넘어 자동으로 tool-results txt 파일로 저장됨. 그러나 **본 세션(hd2d-art-director 서브에이전트)에는 Bash/코드실행 도구가 제공되지 않아** 이 파일을 열어 base64를 디코딩할 방법이 없었다.

시도한 우회로(모두 실패):
- Read 도구로 파일 직접 읽기 → 토큰 초과, 한 줄이라 offset/limit 무의미
- Grep으로 문자 범위 슬라이싱 → 긴 매치 라인은 "Omitted long matching line"으로 생략됨
- `editor_toolset.toolsets.programmatic.ProgrammaticToolset.execute_tool_script`(sandboxed python) → 허용 모듈이 `copy, re, time, json, math, datetime`뿐이라 `base64`/파일 I/O(`open`) 불가
- `AssetTools.write_file`/`read_file` → 텍스트 전용, 프로젝트 경로 제한

**따라서 FXAA 변경이 실제로 글리치를 없앴는지는 육안 확인하지 못했다.** 원인 진단(TSR)과 수정 방향(FXAA 전환)은 cvar 레벨에서 근거가 명확하나, 최종 시각적 검증은 남아있음.

### 다음 조치 제안 (택1, 오너 판단)
- **A안**: Bash 도구가 있는 세션(예: art-pipeline 에이전트 또는 verifier)에 이관해 동일 카메라(`location=(0,-7550,750)`, `rotation=(pitch-6,yaw90,roll0)`)로 재캡처 → before(`raw/S3_비율수정.png`)와 대조
- **B안**: Director가 이 에이전트(hd2d-art-director)에 Bash 도구 권한을 추가 부여
- 변경 자체(FXAA)는 이미 적용된 상태로 유지 — 검증 후 문제 있으면 위 원래값으로 즉시 원복 가능
