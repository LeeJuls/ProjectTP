## 알파 — 클라이언트 단독 완결 루프
**대기실(뽑기·편성) → 전투(턴제 완성형 vs AI) → 결과 → 대기실 순환 + 로컬 저장**

| 요소 | 알파 범위 | 오너 결정 근거 |
|---|---|---|
| 전투 | 현행 턴제 + **데미지·HP·사망·승패** + 스킬 소수 + **간단 AI** | "턴제 완성형" — ATB는 베타(서버)와 함께 설계해 재작업 1회로 압축 |
| 캐릭터 획득 | **재화 없는 뽑기 버튼 1개** → 누르면 **캐릭터 랜덤 조합 생성**(heroes99 파츠 스킨·의상·머리·무기·얼굴을 랜덤 조합해 신규 캐릭터 1기 지급) | **★ 이 랜덤 조합 생성 시스템이 알파의 핵심 개발물**(오너 명시). 베타의 "캐릭터 조립"은 같은 엔진을 유저 주도 파츠 선택으로 여는 것 |
| 대기실 | 보유 목록·파티 편성(4명)·뽑기 버튼 | |
| 결과 | 승패 표시 → 대기실 복귀(루프 폐쇄) | 보상 연결(예: 승리 시 뽑기)은 개발 계획에서 확정 |
| 저장 | 로컬 SaveGame(보유 캐릭터·파티) | |
| 기술 선결 | **UI 체계 정면돌파(UMG·SetText 문제)** — 대기실/HP바/결과는 월드공간 버튼으로 불가 | |
| 제약 | UI는 **터치 친화적으로**(버튼 크기·해상도 대응) — 베타 플랫폼 결정의 자유도 확보 | |

**완료 기준(DoD)**: 외부인이 설명 없이 켜서 "뽑고 → 편성하고 → AI와 싸워 이기고 → 결과 보고 → 대기실로" 루프를 혼자 돈다. 뽑을 때마다 **다른 조합의 캐릭터**가 나온다.

### 핵심 시스템: 캐릭터 랜덤 조합 생성 (기술 방향 메모)
- **사전 굽기 불가**: 현행 파이프라인(Pillow 오프라인 합성→임포트)은 조합 폭발(스킨×의상×머리×무기×얼굴 = 수천+)로 뽑기에 못 쓴다 → **게임 내 조합** 필요.
- **유력안**: 파츠 레이어 5장을 겹쳐 그리는 **합성 머티리얼**(모든 heroes99 파츠 시트가 동일 8×17 그리드 — SubUV 수학 공유, 알파-오버 순서 합성). 대안: RenderTarget 1회 굽기 / 소규모 사전 합성 풀(폴백). **확정은 알파 개발 계획의 선행 스파이크에서.**
- **캐릭터 = 레시피**: 저장·전송 단위는 이미지가 아니라 파츠 ID 조합 `{스킨, 의상, 머리, 무기, 얼굴}`. 로드 시 레시피로 재조립 — 베타의 유저 조립·서버 전송·(후일) 컬렉션의 공통 토대.
- 스탯: 알파는 최소(공통 기본치 수준). 파츠→스탯 연동은 베타 후보.

## 베타 — 서버 전투·캐릭터 시스템·플랫폼
- **서버 권위 전투 + ATB 전환**(본설계 §5·6 — 선후판정·시간동기가 심장)
- **캐릭터 조립·캐릭터 기본 시스템** — 알파의 랜덤 조합 엔진을 **유저 주도 파츠 선택**으로 확장 + 스탯·성장 등 캐릭터 시스템 본격화
- 최적화 / **Windows·모바일 확정** / 2인 테스트(초기 IP 직결, 매치메이킹은 후순위)
- **베타 진입 직전 네트워크 스파이크**: 리슨서버로 ATB 게이지 1개 동기화 실험(오너 결정 시점) — 베타 규모 산정의 도박 방지. **실측 목표**: 매치당 서버 판정 비용 + 서버당 동시 세션 한계(진짜 병목=계산 아닌 세션 관리 — [[알파_개발계획]] §2.56 서버 권위 범위 참조)

### 네트워킹·계정 아키텍처 결정 (오너 확정 2026-07-12)
위 "리슨서버 P2P ↔ 전용서버" 갈림길 중 **리슨서버 P2P로 확정**. 4종 스택:
- **계정**: Steam ID를 플레이어 식별자로 사용(Steamworks — 자체 계정 시스템 없음).
- **매칭**: Epic Online Services(EOS) — 로비/세션/매치메이킹 + P2P NAT 통과 릴레이. 무료 제공(수수료 없음, 인디 PvP에 흔한 선택).
- **전투 실행**: EOS P2P 위 리슨서버 — 매칭된 두 플레이어 중 한쪽 클라이언트가 그 매치의 판정 권위(=서버)를 겸함. [[알파_개발계획]] §서버 권위의 정확한 범위(결과만 서버 확정, 동작은 클라)와 그대로 정합.
- **영속 데이터**: EOS **Player Data Storage**(무료) — "골드만 판간 이월"([[projectTP_허브]] 결정 로그)을 여기 저장. 로컬 저장 대비 변조 리스크 없이, 서버비 0 유지하면서 이월 데이터 보관.
- **비용**: 상시 대여하는 전용 서버가 없어 매치 실행+데이터 저장 전부 **사실상 0**(Steamworks 최초 등록비 $100은 별개, 운영비 아님).

**정확히 0은 아닌 지점 (반영 필요)**:
1. **호스트 우위 리스크**: 리슨서버 = 플레이어 한쪽이 그 매치의 "서버"를 겸함 → [[알파_개발계획]]의 "서버가 확정하니 클라 변조 무의미" 안전망이 **호스트 본인에게는 안 통함**(자기가 서버). 경쟁 PvP라 짚어야 할 트레이드오프 — 알파/베타 소규모는 감수, 문제 커지면 호스트 로테이션·검증 로직 후순위 검토.

## 라이브 — 배포

## 리스크 메모 (본설계 경고 승계)
- 서버 권위 PvP(선후판정·핑 공정성)가 프로젝트 최대 난제 — 베타의 무게중심. 알파에서 이를 잊지 않기 위해 스파이크를 베타 직전에 고정 배치.
- C++ 이관 검토 시점 = 베타 초입(ATB 틱·네트워크·스탯 DT가 트리거 — 턴제전투MVP 때 예약된 결정).
