## 알파 — 클라이언트 단독 완결 루프 (드래프트 단판 vs AI)

> **8화면 루프** = 루트 4(`WBP_Title`·`WBP_Lobby`·`WBP_Draft`·`WBP_BattleHUD`) + 오버레이 4(`WBP_Matching`·`WBP_Transition`·`WBP_Options`·`WBP_Result`). 주 흐름([[UI_화면_규약]] §A-3):
> **Title → Lobby →(Start → Matching → Transition)→ Draft(Roll/Pick/Ban) → BattleHUD(4:4 턴제 vs AI) → Result → 재시작(Draft 재진입) or 로비복귀**
> Matching·Transition·Options은 루트 위에 얹히는 **오버레이**(주 흐름 화살표에 안 나타남). 화면 스펙 전체의 단일 출처는 [[UI_화면_규약]] §A-2 — 이 로드맵은 **스코프만** 규정한다.

**완료 기준(DoD)**: 외부인이 설명 없이 켜서 **"Roll(6장 지급·리롤) → Pick(5픽) → Ban(1밴·교차밴) → 광폭화 포함 4:4 턴제로 AI와 싸워 이기고 → Result(승패·골드) → 재매칭(Draft 재진입) or 로비복귀"** 루프를 혼자 돈다. 매판 1000풀에서 **다른 6장**이 지급된다. 광폭화는 **스코프 포함**이되 정확공식은 balance 재검증 후 구현한다.

### 핵심 시스템: 1000개 사전 대량생성 풀 + 등급축(배율압축)
- **1000개 사전 대량생성(빌드타임·오프라인) = 풀 데이터(레시피) 생성**: 방향성2는 런타임 무한 조합이 아니라 **경계가 명확한 1000개 풀을 사전 정의**한다(대비표 "무한 조합 뽑기 → **1000개 사전 대량생성**"). 생성 대상은 **데이터**(캐릭터 ID·포지션·스킬조합·등급 — [[기획_방향성]] 생성 절차: 스크립트 배치+중복 재배치)이지 이미지 굽기가 아니다.
- **캐릭터 = 레시피, 렌더 = 런타임 합성(A0② 검증 완료)**: 저장 단위는 이미지가 아니라 파츠 ID 조합 `{스킨, 의상, 머리, 무기, 얼굴}` + 등급·포지션 태그. 화면 표시는 **A0②에서 검증된 런타임 합성 머티리얼**(레이어 겹치기)이 레시피로 그린다 — 풀 조정이 CSV 수정만으로 끝나고(1000텍스처 재임포트 불필요), 동시 표시 최대 ~14체(드래프트 6+전투 8)라 런타임 비용 무시 가능. **베타는 같은 합성 엔진을 유저 주도 조립으로 확장**([[알파_개발계획]] S2·A0② 기록과 정합).
- **등급축(GradeId) — 유지 + 배율압축**(2026-07-13 오너 재확인 → Director 판단): 등급차를 **작은 신호/양념**으로 두는 **압축형 고정배율**(등급간 스탯차를 좁게). **패리티 보장(매치별 등급분포 강제) 아님** — "고등급 뽑으면 유리" 기대감·"저등급으로 고등급 격파" 역전서사·밴 우선순위 휴리스틱을 살리되, 판이 순수 RNG로 갈리는 "불합리 폭발"만 억제. **정확한 압축 배율 수치는 balance-designer 후속**([[알파_개발계획]] §4.5) — 여기선 방향(압축형 고정배율, **패리티 로직 없음**)만 확정.
- **스탯·ID**: 알파 스탯은 최소(공통 기본치 + 등급 배율 보정). 포지션(탱/딜/힐)→스탯/킷 연결·등급×포지션 확률표는 balance 소관. **캐릭터 ID 스킴에는 등급 세그먼트만 반영**하고 포지션 인코딩은 **미확정으로 남긴다**(상세 = [[데이터_서버_규약]] §4, 별도 갱신 예약 — 두 축을 동시에 못박지 않기 위함).

## 베타 — 서버 전투·캐릭터 시스템·플랫폼
- **서버 권위 전투 + ATB 전환**(본설계 §5·6 — 선후판정·시간동기가 심장)
  - **왜 ATB가 여전히 베타 축인가**: 알파↔베타 단계 구분 축은 "PvP냐"가 아니라 **"턴제(정지)냐 실시간 ATB냐"**다. 방향성2는 획득·매치 모델(드래프트·밴)만 바꿨고 전투는 **4:4 턴제 MVP 재사용**으로 명시(대비표)했으므로, 턴제→ATB 전환은 방향성2와 무관하게 베타에 그대로 남는다(서버 전환과 묶어 재작업 1회로 압축).
- **캐릭터 조립·캐릭터 기본 시스템** — 알파의 **파츠 레시피·사전생성 파이프라인을 런타임 유저 주도 조립**으로 전환·확장 + 스탯·성장 등 캐릭터 시스템 본격화
- 최적화 / **Windows·모바일 확정** / 2인 테스트(초기 IP 직결, 매치메이킹은 후순위)
- **베타 진입 직전 네트워크 스파이크**: 리슨서버로 ATB 게이지 1개 동기화 실험(오너 결정 시점) — 베타 규모 산정의 도박 방지. **실측 목표**: 매치당 서버 판정 비용 + 서버당 동시 세션 한계(진짜 병목=계산 아닌 세션 관리 — [[데이터_서버_규약]] §3-B 서버 권위 범위 참조)

### 네트워킹·계정 아키텍처 결정 (오너 확정 2026-07-12)
위 "리슨서버 P2P ↔ 전용서버" 갈림길 중 **리슨서버 P2P로 확정**. 4종 스택:
- **계정**: Steam ID를 플레이어 식별자로 사용(Steamworks — 자체 계정 시스템 없음).
- **매칭**: Epic Online Services(EOS) — 로비/세션/매치메이킹 + P2P NAT 통과 릴레이. 무료 제공(수수료 없음, 인디 PvP에 흔한 선택).
- **전투 실행**: EOS P2P 위 리슨서버 — 매칭된 두 플레이어 중 한쪽 클라이언트가 그 매치의 판정 권위(=서버)를 겸함. [[데이터_서버_규약]] §3-B 서버 권위의 정확한 범위(결과만 서버 확정, 동작은 클라)와 그대로 정합.
- **영속 데이터**: EOS **Player Data Storage**(무료) — "골드만 판간 이월"([[projectTP_허브]] 결정 로그)을 여기 저장. 로컬 저장 대비 변조 리스크 없이, 서버비 0 유지하면서 이월 데이터 보관.
- **비용**: 상시 대여하는 전용 서버가 없어 매치 실행+데이터 저장 전부 **사실상 0**(Steamworks 최초 등록비 $100은 별개, 운영비 아님).

**정확히 0은 아닌 지점 (반영 필요)**:
1. **호스트 우위 리스크**: 리슨서버 = 플레이어 한쪽이 그 매치의 "서버"를 겸함 → [[알파_개발계획]]의 "서버가 확정하니 클라 변조 무의미" 안전망이 **호스트 본인에게는 안 통함**(자기가 서버). 경쟁 PvP라 짚어야 할 트레이드오프 — 알파/베타 소규모는 감수, 문제 커지면 호스트 로테이션·검증 로직 후순위 검토.
