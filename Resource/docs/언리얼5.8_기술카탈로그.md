# 📚 언리얼 5.8 기술 카탈로그 (projectTP 선별)

> UE 5.8 공식 문서를 폭넓게 읽고 **projectTP 스택에 실제로 닿는 것만** 골라 3분류한 참고 자료.
> **이건 실행 계획이 아니다.** 구체적 병목이 생겼을 때 "그때 뭘 꺼내 쓸지"를 미리 조사해둔 카탈로그다. **여기 있다고 채택된 게 아니다.**

## 0. 조사 요약 (먼저 읽을 것)

### ★ 헤드라인 1 — 5.8은 우리 영역을 거의 건드리지 않았다

[UE 5.8 릴리즈노트](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-5-8-release-notes) 전문을 검색·확인한 결과:

- **Blueprint 전용 섹션이 없다.** 릴리즈노트 최상위 섹션은 What's New / Rendering / Character and Animation / MetaHuman / Worldbuilding / PCG / Framework / Developer Iteration / Audio / Platform / Content Pipeline / Editor / Production Rendering Pipeline 뿐이다.
- **"DataTable" · "CSV" · "Source Control" · "localization" · "StringTable" 전문 검색 결과 = 0건.** 이건 "못 찾았다"가 아니라 **확정된 부정(negative) 결과**다 — 5.8은 이 영역에 변경이 없다.
- **UMG/Slate 전용 섹션도 없다.** UI 관련 5.8 변경은 **Enhanced Input ↔ Common Input/UI 통합 + Input Debugger** 하나뿐이다.

→ **함의**: 진행 중인 [[features/전투완성/청사진|A1 전투완성]]에 "5.8 신기능으로 더 잘하는 법"은 **사실상 없다.** 우리 CSV→DataTable 임포트도, BP 전투 로직도, UMG HUD도 5.8 기준 **변경 없음**으로 안심하고 진행하면 된다.

### ★ 헤드라인 2 — 우리가 피 흘려 알아낸 함정의 **공식 근거**를 찾았다

이번 조사의 가장 큰 수확은 신기능이 아니라 **이미 겪은 문제의 공식 문서 근거**다. 전부 §1에 상세:

| 우리가 겪은 것 | 공식 문서가 말하는 것 |
|---|---|
| MCP `add_event` **병렬 호출 시 이름 충돌**([[언리얼_MCP_실전노하우]] §8 함정⑤) | 공식 Unreal MCP 문서: **"calls run serially on the game thread (no overlapping tool calls)"** — 병렬 호출은 애초에 지원되지 않는다 |
| Bloom 1.3 → **화면 워시아웃**([[HD2D_기법_지식베이스]]) | Standard Bloom은 **"not conservative (it can result in an overall brightening of the image)"** — 그리고 진짜 레버는 Intensity가 아니라 **Threshold**다 |
| AutoExposure min=max=1.0으로 노출 고정 | 공식: **"If Min Brightness is equal to Max Brightness, auto exposure is disabled"** — 우리 방식이 문서화된 정식 비활성화 방법이었다 |
| 런타임 `SetText`가 렌더에 무반영(미해결) | UMG 공식: **"바인딩을 걸어둔 프로퍼티에 Set을 직접 호출하면 바인딩이 깨진다"** — 같은 계열 함정일 가능성 |

## 1. 지금 쓸 것 (6건)

> 지금 돌고 있는 작업(A1 전투 로직 · HD-2D 룩 · UMG HUD)에 **당장 접점이 있는 것.**

### 1-1. ★ Widget Component + **UMG 프로퍼티 바인딩 함정** — HP게이지·부동 데미지숫자

- **문서**: [Widget Components](https://dev.epicgames.com/documentation/en-us/unreal-engine/widget-components-in-unreal-engine) · [Property Binding for UMG](https://dev.epicgames.com/documentation/en-us/unreal-engine/property-binding-for-umg-in-unreal-engine) · [Widget Interaction Components](https://dev.epicgames.com/documentation/en-us/unreal-engine/umg-widget-interaction-components-in-unreal-engine)

**문서에서 읽은 사실 (Widget Component)**
- `Space` 프로퍼티: **World Space**("메시로 렌더되어 **다른 오브젝트에 가려질 수 있다**") vs **Screen Space**("월드 바깥에 렌더되어 **절대 가려지지 않는다**").
- 위젯을 **렌더타겟에 그린 뒤** 그 렌더타겟을 월드에 표시하는 구조.
- `Draw Size`, `Pivot`, **`Redraw Time`**(0=매 프레임 재드로, 1=1초마다), `Manually Redraw`, `Window Focusable`.
- **`Draw at Desired Size`는 "Expensive if is done every frame"** 이라고 문서가 직접 경고한다.
- 월드 위젯과의 **클릭 상호작용은 Widget Interaction Component**(레이캐스트)가 별도로 필요하다.

**★문서에서 읽은 사실 (Property Binding — 우리에게 가장 중요)**
> **"If you bind a property of a Widget and then directly call the Set function on that Widget, it will break the binding."**
> (프로퍼티에 바인딩을 걸어놓고 그 위젯에 Set을 직접 호출하면 **바인딩이 깨진다**.)

- 바인딩 방식은 2종: **Function 바인딩**(Bind 클릭 → 함수 자동 생성, 반환값이 프로퍼티로) / **Property 바인딩**(BP 변수 직결, 변수 갱신 시 위젯 자동 갱신).

**projectTP 접점**
- **[[features/전투완성/청사진|전투완성]] F3(HP게이지)** — 현재 유닛 머리 위 표시는 `TextRenderComponent` 임시 구현이고, [[언리얼_MCP_실전노하우]]에 **단면 렌더로 안 보이는 함정**·**런타임 `SetText`가 렌더에 무반영되는 미해결 버그**가 기록돼 있다. WidgetComponent가 이 용도의 공식 경로다.
- **★"바인딩 + Set 혼용 = 바인딩 파괴" 는 우리 `SetText` 미해결 버그와 같은 계열일 수 있다.** F3/U단계에서 HP 텍스트·게이지를 배선할 때 **바인딩과 Set 중 하나만 골라 쓸 것**(둘을 섞으면 조용히 깨진다 — 우리가 이미 겪은 증상과 정확히 일치).
- [[UI_화면_규약]] §E의 후보 컴포넌트 **`WBP_DamageNumber`·`WBP_UnitFrame`**(전투 HUD 명세 소관)이 정확히 이 기능 위에 올라간다.
- **★우리 함정 2개와 반드시 교차 확인**(둘 다 [[언리얼_MCP_실전노하우]] 기록):
  1. **비균등 스케일 부모 전단(shear) 함정(§7 함정②)** — `BP_BattleSpawnPoint`의 Sprite 컴포넌트는 `scale(6.48, 2.59, 1)` + `roll 90`이다. 그 자식으로 WidgetComponent를 붙이면 같은 전단이 재발한다 → **`bAbsoluteLocation/Rotation/Scale = true` 선적용이 사실상 필수.**
  2. **클릭 방패 함정(§10 함정⑧)** — World space 위젯은 콜리전 쿼드다. 유닛 클릭 트레이스를 가로챌 수 있으니 상호작용이 필요 없으면 **`NoCollision`**.
- **World vs Screen 판단 재료**: 우리 무대는 스프라이트 쿼드가 3cm 간격으로 촘촘히 스태거돼 있어 **World space는 오클루전 리스크가 실재**한다. Screen space는 절대 안 가려지지만 3D 원근이 안 붙는다.
- **성능**: 4:4 = **동시 8개 HP게이지**. `Redraw Time`을 반드시 손볼 것(매 프레임 재드로는 낭비 — HP는 피격 순간에만 바뀐다).

### 1-2. ★ DataTable — **`Import Key Field`** · 캐싱 금지 · `Get Data Table Row`

- **문서**: [Data-Driven Gameplay Elements](https://dev.epicgames.com/documentation/unreal-engine/data-driven-gameplay-elements-in-unreal-engine) · [Get Data Table Row (BP)](https://dev.epicgames.com/documentation/en-us/unreal-engine/BlueprintAPI/Utilities/GetDataTableRow)

**문서에서 읽은 사실**
- 행 구조체는 **`FTableRowBase` 상속** 필수. CSV 컬럼 헤더 ↔ 구조체 멤버 1:1.
- **"The first column is expected to be named `Name`"** — 기본 행 키 컬럼명은 `Name`이다. **★단, 임포트 옵션 `Import Key Field`로 어느 컬럼을 키로 쓸지 오버라이드할 수 있다**(CSV 기본값 `Name`, JSON 기본값 `field`).
- 임포트 옵션 3종: **`Ignore Extra Fields`**, **`Ignore Missing Fields`**, **`Import Key Field`**.
- 에셋 레퍼런스 컬럼은 **타입을 큰따옴표로 감싸야** 한다(`"Texture2D'/Game/Textures/Icon'"`) — 안 그러면 임포터가 평문으로 오해한다.
- **★caveat(원문)**: "All structures and curves returned **should not be cached further than the local scope of a function**" — 리임포트한 데이터가 즉시 반영되게 하려면 **행 구조체를 함수 밖(멤버 변수 등)에 들고 있지 말 것.**
- **`Get Data Table Row`** BP 노드는 **`Row Found` / `Row Not Found` 두 개의 exec 핀**으로 갈라진다(없는 행 조회가 **크래시가 아니라 실패 분기**로 안전하게 떨어진다). `Out Row`는 와일드카드 구조체 핀(우클릭 → Split Struct Pin).
- JSON도 대체 임포트 포맷으로 공식 지원(CSV가 주 경로).

**projectTP 접점**
- **[[features/전투완성/청사진|전투완성]] F2(데이터 부트스트랩) · F3(스탯 로드)** 에 직결. `DT_JobStats`·`DT_Skills`·`DT_Motions`·`DT_Strings` 전부 해당.
- **★즉시 점검 1 — `Import Key Field`**: 우리 CSV 규약([[데이터_서버_규약]] §4)은 **0번 컬럼 = `Id`**(`Name`이 아님)다. UE 기본값은 `Name`이므로, **각 DataTable의 임포트 설정에서 `Import Key Field = Id`가 실제로 잡혀 있는지 확인**해야 한다. 안 잡혀 있으면 RowName 의미가 우리 `Id` 외래키 규약과 어긋난 채로 굴러가고 있을 수 있다(지금은 우연히 동작해도, 리임포트·행 추가 시 터진다).
- **★즉시 점검 2 — 캐싱 금지**: F3 스탯 로드가 행 구조체를 **유닛 멤버 변수에 캐싱**하는 패턴이면 공식 caveat와 충돌한다. 증상은 **"CSV를 고치고 리임포트해도 값이 안 바뀌는"** 유령 버그 — 밸런스 튜닝을 반복할 우리에게 특히 아프다.
- **`Row Not Found` 핀**: 우리 외래키(`SkillId`·`MotionRow` 등) 참조 무결성이 깨졌을 때 **크래시 대신 방어 분기**로 처리할 수 있다는 뜻 → gameplay-engineer의 에러 처리 패턴 근거.

### 1-3. ★ Format Text — `{0}`은 Epic 공식 관용구, 그러나 **"동적 포맷 문자열" 함정**이 있다

- **문서**: [Format Text (BP)](https://dev.epicgames.com/documentation/en-us/unreal-engine/BlueprintAPI/Utilities/Text/FormatText) · [FText](https://dev.epicgames.com/documentation/unreal-engine/ftext-in-unreal-engine) · [Text Localization](https://dev.epicgames.com/documentation/unreal-engine/text-localization-in-unreal-engine)

**문서에서 읽은 사실**
- **`{0}` 0-index 포지셔널이 Epic 자신의 문서 예시**다: `FText::Format(LOCTEXT("ExampleFText", "You currently have {0} health left."), CurrentHealth)`. → 우리 [[데이터_서버_규약]] §1-B의 `{0}` 표준화는 **우회책이 아니라 정식 관용구**임이 확인됐다.
- **★함정(원문 요지)**: Format 핀이 **리터럴 문자열이면 인자 핀이 자동 생성**된다. 그러나 **Format 핀이 다른 노드에 연결(동적)돼 있으면 인자 핀이 자동 생성되지 않고, 노드 Details 패널에서 인자를 수동 선언**해야 한다.
- 인자 핀이 받는 타입: **Byte, Integer, Float, Text, String, Name, Boolean, Object, ETextGender**.
- 자주 쓰는 패턴은 **`FTextFormat`으로 사전 컴파일**하면 성능상 유리하며, 컬처가 바뀌면 자동 재컴파일된다.
- ICU 기반의 **복수형/성별 문법**도 지원한다(`"{NumCats}|plural(one=is,other=are)"`, `"{Gender}|gender(Le,La)"`) — 단 이건 **명명 토큰**을 쓰는 별도 기능이며, 단순 `{0}` 치환과는 다른 층이다.
- DataTable 텍스트 임포트는 **`OnPostDataImport`로 후처리해 결정론적 키를 부여**할 수 있다고 문서가 명시.

**projectTP 접점**
- **★[[features/전투완성/청사진|전투완성]] F7(스킬메뉴) 직격**: 우리 스킬 설명은 **`DescKey` → `strings.csv` → DataTable에서 런타임에 꺼낸 문자열**이 포맷 템플릿이 된다. 즉 **Format 핀이 "동적"** 이다 → **인자 핀이 자동 생성되지 않는다.** [[데이터_서버_규약]] §1-B가 "BP 배선은 F7 몫"이라고만 적어둔 그 배선이 **이 수동 선언 절차를 반드시 포함**해야 한다. 모르고 들어가면 "왜 핀이 안 생기지"로 시간을 태운다.
- §1-B ④의 엣지케이스 P1(플레이스홀더 개수 ≠ 값 개수)도 이 수동 선언 구조 위에서 다뤄야 한다.
- 복수형/성별 ICU 기능은 **지금 불필요**(ko/ja는 복수형 문법이 없고, en도 알파 문자열엔 해당 케이스 없음).

### 1-4. ★ 공식 Unreal MCP 문서 — **"게임스레드 직렬 실행, 병렬 호출 불가"가 문서에 명시돼 있다**

- **문서**: https://dev.epicgames.com/documentation/unreal-engine/unreal-mcp-in-unreal-editor

**문서에서 읽은 사실 (5.8 신규 Experimental 플러그인)**
- 에디터 프로세스에 **내장된 MCP 서버**. 기본 바인딩 **`http://127.0.0.1:8000/mcp`** — 우리가 쓰는 바로 그 주소다.
- 지원 클라이언트에 **Claude Code**가 명시돼 있고, `ModelContextProtocol.GenerateClientConfig ClaudeCode` 콘솔 명령으로 설정을 생성한다.
- **명시된 제약**:
  - **HTTP/SSE 전용**(stdio·WebSocket 없음) → [[언리얼_MCP_실전노하우]] §1의 "raw HTTP 직접호출 금지(SSE 스트림이라 urllib 파싱 실패)"와 정확히 일치.
  - **loopback 전용 · 인증 계층 없음.**
  - **★"calls run serially on the game thread (no overlapping tool calls)"** — **툴 호출은 게임스레드에서 직렬 실행되며 겹칠 수 없다.**
  - Live Coding이 새 `UFUNCTION` 선언을 전파하지 못함.
  - "Toolset Registry" 어댑터는 **에디터 전용**.

**projectTP 접점**
- **★[[언리얼_MCP_실전노하우]] §8 함정⑤의 공식 근본원인**: 우리는 "`add_event`를 **병렬로 호출하면** 이름이 무시되고 `커스텀이벤트` 기본명으로 생성된다"를 **실측으로** 알아냈고 "순차 호출하라"를 경험칙으로 남겼다. 공식 문서는 **애초에 툴 호출이 게임스레드 직렬이라 겹칠 수 없다**고 명시한다 → 우리 경험칙이 **버그 회피가 아니라 문서화된 설계 제약**이었다. **MCP 툴은 병렬 호출하지 마라**(에이전트가 "독립 작업은 한 메시지에 병렬로" 규칙을 MCP 툴에도 적용하려 할 때 이 항목을 근거로 막을 것).
- **Experimental 지위**는 우리 함정 목록(§2 파라미터 스키마 불일치, §7 함정③ 인라인 구조체 비결정성 등)이 **버그가 아니라 예상 가능한 미성숙**임을 뜻한다 → 엔진 마이너 업그레이드 때 MCP 툴 시그니처가 바뀔 수 있다는 리스크를 안고 간다. (서드파티 UmgMcp는 별개 — [[언리얼_MCP_실전노하우]] §21)
- **후속 확인 가치**: 우리 `.mcp.json`의 `unreal-mcp` 서버가 **이 공식 플러그인과 동일한 것인지**(툴 이름 `list_toolsets`/`describe_toolset`/`call_tool`이 문서의 "Toolset Registry" 용어와 일치) 한 번 대조해둘 것.

### 1-5. ★ 포스트프로세스 3종 — 우리 실측 세팅의 **공식 근거 + 놓치고 있던 레버**

- **문서**: [Bloom](https://dev.epicgames.com/documentation/en-us/unreal-engine/bloom-in-unreal-engine) · [Auto Exposure](https://dev.epicgames.com/documentation/unreal-engine/auto-exposure-in-unreal-engine) · [Cinematic Depth of Field](https://dev.epicgames.com/documentation/en-us/unreal-engine/cinematic-depth-of-field-in-unreal-engine) · [Color Grading and the Filmic Tonemapper](https://dev.epicgames.com/documentation/unreal-engine/color-grading-and-the-filmic-tonemapper-in-unreal-engine) · [Post Process Effects](https://dev.epicgames.com/documentation/en-us/unreal-engine/post-process-effects-in-unreal-engine)

**① Bloom — ★우리가 놓친 레버는 `Threshold`다**
- Bloom **Intensity는 단순 선형 스칼라**이고, 그 위에 **`Threshold`(휘도 컷오프, -1 = 전부 포함)** 가 따로 있다.
- 문서 원문: Standard 방식(1/2~1/32 해상도 가우시안 다중 블러)은 **"not conservative (it can result in an overall brightening of the image)"**.
- Convolution 방식(커널 기반)은 더 물리적으로 정확하지만 시네마틱·고사양 지향.
- **projectTP 접점**: [[HD2D_기법_지식베이스]]에 "Bloom 1.3은 워시아웃이었다 → 0.45로"라고만 적혀 있다. 문서는 **워시아웃의 원인이 Standard Bloom의 비보존적 특성**이며, **밝은 하이라이트만 골라 번지게 하는 정식 레버는 Intensity가 아니라 Threshold**임을 알려준다 → **hd2d-art-director가 Threshold를 올려 Intensity를 더 줄 수 있는지** 실험할 여지가 있다(도트 스프라이트의 밝은 부분만 은은히 번지는 옥토패스 룩).

**② Auto Exposure — 우리 방식이 공식 방식이었다**
- 문서 원문: **"If Min Brightness is equal to Max Brightness, auto exposure is disabled."**
- 메터링 모드는 Histogram(기본, 64-bin) / Basic(빠름). Exposure Compensation은 로그 스케일(-1 = 절반 밝기).
- **projectTP 접점**: 우리 `AutoExposure min=max=1.0`([[HD2D_기법_지식베이스]])은 **임시방편이 아니라 문서화된 정식 비활성화 방법**이다. 안심하고 유지.

**③ Depth of Field — ★틸트시프트는 Cinematic DoF 외에 길이 없다**
- **Gaussian DoF는 "has been removed for use with the Deferred Renderer and Desktop Forward Renderer and only supports mobile platforms"** (4.23부터). 데스크톱에서 보려면 Mobile Previewer로 전환해야 한다.
- Circle DoF는 엔진 enum(`DOFM_CircleDOF`)으로 남아있으나 현행 문서의 주 경로가 아니다.
- **Cinematic DoF가 유일한 데스크톱/콘솔 경로**: `Aperture(F-stop)`(낮을수록 얕은 심도), `Focal Distance`(cm), **`Diaphragm Blades`(4~16 — ★Epic/Cinematic 스케일러빌리티에서만 렌더된다)**, Filmback(센서 mm), Focus 모드(Manual/Tracking/None).
- **문서에 틸트시프트/미니어처 전용 가이드는 없다.**
- **projectTP 접점**: [[HD2D_기법_지식베이스]]의 HD-2D **5축 중 "틸트시프트 DoF"만 미완**("무대 규모에 맞춰 조정"으로만 적혀 있음). **정직한 결론**: 엔진이 "틸트시프트 버튼"을 주지 않는다. **Cinematic DoF의 낮은 F-stop + Focal Distance 조합으로 근사**해야 하며, Gaussian DoF는 데스크톱에서 **아예 못 쓴다**(이 사실을 모르면 삽질한다). `Diaphragm Blades`가 낮은 스케일러빌리티에서 무시된다는 것도 A/B 스크린샷 비교 시 함정.

### 1-6. Paper 2D Sprite Material — **엔진도 Masked+Lit 스프라이트를 네이티브 지원**(대조 레퍼런스)

- **문서**: https://dev.epicgames.com/documentation/unreal-engine/paper-2d-sprite-material-in-unreal-engine
- **문서에서 읽은 사실**: Paper2D 기본 머티리얼 8종 — `DefaultSpriteMaterial`, `DefaultLitSpriteMaterial`, **`MaskedLitSpriteMaterial`**, `MaskedUnlitSpriteMaterial`, `OpaqueLit/UnlitSpriteMaterial`, `TranslucentLit/UnlitSpriteMaterial`. **Lit 계열은 "레벨의 라이트 영향을 받는다"** 고 명시. 커스텀 Lit 머티리얼도 표준 Material Editor로 제작 가능.
- **projectTP 접점**:
  - 우리 `M_Sprite_Lit`(Masked+TwoSided+Lit)이 하는 일은 **엔진이 `MaskedLitSpriteMaterial`로 이미 표준화해둔 셰이딩 모델**과 동일하다 — 우리 방식이 이단이 아님을 공식 문서가 뒷받침한다.
  - **지금 유용한 이유**: art-pipeline이 **F0⑥ 마스터 머티리얼 `Freeze` 파라미터 신설**(DYING/BLOCK 정지용, [[features/전투완성/청사진]] 리스크2) 작업 중이다. 엔진 기본 Lit 스프라이트 머티리얼을 **대조군**으로 열어보면 우리 커스텀 그래프에 빠진 게 있는지 싸게 확인된다.
  - **대체가 아니다**: 우리가 Paper2D 컴포넌트를 안 쓰는 이유는 그대로 유효하다(§3-7).

## 2. 나중에 후보 (13건) — **재검토 시점 명시**

### 2-1. StateTree — BP로 확장 가능한 계층형 상태머신 (**5.8에서 개선됨**)
- **문서**: [StateTree 개요](https://dev.epicgames.com/documentation/unreal-engine/overview-of-state-tree-in-unreal-engine) · [5.8 릴리즈노트](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-5-8-release-notes)
- **사실**: "behavior tree의 Selector + state machine의 State/Transition을 결합한 **범용 계층형 상태머신**". **C++ 없이 BP로 확장 가능**하다고 문서가 명시(`UStateTreeTaskBlueprintBase`·`UStateTreeEvaluatorBlueprintBase`·`UStateTreeConditionBlueprintBase` 상속). **5.8 개선**: 명시적 시작 상태 정의, Compiler Manager, 트리 최신성 추적, 프로퍼티 바인딩 강화·역방향 바인딩.
- **projectTP 접점**: 우리 전투 상태머신(Idle→AwaitTarget→Executing→TurnEnd→End)과 **상태이상(STUN/ATK_DOWN)** 은 지금 **손으로 만든 BP 상태머신**이다([[features/전투완성/청사진]] F4~F5, E2 raw).
- **재검토 시점**: **A1 중에는 절대 손대지 마라**(F0~F3 완료, 상태 전이 계약 확정 — 갈아타면 전부 재검증). **A2에서 스킬 30원자·상태이상이 늘어 BP 분기가 감당 안 될 때** 재검토. 그때도 "BP 상태머신을 StateTree로 이관하는 비용 vs 얻는 것"을 실측해야 한다.

### 2-2. Data Validation 플러그인 + Asset Action Utility — CSV 참조 무결성 검증의 공식 경로
- **문서**: [Data Validation](https://dev.epicgames.com/documentation/en-us/unreal-engine/data-validation-in-unreal-engine) · [Blutility 모듈](https://dev.epicgames.com/documentation/unreal-engine/API/Editor/Blutility)
- **사실**:
  - 내장 플러그인. 확장 경로 2종: UObject에 `IsDataValid` 오버라이드 / **`UEditorValidatorBase` 파생**(C++·**Blueprint**·Python) — `CanValidateAsset` + `ValidateLoadedAsset`(각 경로에서 `AssetPasses`/`AssetFails` 호출 필수).
  - **에셋 저장 시 자동 실행**(기본 활성), CI에서도 실행, **헤드리스 커맨드릿**: `UnrealEditor-Cmd.exe <PROJECT>.uproject -run=DataValidation`.
  - **`UAssetActionUtility`**: 파생 클래스의 함수가 **Content Browser 우클릭 메뉴**에 뜬다("에셋 선택 → 우클릭 → 검증" 패턴).
  - **DataTable 행 전용 검증 기능은 없다**(외래키 체커 같은 건 직접 짜야 한다 — 이건 *메커니즘*이지 완제품이 아니다).
- **projectTP 접점**: [[데이터_서버_규약]] §4가 예고한 **검증 스크립트**(외래키 `<대상>Id` 실제 해석 여부, `#id_txt` 슬러그 자동 대조, enum UPPER_SNAKE 유효성)의 공식 뼈대. **BP로 짤 수 있어 BP-only MVP 원칙과 충돌하지 않는다**는 게 핵심.
- **재검토 시점**: **캐릭터 1000풀 생성(A2 이후)** 으로 손검증이 불가능해질 때. 알파 5스킬 규모에선 눈으로 본다.
- **⚠ 선결 확인**: 문서가 "Blueprint 검증기는 시작 시 자동 등록"이라 하면서 동시에 **"Data Validation system only runs C++ validation rules by default"** 라고도 적어 **자기모순**이다(§4). 커맨드릿에서 BP 검증기가 실제로 도는지 **먼저 1회 실증**할 것.

### 2-3. Editor Utility Widget (Beta) + Data Asset — **SettingTool의 공식 패턴**
- **문서**: [Editor Utility Widgets](https://dev.epicgames.com/documentation/en-us/unreal-engine/editor-utility-widgets-in-unreal-engine) · [Data Assets](https://dev.epicgames.com/documentation/en-us/unreal-engine/data-assets-in-unreal-engine)
- **사실**: EUW는 **UMG 기반**("you can set up Widgets in a Blueprint like you would for any other UMG Widget Blueprint"). Content Browser 우클릭 → Editor Utilities → Editor Utility Widget → 우클릭 **"Run Editor Utility Widget"** → Level Editor **Tools 드롭다운**에 등록. **Beta**. Data Asset은 Property Specifier로 멤버 변수를 **Details 패널에서 바로 편집**하게 노출(별도 툴 없이도 편집 가능).
- **projectTP 접점**: [[데이터_서버_규약]] §1이 예고한 **"언리얼 내 SettingTool"**(카메라 6종 등 흩어진 튜닝값 통합)의 **정확히 그 공식 패턴**이다. 우리가 후보로 적어둔 "Editor Utility Widget + DataAsset 조합"이 문서상 유효함이 확인됐다. **UmgMcp로 이미 위젯을 자동 배치하고 있으므로 제작 비용도 낮다**(같은 스킬셋).
- **재검토 시점**: 튜닝값이 더 흩어져 **오너가 값 하나 바꾸려고 BP를 열어야 하는 고통이 실제로 발생**할 때(A2 이후 예상).

### 2-4. Material Parameter Collection + Material Function — 스프라이트 머티리얼 공유 로직·전역 파라미터
- **문서**: [Material Parameter Collections](https://dev.epicgames.com/documentation/en-us/unreal-engine/using-material-parameter-collections-in-unreal-engine) · [Material Functions](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-material-functions-overview) · [Instanced Materials](https://dev.epicgames.com/documentation/en-us/unreal-engine/instanced-materials-in-unreal-engine)
- **사실**:
  - **MPC**: 스칼라·벡터 각 1024개까지, **머티리얼 1개당 최대 2개 컬렉션 참조**. BP `Set Scalar/Vector Parameter Value`로 갱신 — 문서 원문: **"much more efficient than setting many different parameters on various Material instances."**
  - **Material Function**: 그래프 일부를 재사용 애셋으로 패키징. **변경이 참조하는 모든 머티리얼에 자동 전파.**
  - **MID vs Instance Constant**: MID는 런타임 스크립트로 생성·수정, Instance Constant는 사전 계산. 정적 파라미터 조합 폭발은 **셰이더 퍼뮤테이션 폭증** 경고.
- **projectTP 접점**:
  - **F0⑥ `Freeze` 파라미터**를 8유닛 × 모션마다 MID로 개별 세팅하는 대신, **전역 성격 파라미터는 MPC** 쪽이 문서상 더 효율적이다(단 Freeze는 유닛별이라 MID가 맞다 — MPC는 "전역 아웃라인 토글" 같은 것에).
  - `M_Sprite_Lit` / `M_Sprite_Flipbook_Lit` / `M_Sprite_PartComposite`가 **공유하는 UV 애니메이션 로직**(`floor((Time−TimeOffset)×8) % FrameCount`)은 **Material Function으로 추출** 후보 — 한 곳만 고치면 전부 전파된다.
- **재검토 시점**: 스프라이트 머티리얼이 **3개를 넘어 로직 중복이 아플 때**, 또는 전역 룩 파라미터(상태이상 틴트 등)가 생길 때.

### 2-5. Enhanced Input (+ Common UI) — **게임패드·Steam Deck**
- **문서**: [Enhanced Input](https://dev.epicgames.com/documentation/en-us/unreal-engine/enhanced-input-in-unreal-engine) · [Common UI 플러그인](https://dev.epicgames.com/documentation/unreal-engine/common-ui-plugin-for-advanced-user-interfaces-in-unreal-engine) · [CommonUI Input 기술 가이드](https://dev.epicgames.com/documentation/en-us/unreal-engine/commonui-input-technical-guide-for-unreal-engine)
- **사실**:
  - Enhanced Input은 **UE5 기본 활성**. Input Action(bool/Axis1D/2D/3D), **Input Mapping Context(런타임 추가·제거·우선순위 변경)**, Modifier(데드존·감도·축 반전), Trigger(Explicit/Implicit/Blocker). **게임패드 완전 지원.**
  - **Common UI**: "**Cardinal navigation management for gamepads**"(게임패드 방향 포커스 내비)와 Input Routing(다층 UI 포커스 관리)을 명시적으로 제공. 게임패드는 **보이지 않는 synthetic cursor**(`FCommonAnalogCursor`)로 마우스 클릭을 흉내내는 구조 — 문서 원문: "**if you set up your UI to work with a mouse, all you need for CommonUI to behave correctly is for an invisible cursor to be in the right location and click like a mouse**."
  - **★도입 비용 주의**: Common UI는 프로젝트 뷰포트 클래스를 **`UCommonGameViewportClient`로 교체**해야 한다 — 위젯 하나 바꾸는 게 아니라 **프로젝트 레벨 변경**이다.
  - **5.8 변경**: 릴리즈노트가 **"Enhanced Input과 Common Input/UI의 통합"** + **Input Debugger** 신설을 명시.
- **projectTP 접점**: [[UI_화면_규약]] §J-2(게임패드 알파 포함 여부, 오너 미결정)의 실행 수단. **Mapping Context 런타임 교체**는 우리 루트 스왑 구조(Draft↔BattleHUD↔Result)와 잘 맞는다. **"마우스로 만들어두면 CommonUI가 커서를 흉내낸다"** 는 문서 서술은 **지금 마우스 전용으로 만들어도 나중이 크게 손해가 아니라는** 뜻 — §J-2를 미루는 결정을 뒷받침한다.
- **재검토 시점**: 오너가 §J-2를 결정할 때. **⚠ 그때 반드시 §4의 문서 모순**(CommonUI+Enhanced Input 페이지는 "출하 권장 안 함"(5.2 시절 문구), 5.8 릴노트는 "통합 완료")**을 먼저 해소**할 것.

### 2-6. 네트워킹 코어 — 리플리케이션·RPC·**리슨서버 공정성 경고**
- **문서**: [Networking Overview](https://dev.epicgames.com/documentation/unreal-engine/networking-overview-for-unreal-engine) · [RPC](https://dev.epicgames.com/documentation/en-us/unreal-engine/remote-procedure-calls-in-unreal-engine) · [Actor Role/Remote Role](https://dev.epicgames.com/documentation/unreal-engine/actor-role-and-remote-role-in-unreal-engine) · [Actor Relevancy](https://dev.epicgames.com/documentation/unreal-engine/actor-relevancy-in-unreal-engine) · [Dedicated Server 셋업](https://dev.epicgames.com/documentation/unreal-engine/setting-up-dedicated-servers-in-unreal-engine)
- **사실**:
  - RPC 3종: **Server**(클라가 호출→서버 실행, **호출자가 액터를 소유해야 함**), **Client**(서버→소유 클라), **NetMulticast**(서버→서버+관련 클라 전원). 기본 **Unreliable**. **Server RPC에는 `_Validate` 함수**를 붙일 수 있고 **검증 실패 시 호출자를 자동 접속 해제**한다.
  - `ENetRole`: `ROLE_Authority`(서버만), `ROLE_AutonomousProxy`(클라 조종, RPC 호출 가능), `ROLE_SimulatedProxy`(상태 변경·RPC 불가).
  - **대부분의 액터는 기본적으로 리플리케이트되지 않는다**(수동 지정). `NetDormancy`(유휴 시 갱신 중단), `NetUpdateFrequency`, `bAlwaysRelevant`.
  - **★리슨서버 vs 전용서버(공식 서술)**: 리슨서버는 **호스트가 진짜 게임 상태 위에서 직접 플레이하므로 이점을 가진다**. 전용서버는 **모든 클라를 대칭 취급하고, 치트에 더 강하며, 확장성이 좋다**.
- **★projectTP 접점 (정직한 경고)**: 우리는 [[로드맵_버전계획]]에서 **리슨서버 P2P를 서버비 0을 이유로 확정**했다. 그런데 **경쟁 PvP + 리슨서버 = 호스트가 곧 서버 권위자**다 → [[데이터_서버_규약]] §3-B의 **"서버 권위로 치트 방지"** 논리가 **호스트 본인에게는 성립하지 않는다.** 공식 문서 2곳이 이 조합을 명시적으로 경계한다.
  - **좋은 소식**: `_Validate`(서버 측 쿨다운·코스트 재검증), `ROLE_Authority` 게이팅은 **상대 클라의 치트는 확실히 막는다.** 문제는 오직 **호스트 본인**이다.
  - **결정 필요(오너)**: 랭킹 없는 캐주얼 매치면 수용 가능. **랭킹/보상이 걸리면 전용서버가 불가피**해질 수 있다.
- **재검토 시점**: **Steam PvP 착수 직전(베타 네트워크 스파이크).** 그때 "호스트 치트 허용 범위"를 오너가 명시적으로 결정해야 한다.

### 2-7. Online Subsystem Steam / EOS — 매칭·세션·클라우드 저장
- **문서**: [OSS Steam](https://dev.epicgames.com/documentation/en-us/unreal-engine/online-subsystem-steam-interface-in-unreal-engine) · [OSS EOS 플러그인](https://dev.epicgames.com/documentation/en-us/unreal-engine/online-subsystem-eos-plugin-in-unreal-engine) · [Online Services EOS 설정](https://dev.epicgames.com/documentation/en-us/unreal-engine/enable-and-configure-online-services-eos-in-unreal-engine) · [Session Interface](https://dev.epicgames.com/documentation/unreal-engine/online-subsystem-session-interface-in-unreal-engine) · [OSS vs Online Services](https://dev.epicgames.com/documentation/unreal-engine/online-subsystems-and-services-in-unreal-engine)
- **사실**:
  - **★상위 갈림길**: 구세대 **Online Subsystem(OSS)** 과 신세대 **Online Services**가 **병존**하며, 문서는 **Online Services가 결국 OSS를 대체할 방향**이라고 말한다. Steam·EOS 각각 **양쪽 트랙에 별도 구현**이 있다 → **어느 트랙을 탈지가 모든 하위 결정(세션 API·매칭 API·저장 API)을 좌우한다.**
  - **Steam**: Steamworks SDK 설치 + `[OnlineSubsystemSteam]` INI + **App ID**(개발용 `480`) 필요. 기능: **Lobbies·GameServer 매치메이킹**, 리더보드, 도전과제, Voice, **UserCloud/SharedCloud**, External UI. **Lobbies(P2P·`bUseLobbiesIfAvailable`)가 리슨서버 호스팅에 쓰이는 경로**. `SteamNetDriver`가 기본(끄면 표준 `IPNetDriver`). Shipping 빌드에선 **Steam DRM 체크**(구독 확인 실패 시 종료).
  - **EOS**: Epic Developer Portal 등록(Client/Product/Sandbox/Deployment ID) → INI. **Steam+EOS 크로스플레이는 별도 "EOS Plus" 플러그인 — Beta, 기능 커버리지 좁음.** EOS P2P는 `SocketSubsystemEOS.NetDriverEOSBase`. **Player/Title Data Storage용 Client Encryption Key 설정 존재.**
  - **Session 흐름**: `CreateSession` → `FindSessions` → `StartMatchmaking` → `JoinSession` + `GetResolvedConnectString` → travel. `FOnlineSessionSettings`에 **"player-hosted vs dedicated"** 구분이 있다(리슨서버 = player-hosted).
  - **5.8 변경 없음**(릴리즈노트에 OSS/EOS/Steam 섹션 자체가 없음).
- **projectTP 접점**: [[로드맵_버전계획]]이 확정한 **"Steam ID + EOS 매칭 + 리슨서버 P2P + Player Data Storage"** 의 모든 조각이 문서상 실재함을 확인. **다만 그 조합(Steam+EOS 크로스)이 요구하는 "EOS Plus"가 Beta**라는 게 새로 드러난 리스크.
- **재검토 시점**: **Steam PvP 착수 시(베타).** 그때 **첫 결정 = OSS 트랙 vs Online Services 트랙**(되돌리기 비쌈).

### 2-8. 퍼포먼스 — **`stat` 명령 먼저, Unreal Insights는 그 다음**
- **문서**: [Stat Commands](https://dev.epicgames.com/documentation/en-us/unreal-engine/stat-commands-in-unreal-engine) · [Unreal Insights](https://dev.epicgames.com/documentation/en-us/unreal-engine/unreal-insights-in-unreal-engine) · [프로파일링 입문](https://dev.epicgames.com/documentation/en-us/unreal-engine/introduction-to-performance-profiling-and-configuration-in-unreal-engine)
- **사실**:
  - **1단계(무설정)**: PIE 콘솔에 **`stat unit`**(Frame/Game/Draw/GPU/RHIT), **`stat game`**(게임플레이 틱 비용), `stat fps`, `stat gpu`, `stat memory`. **`stat startfile`/`stopfile`** 로 `.uestats` 캡처 → Session Frontend Profiler에서 분석.
  - **2단계(깊이)**: Unreal Insights — **Timing(CPU/GPU) · Memory · Networking · Slate(UMG) · Asset Loading · Cooking** Insights. Trace Server는 5.3부터 백그라운드 자동 실행(포트 1981).
  - 공식 권고: **온스크린 FPS/프레임타임부터 보고**, 필요할 때만 깊이 들어가라. **실제 타겟 하드웨어에서** 측정하라.
  - **⚠ Shipping 빌드는 `stat`·프로파일링 도구를 제거**한다(Development 빌드에서 측정할 것).
  - **5.8**: Audio Insights가 Production Ready(오디오 미착수라 무관), World Partition 스트리밍 디버깅 추가(오픈월드 아님 — 무관).
- **projectTP 접점**: **Slate/UMG Insights**가 우리에겐 핵심 후보 — 전투 HUD(8유닛 HP게이지 + 부동 숫자 + `Redraw Time` 설정)가 무거워지면 여기서 잡는다.
- **재검토 시점**: **지금은 병목이 없다.** PIE가 눈에 띄게 느려지거나 F9b 풀플레이에서 프레임 저하가 관측될 때 — **`stat unit` 한 줄부터.**

### 2-9. Sequencer + Camera Cut Track + Cine Camera — 카메라 연출의 공식 대응품
- **문서**: [Sequencer 개요](https://dev.epicgames.com/documentation/en-us/unreal-engine/unreal-engine-sequencer-movie-tool-overview) · [Camera Cut Track](https://dev.epicgames.com/documentation/unreal-engine/cinematic-camera-cut-track-in-unreal-engine) · [Cinematic Cameras](https://dev.epicgames.com/documentation/unreal-engine/cinematic-cameras-in-unreal-engine) · [Shots/Takes](https://dev.epicgames.com/documentation/unreal-engine/sequences-shots-and-takes-in-unreal-engine)
- **사실**: Level Sequence = **애셋(트랙·키프레임) + 액터(레벨 바인딩·재생)**. 재생 중 **플레이어 입력·카메라·HUD 가시성 제어** 가능. **Camera Cut Track**이 활성 카메라를 결정하며 **"Can Blend" 토글로 카메라 간 크로스블렌드**, **시퀀스 카메라 → 게임플레이 카메라 블렌드도 지원**. **Cine Camera**: Filmback/Lens(F-stop·블레이드), Focus(수동/**액터 트래킹 오토포커스**/끄기 + 보간·디버그 시각화), **Look-at Tracking**(지정 액터를 향해 회전 오버라이드 + 보간속도·오프셋). Shot/Sub-Sequence/Take 계층으로 비파괴 편집.
- **projectTP 접점**: 우리 OTS 공격 컷은 **BP 커스텀**(카메라 블렌드 + 컷 순간 스프라이트 yaw 스냅)으로 **이미 동작한다**([[카메라연출_원칙]]). Sequencer의 **Can Blend**·**Look-at Tracking**·**Focus 보간**이 우리가 손으로 만든 것의 공식 대응품이다.
- **재검토 시점**: [[카메라연출_원칙]] 백로그의 **"연속 카메라 워크 도입"**(그때는 빌보딩을 틱/머티리얼로 승격해야 함) 또는 **스킬별 컷이 여러 종으로 늘어 BP 분기가 감당 안 될 때**. **지금 갈아엎을 이유는 없다.**

### 2-10. Gameplay Ability System (GAS) — **A2 스킬 재편 시 마지막 싼 결정 시점**
- **문서**: https://dev.epicgames.com/documentation/en-us/unreal-engine/gameplay-ability-system-for-unreal-engine
- **사실**: **어빌리티(액티브/패시브) + 어트리뷰트(Attribute Set) + Gameplay Effect(어트리뷰트 수정) + Ability Task(비동기)** 프레임워크. Ability System Component가 인터페이스 구현. 대상 장르로 **RPG·액션·MOBA** 명시.

## 3. 무관 / 스킵 (11건) — **유명하지만 우리와 안 맞는 것**

### 3-1. ❌ Lumen (동적 GI·리플렉션) — **정적 라이팅과 비호환**
- **문서**: [Lumen GI/Reflections](https://dev.epicgames.com/documentation/en-us/unreal-engine/lumen-global-illumination-and-reflections-in-unreal-engine) · [Lumen 기술 상세](https://dev.epicgames.com/documentation/en-us/unreal-engine/lumen-technical-details-in-unreal-engine) · [Lumen 퍼포먼스 가이드](https://dev.epicgames.com/documentation/en-us/unreal-engine/lumen-performance-guide-for-unreal-engine)
- **스킵 사유(전부 문서 근거)**:
  - **"Static Lights are unsupported"** — Lumen을 켜면 **스태틱 라이트 기여가 꺼지고, 프리컴퓨티드 라이팅·라이트맵이 비활성화**된다. **"incompatible with static lighting and forward shading."** 우리는 **고정 카메라의 정적 디오라마**다 — 베이크 친화 구조를 버려야 한다.
  - **HW 레이트레이싱은 NVIDIA RTX 2000 시리즈+ / AMD RX 6000 시리즈+** 를 요구한다 → **Steam 최소사양 바닥을 올린다.**
  - 라이트 누수 방지를 위해 **벽 두께 10cm 이상의 모듈러 지오메트리**를 요구하고, **고폴리 메시엔 Nanite가 필요**하다 — Village Pack 배경에 그런 보장이 없다.
  - **전역 라이팅 변화가 수 초에 걸쳐 전파**(느린 수렴) — 컷 전환이 잦은 우리에겐 오히려 리스크.
  - 퍼포먼스 가이드의 비용 축(오픈월드, 콘솔 60fps 예산, 인스턴스 10만개/TLAS 재빌드)이 **8유닛 고정 디오라마와 전혀 다른 프로파일**이다.
- **5.8의 Lumen Lite**(Medium 티어 = Irradiance Field 프로브 GI)도 같은 이유로 스킵 — **해결할 문제 자체가 없다.**

### 3-2. ❌ MegaLights (5.8 Production Ready)
- **문서**: https://dev.epicgames.com/documentation/en-us/unreal-engine/megalights-in-unreal-engine
- **스킵 사유**: **다수의 동적·그림자 지는 로컬/영역 광원**을 확률적 샘플링으로 값싸게 처리하는 기능이며 **하드웨어 레이트레이싱을 요구**한다. 우리 씬의 광원은 **2개**(KeySun·FillSky). 해결할 문제가 없다. (스킬 VFX 포인트라이트를 대량으로 깔게 되면 그때 재검토)

### 3-3. ❌ Nanite
- **스킵 사유**: 우리 지오메트리는 **스프라이트 쿼드(삼각형 2개)** 와 저폴리 배경 팩이다. 가상화 지오메트리가 절감할 삼각형이 **애초에 없다.** (Lumen 문서가 "고폴리 메시엔 Nanite 필요"라고 말하는데, 우리에겐 **고폴리 메시가 없다** — 이중으로 무관.)

### 3-4. ❌ Mass Entity Framework
- **문서**: https://dev.epicgames.com/documentation/en-us/unreal-engine/overview-of-mass-entity-in-unreal-engine
- **스킵 사유**: Fragment/Archetype/Chunk 기반 **데이터 지향 대량 엔티티 배치 처리** 프레임워크. 문서 전체가 대량 엔티티의 메모리·배치 최적화를 겨냥한다. 우리는 **유닛 8기**다. 아키텍처 복잡도만 사고 이득이 0인 전형적 오버킬.

### 3-5. ❌ Iris 리플리케이션 시스템 (5.8 간판 네트워킹 기능)
- **문서**: [Iris 소개](https://dev.epicgames.com/documentation/unreal-engine/introduction-to-iris-in-unreal-engine) · [릴리즈노트](https://dev.epicgames.com/documentation/unreal-engine/unreal-engine-5-8-release-notes)
- **스킵 사유(문서 근거)**: Iris의 목적은 **"더 크고 상호작용적인 월드 · 더 많은 플레이어 수 · 더 낮은 서버 비용"**(포트나이트 BR 경험 기반)이다. 우리는 **2인 매치 · 8유닛 · 턴제 이벤트성**이다. 레거시 리플리케이션으로 차고 넘친다.
- **★상태 표기 모순(§4)**: 릴리즈노트는 "**production-ready for licensees** in UE 5.8", Iris 문서 페이지는 여전히 **"Experimental"**. 어느 쪽이든 우리 결론(불필요)은 안 바뀐다.

### 3-6. ❌ Network Prediction 플러그인
- **문서**: https://dev.epicgames.com/documentation/unreal-engine/API/PluginIndex/NetworkPrediction
- **스킵 사유**: **Beta — "출하 시 주의"** 명시. 대상 문제는 **연속적인 매 프레임 이동·물리 시뮬레이션의 롤백/재시뮬레이션**이다(Mover가 이걸 쓴다). 우리는 **턴제·이벤트성 스킬 판정**이다 — **문제의 형태가 다르다.** 우리 패턴(클라 연출 + Server RPC 요청 + 서버 정수 판정 + NetMulticast 브로드캐스트)은 **2-6의 평범한 RPC 문서**에 그대로 대응된다.
- **⚠ 부수 발견**: **"결정론이면 롤백이 불필요하다"** 를 뒷받침하는 **공식 문서를 찾지 못했다**(§4). 우리 가정이 틀렸다는 뜻은 아니지만, **공식 근거는 없다**는 걸 알고 가야 한다.

### 3-7. ❌ Paper2D 컴포넌트 라인 (Sprite / Flipbook / TileMap / Sprite Editor)
- **문서**: [Paper2D 개요](https://dev.epicgames.com/documentation/unreal-engine/paper-2d-overview-in-unreal-engine) · [Flipbooks](https://dev.epicgames.com/documentation/unreal-engine/paper-2d-flipbooks-in-unreal-engine) · [TileSets/TileMaps](https://dev.epicgames.com/documentation/unreal-engine/paper-2d-tile-sets-and-tile-maps-in-unreal-engine)
- **스킵 사유**: 우리는 **의도적으로** `UPaperSprite`/`UPaperFlipbook`을 버리고 **스태틱메시 쿼드 + 머티리얼 UV 애니메이션**을 택했다 — (a) DataTable 구동(RowIndex/FrameCount)으로 8캐릭터 × 다수 모션을 **머티리얼 파라미터만으로** 갈아끼우고 (b) 3D Lit 통합·캐스트섀도우를 완전 제어하기 위함. Flipbook은 **프레임별 가변 duration**이라는 장점이 있으나 `UPaperFlipbookComponent` 종속이라 우리 데이터 구동 구조와 충돌한다.
- Sprite Editor의 collision/socket/pivot 툴과 "Apply Paper 2D Import Settings" 프리셋은 **UPaperSprite 애셋 전용**이라 우리 스태틱메시 쿼드엔 **적용 자체가 불가**하다(문서로 확인).

### 3-8. ❌ Sprite Atlas Group (드로우콜 절감 후보였음)
- **문서**: https://dev.epicgames.com/documentation/unreal-engine/paper-2d-settings-in-the-unreal-engine-project-settings
- **스킵 사유(문서 원문)**: Project Settings 체크박스 설명이 **"Enables experimental (incomplete and unsupported) texture atlas groups"** 다. **5.8 시점에도 공식적으로 "미완성·미지원".** 8캐릭터 텍스처 아틀라스화로 드로우콜을 줄일 물건이 아니다. (애초에 우리는 Paper2D 스프라이트 애셋을 안 쓴다 — 이중으로 무관)

### 3-9. ❌ String Table — **우리 CSV DataTable이 구조적으로 우월하다**
- **문서**: https://dev.epicgames.com/documentation/unreal-engine/using-string-tables-for-text-in-unreal-engine
- **스킵 사유(문서 근거 — 우리 결정을 강화한다)**:
  - 문서 원문 경고: **"String Table assets are binary, so they cannot be resolved if a merge conflict happens"** — **바이너리라 머지 충돌 해결 불가.**
  - CSV 기반 String Table은 **"aren't staged automatically"** — 패키징 시 **수동으로 Additional Non-Asset Directories에 추가**해야 한다.
  - 에디터 String Table 에디터는 **메타데이터를 편집할 수 없다.**
  - **★우리가 겪은 `set_entry` dirty 무음 소실 버그는 5.8 문서에 언급이 없다**(알려진 이슈로도, 수정 사항으로도) — §4.
- **projectTP 접점**: [[데이터_서버_규약]] §1-A의 배제 결정이 **문서가 명시한 두 약점(바이너리 머지 충돌·수동 스테이징)까지 구조적으로 회피**하고 있었다. **현행 유지.**

### 3-10. ❌ Verse — **UEFN/Fortnite 전용임을 문서로 확정**
- **스킵 사유(검증된 부정 결과)**: **UE 5.8 릴리즈노트 전문에 "Verse"가 단 한 번도 등장하지 않는다.** 공식 Verse 문서는 **전부 `/documentation/fortnite/` 또는 `/documentation/en-us/uefn/` 네임스페이스**에만 존재하고 `/documentation/unreal-engine/`에는 없다. 정의 문장도 "Create your own rules and behaviors for your game by programming with Verse **in UEFN**!"으로 **Fortnite Creative 섬**에 한정된다. → **일반 UE 프로젝트(우리처럼 BP/C++)에 쓸 수 있다는 공식 근거 없음.** (외부에서 "언젠가 UE로 온다"는 추측이 돌지만 **공식 문서엔 없다.**)

### 3-11. ❌ 5.8 모바일 렌더러 개선 · Steam Frame(XR) 지원
- **스킵 사유**: 5.8 Platform 섹션은 **전부 모바일/Android/iOS/XR**이다. 우리 타겟은 **PC/Steam**([[기획_방향성]] 방향성2 확정). 모바일 수집형(방향성1)은 백업 보관됨.
- **⚠ 이름 주의**: 5.8의 **"Steam Frame Support (Experimental)"** 는 **Valve의 차세대 XR 헤드셋 하드웨어** 지원이지 **Steam 스토어/매치메이킹 배포와 무관**하다 — 이름 때문에 오해하기 쉽다.

## 4. 미확인 / 문서 모순 (8건) — **정직 고지**

> **없는 걸 지어내지 않았다.** 아래는 **확인하지 못했거나, 공식 문서끼리 서로 모순되는** 것이다. **사실인 것처럼 인용하지 마라.**

| # | 항목 | 상태 |
|---|---|---|
| 1 | **Iris 상태**: 릴리즈노트 "production-ready for licensees" vs Iris 문서 페이지 "Experimental" | **문서 간 모순** — 해결 못 함. (우리 결론엔 영향 없음) |
| 2 | **CommonUI + Enhanced Input 출하 가능성**: 통합 가이드 페이지는 **"We do not recommend attempting to ship titles with this feature at this time"**(5.2 시절 문구인데 5.8 URL로 서빙 중) vs 5.8 릴리즈노트 **"unification of Enhanced Input and Common Input/UI"** | **문서 간 모순** — 게임패드 착수 전(2-5) **반드시 실물로 확인**할 것 |
| 3 | **Data Validation의 BP 검증기가 커맨드릿에서 도는가**: 같은 페이지가 "BP 검증기 자동 등록"과 **"only runs C++ validation rules by default"** 를 동시에 말한다 | **문서 자체 모순** — 2-2 착수 시 **1회 실증 필수** |
| 4 | **GAS의 BP-only 가능 여부 / 리플리케이션 지원** | **문서에 명시 없음** — 2-10 재검토 시 최우선 확인 |
| 5 | **"결정론적이면 롤백/보정이 불필요하다"** 는 우리 가정 | **공식 근거 없음** — 이를 뒷받침하는 UE 문서를 찾지 못했다. 가장 가까운 공식 자료(Network Prediction)는 **반대 가정**(롤백 필요) 위에 서 있다 |
| 6 | **StringTable `set_entry` dirty 무음 소실 버그의 5.8 수정 여부** | **문서에 언급 없음**(알려진 이슈로도, 수정으로도) — 우리 회피책(CSV DataTable) 유지 |
| 7 | **EOS Player Data Storage의 BP 레벨 읽기/쓰기 방법** | **UE 문서엔 없음.** 플러그인 설정 페이지의 **Client Encryption Key** 언급만 확인. 상세는 별도 EOS SDK 문서 사이트에 있으나 **fetch 실패** — Steam PvP 착수 시 재시도 |
| 8 | **픽셀아트 스프라이트시트 권장 Compression Setting** | **문서에 명시 없음.** Filter=Nearest·MipGen=NoMipmaps는 문서로 확인됐으나, 무손실 플랫컬러용 권장 압축 포맷(우리는 `TC_EditorIcon` 사용 중)을 명시한 페이지를 못 찾았다 |

> **조사 범위 밖(의도적)**: 오디오(WASAPI 전환), PCG, Control Rig, MetaHuman, Motion Design, Media, Worldbuilding/World Partition — 우리 스택과 접점이 없어 파지 않았다.

## 5. 이 카탈로그를 쓰는 법

1. **평소엔 안 본다.** 이건 병목이 생겼을 때 여는 서랍이다.
2. 병목이 생기면 **§2에서 그 병목의 트리거를 찾는다**(예: "HUD가 느리다" → 2-8 `stat unit`부터).
3. **§3(무관)은 "왜 안 쓰기로 했는지"를 다시 설명하지 않기 위한 기록**이다. 누군가 "Lumen 켜면 예뻐지지 않나요?"라고 물으면 3-1을 가리키면 된다.
4. **§4(미확인)를 사실처럼 인용하지 마라.** 필요해지면 그때 문서를 열어 확인하고 **이 표를 갱신**한다.
5. **§1은 지금 당장 읽을 가치가 있다** — 특히 **1-2(`Import Key Field`·캐싱 금지)** 와 **1-3(동적 Format Text 인자 핀)** 은 A1에서 실제로 밟을 수 있는 지뢰다.
