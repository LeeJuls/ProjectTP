### 1-A. 스트링 다국어 구조 (오너 확정 2026-07-08 — 알파에 뼈대, 콘텐츠는 ko만)
- **스키마**: `Resource/data/strings.csv` — `Key,ko,ja,en` (컬럼 추가 = 언어 추가. 컬럼명은 ISO 639-1 코드 권장 — UE 문화권 코드와 1:1 대응, 오너 표기 "jp"→"ja")
- **구현**: UE StringTable이 아니라 **DataTable**(행 구조체에 언어 컬럼) 채택. 이유 ①컬럼 확장 자유 ②StringTable dirty·런타임 SetText 버그 이력 회피 ③MCP·BP에서 완전 제어
- **런타임 전환**: GameInstance `CurrentLanguage`(SaveGame 영속) → `GetLocalizedString(Key)` 헬퍼가 현재 언어 컬럼 반환(빈칸 폴백=ko) → 언어 변경 시 `OnLanguageChanged` 브로드캐스트 → 열린 위젯 일괄 재바인딩
- **UI**: 옵션 팝업 `WBP_Options`(대기실에서 열림)에 언어 선택 버튼 — 화면 목록에 오버레이로 편입([[UI_화면_규약]] §A-2)
- **폰트 메모**: 언어별 글리프 커버(ja 등)는 폰트 선정에 영향 — UMG 폰트 확정(A0②) 시 확장 가능성 고려, 실제 ja/en 폰트 검증은 콘텐츠 추가 시(베타 후보)

### 1-B. 값 치환 — 포지셔널 플레이스홀더 규칙 (오너 요구 2026-07-14, system-ui 설계)

> **문제의식(오너)**: "값마다 스트링을 다 만들면 스트링이 폭증한다." 공격력 버프가 50%·60%… 여러 종이면 `"공격력 50% 증가"`·`"공격력 60% 증가"`를 각각 키로 만들지 말고 **템플릿 1개 + 런타임 값 주입**으로 접는다. 이 절이 `strings.csv` 값 안의 **치환 문법·값 소스 매핑·표시 변환**의 SSOT다. [[UI_화면_규약]] §G는 이 절을 가리키는 참조만 둔다(중복 서술 금지).

#### ① 문법 — `{0}` 0-index 포지셔널 (UE `Format Text` 네이티브)

- **문법 확정**: `strings.csv`의 `ko`/`ja`/`en` 값 안에 `{0}`,`{1}`,`{2}`… **0-index 포지셔널 토큰**을 그대로 삽입. 런타임에 값 배열을 순서대로 꽂는다(`{0}`←값[0], `{1}`←값[1], …).
- **엔진 지원**: 언리얼 BP `Format Text` 노드가 이 패턴을 네이티브 지원 — 포맷 문자열의 `{0}`,`{1}` 토큰마다 입력 핀이 자동 생성됨(커스텀 파싱 로직 불필요). **실제 BP 배선은 F7(스킬메뉴) 구현 시 gameplay/umg-engineer 몫**, 이 절은 규칙·매핑만 정한다.
- **선례(이미 라이브)**: `strings.csv`의 `UI_BATTLE_COOLDOWN_FORMAT = "CD {0}"` · [[UI_화면_규약]] §G의 `UI_DRAFT_PICK_COUNT = "{0}/5"` — `{0}` 포지셔널은 이미 프로젝트에서 발현 중이며, 이 절은 그 관행의 **정식 규약화**다.
- **명명 토큰(`{atk}`)은 후순위**: 같은 노드가 `{atk}` 명명 토큰도 네이티브 지원(자기설명적이라 가독성 우위)하나, 오너 확정에 따라 **표준은 숫자 0-index로 고정**한다. 가독성은 §②의 `#desc_args` 주석 범례가 보완.

**예시** (주입값은 §③ 표시 변환 적용 후):

| Key(예시) | 템플릿(ko) | 값 배열 | 렌더 결과 |
|---|---|---|---|
| (오너 예시)`str_hero_passive_skill_001` | `공격력이 {0}% 증가하고, 방어력이 {1}% 증가합니다` | `[50, 60]` | 공격력이 50% 증가하고, 방어력이 60% 증가합니다 |
| 단일치환(라이브 선례) | `CD {0}` | `[1]` | CD 1 |
| 2치환(베기 전환 후보) | `대상에게 공격력의 {0}% 물리 피해. 쿨다운 {1}턴.` | `[130, 1]` | 대상에게 공격력의 130% 물리 피해. 쿨다운 1턴. |
| 4치환(파이어볼+약화 후보) | `공격력의 {0}% 마법 피해. 쿨다운 {1}턴. {2}% 확률로 공격력 {3}% 약화.` | `[170, 2, 35, 25]` | 공격력의 170% 마법 피해. 쿨다운 2턴. 35% 확률로 공격력 25% 약화. |

> **언어 무관 인덱스 결속(중요)**: `{0}`이 가리키는 **의미값은 ko/ja/en 전 언어에서 동일**하다(값[0]은 항상 같은 데이터). 언어마다 다른 것은 **문장 내 토큰 위치**뿐 — 어순이 다른 언어(en 등)는 같은 `{0}`,`{1}`을 제 문법에 맞게 재배치한다. 이것이 순차소비(`%s`)가 아닌 **포지셔널**을 쓰는 이유다(ICU MessageFormat과 동일 원리 — 로컬라이즈 어순 역전 대응).

#### ② 값 소스 매핑 — **자유형 채택** (핵심 결정, system-ui 판정)

**결론: 자유형(free-form). `skills.csv`에 값 바인딩 컬럼(`DescValues`류)을 신설하지 않는다.** 인덱스→값 매핑은 **스킬별로 DescKey 작성자가 정하고 F7 BP 배선이 수동 결선**한다(전역 고정 컬럼순서 강제 안 함).

| 안 | 방식 | 판정 |
|---|---|---|
| **(a) 자유형** | 매핑은 스킬별 BP 배선이 결정, 전역 컬럼순서 없음 | **채택** |
| (b) 컬럼 신설 | `skills.csv`에 `DescValues="130;1"`(값) 또는 `"PowerRate;CooldownTurns"`(필드참조) 배열 컬럼 | 기각 |

**자유형 채택 근거:**
1. **스킬 다양성(데이터로 확증)**: 알파 5스킬조차 표시 값 집합이 제각각 — 공격=PowerRate만 / 막기=EffectValue+CD / 파볼=PowerRate+CD(향후 +Chance+AtkDown). 스킬 유형마다 PowerRate·EffectValue·EffectChance·Cooldown/Duration으로 갈려 **전역 고정 컬럼순서가 부자연스럽다**.
2. **변환이 어차피 BP 경계에 산다(§③)**: 각 값은 채널별 표시 변환(율→×100, 턴→raw)이 달라 BP는 값의 **의미 채널을 이미 알아야** 변환을 고른다. 그 필드를 `DescValues`에 또 적는 것은 **중복**일 뿐 BP측 필드 지식을 없애지 못한다(안 b는 컬럼만 늘고 로직은 그대로 — 최악).
3. **이중 진실 회피**: 값을 그대로 적는 `DescValues="130;1"`은 `PowerRate`/`EffectValue` 컬럼과 **출처 이원화**→드리프트(§4 "문자열 원문 저장 금지"·"손으로 쓰면 거짓말" 위반). 필드참조형은 미니 DSL(문자열 필드명 리플렉션)을 도입하면서도 변환 로직은 못 없앤다.
4. **데이터 스키마 무변경**: 알파 5스킬은 이 패턴 불필요(순수 규칙 신설). 자유형은 신규 컬럼 0.

**자유형 약점(암묵 매핑) 보완 — `#desc_args` 주석 범례(선택, 안 b 아님)**:
- 자유형의 유일한 약점 = 인덱스→필드 매핑이 BP에만 있어 CSV만 봐선 `{0}`의 뜻을 모름.
- 보완: `skills.csv`에 `#desc_args` **주석 컬럼**(헤더 `#` 접두 → 임포터 무시, §4 `#id_txt` 선례와 동일 패턴)에 사람용 범례 기재. 예: `#desc_args = "{0}=PowerRate×100 {1}=CooldownTurns"`.
- **이것은 안 (b)가 아니다**: 게임이 읽지 않는 순수 주석(데이터 바인딩 아님) — 값 소스는 여전히 자유형·BP 결선이고, `#desc_args`는 `#id_txt`처럼 가독성 보조일 뿐(진실 아님, 검증 스크립트가 대조 가능).
- **적용 시점**: 치환형 DescKey가 실제 필요한 스킬이 생길 때(F7/A2). **알파 5스킬엔 지금 신설하지 않는다.**

#### ③ 값 표시 변환 — **데이터는 순수, 변환은 BP 표시 시점**

**원칙: 원본 데이터는 순수 값으로 두고, 표시 포맷(율→퍼센트 등)은 UI 경계(BP `Format Text` 직전)에서 계산한다.** 데이터에 표시값을 미리 굽지 않는다(관심사 분리).

**근거(2줄)**: ① 서버 권위 판정이 순수 값으로 계산한다(§3-B `max(1,floor(Atk×PR−Def))`) — 미리 구운 `"130%"` 문자열로는 연산 불가. ② 한 순수 값이 여러 표현(퍼센트·게이지·툴팁)에 재사용된다 — 표시형을 데이터에 박으면 값 변경 시 숫자와 문자열을 이중 수정해야 해 드리프트(§4 위반).

**변환은 2부분으로 분해**:
- **숫자(플레이스홀더에 꽂히는 값)**: BP가 채널별 변환 후 **순수 숫자만** 주입.
- **단위·방향어(`%`·`턴`·`증가`·`감소`)**: **템플릿 텍스트**에 포함(언어별 로컬라이즈). 플레이스홀더엔 숫자만 들어간다.

**채널별 숫자 변환표** (F7 배선 참조 — 계수 값이 아닌 표시 *형태*라 **system-ui 소관, balance 무관**):

| 채널(필드) | 원본 예 | 변환 | 주입값 | 템플릿 예 |
|---|---|---|---|---|
| 배율(PowerRate) | 1.3 | `×100` | 130 | `{0}%` |
| 율=감소/증가(EffectValue as rate) | 0.25 | `×100` | 25 | `{0}% 감소` |
| 확률(EffectChance) | 0.35 | `×100` | 35 | `{0}% 확률` |
| 턴수(Cooldown/Duration Turns) | 1 | raw(변환 없음) | 1 | `{0}턴` |

> **★`1−r` 하지 마라(오너 명시)**: ATK_DOWN `EffectValue=0.25`는 **이미 감소율 r**이다. "공격력 25% 감소"를 표시하려면 `0.25×100=25`만 하면 된다 — **`1−r`(=0.75) 계산 불필요**. 방향(감소)은 템플릿의 "감소" 단어가 담고, 플레이스홀더엔 감소 *크기* r×100만 넣는다. `1−r=0.75`(최종 배율)는 **전투 연산(step4, [[상태이상_확정]] §6)** 값이지 표시값이 아니다 — 절대 혼용 금지.
> **반올림**: 율×100 후 소수 처리(정수/소수 자릿수)는 UI 포맷 선택(기본=정수, 알파 값은 전부 0.05 배수라 정수로 떨어짐). F7 세부.

#### ④ 엣지케이스 (방어적 설계 — F7 반영, qa-critic 후속 검증)

| # | 케이스 | 처리 |
|---|---|---|
| P1 | 플레이스홀더 개수 ≠ 주입 값 개수 | BP는 템플릿 `max_index+1`개를 정확히 주입. 검증룰(§4 V-패턴 확장): `max_index+1 == 값개수`. 미달 토큰 → **빈 문자열 폴백(크래시 금지)**, 초과 값 → 무시 |
| P2 | 값 미제공(널) | 해당 채널 기본값 또는 빈칸 폴백, 크래시 금지 |
| P3 | 표시 텍스트에 리터럴 `{` 필요 | UE FText 이스케이프(백틱)로 처리 — F7 세부(알파 문자열엔 리터럴 중괄호 없음, 현행 무이슈) |
| P4 | 언어별 어순 역전 | 포지셔널이 네이티브 해결(§① 언어 무관 결속). 순차소비 금지 |

#### ⑤ 기존 확정 문자열의 치환형 전환 — **적용 후보(F7)만, 지금 변경 금지**

## 3. 스킬 로직·서버 권위·라이브옵스 배송 (오너 질문 2026-07-08, Director 판정)

### 3-A. 스킬 로직 아키텍처 — 데이터 드리븐 (Lua 미채택)

**결론: 알파는 스크립트 불필요. 스킬 효과는 "효과 원자 + 데이터 조합"(데이터 드리븐)으로 심는다.** 오너가 과거 Lua로 푼 두 문제를 우리는 다른 축으로 해결:

| Lua를 쓰던 이유 | 우리의 해법 | 시점 |
|---|---|---|
| ① 핫패치(앱스토어 심사 우회) | **서버 권위 판정**이 구조적으로 해결 — 판정(결과 수치)이 서버에서 확정되면 클라 재심사 없이 서버 배포로 핫패치. 수치는 CSV를 서버가 내려줌 | 베타(서버 도입) |
| ② 복잡한 버프/디버프 | **데이터 드리븐 효과 시스템** — skills.csv의 `EffectType/EffectValue/EffectDurationTurns`를 효과 원자 조합 구조로 확장. 기존 효과 조합이면 새 스킬=데이터만 | 알파부터 |

- **지금 심을 규율**: 스킬 효과를 코드에 하드코딩하지 말고 **효과 원자(enum+파라미터)를 데이터로 조합**한다. 이게 미래 유연성의 90%이며, 언어(BP/C++)와 직교 — C++로 짜도 효과가 데이터 드리븐이면 유연성은 데이터층에서 나온다.
- **스크립트 재검토 = 라이브, 서버가 못 덮는 부분이 남을 때만**. 후보: UnLua·sol2(UE Lua 바인딩, UE 네이티브 Lua 미지원)·UE-Angelscript 포크. 서버 권위+데이터 드리븐이 Lua 용도의 대부분을 선점하므로 실제 도입 가능성은 낮음.
- BP-only MVP 원칙 유지. 전투 코어 C++ 이관은 베타 초입 검토(기존 결정) — 그때도 효과 시스템은 데이터 드리븐 유지.

### 3-B. 서버 권위의 정확한 범위 (오너 우려 2026-07-08 — "서버가 전투 다 돌리면 비용·속도 터진다")
**서버 권위 ≠ 풀 서버 시뮬레이션.** 오너 우려는 "서버가 모든 스킬 동작을 실행"할 때만 성립 — 우리는 그렇게 안 한다.

| 층 | 담당 | 비용 |
|---|---|---|
| 스킬 **동작**(애니·VFX·카메라·걸어나가기) | **클라**(항상) | 서버 부하 0 |
| 스킬 **결과**(데미지·상태 확정) | 서버 = `max(1,floor(Atk×PR−Def))` 정수 곱셈 몇 번 | 사실상 0 |
| 결과 브로드캐스트 | 서버 → 클라들 | 경량 메시지 |

- **싸게 만드는 특성**: 턴제/ATB(초당 60틱 물리 아님, 이벤트성)·소규모 4v4(8유닛/매치)·결정론 공식(부동소수점 시뮬 없음) → 판정 서버 1대가 수천~수만 동시 매치 감당.
- **진짜 병목 = 계산이 아니라 동시 세션 관리·라우팅.** 클라로 계산 빼도 안 풀림 → 아키텍처(리슨서버 P2P↔전용서버, 매치메이킹, 지역분산)로 해결.
- **클라 예측 + 서버 확정**: 결정론이라 예측=확정 거의 일치 → 롤백 없음. 레이턴시(100~200ms)는 걸어나가기+컷 연출이 은폐(턴제라 관대).
- **실측은 베타 네트워크 스파이크**: "매치당 판정 비용"·"서버당 동시 세션 한계"를 숫자로 뽑아 규모 산정([[로드맵_버전계획]] 스파이크 항목).

### 3-C. 클라 표시 데이터 hotfix — 원격 데이터 배송 (오너 질문 2026-07-08)
클라가 표시하는 데이터(스킬 설명·value·밸런스 수치)를 앱스토어 재심사 없이 갱신하려면 **바이너리에 굽지 말고 런타임 원격 페치**. 순수 데이터라 **앱스토어 정책상 합법**(로직 원격 갱신만 회색지대 4.7 — Lua가 걸렸던 지점. 우리 value·텍스트는 로직 아님).

| 원격 갱신 | iOS 정책 |
|---|---|
| 데이터(설명·value·테이블·이미지) | ✅ 명시적 허용(라이브 게임 표준) |
| 실행 로직(BP·스크립트) | ⚠ 회색(4.7) → 효과 원자 데이터 조합으로 회피 |

- **배송 흐름**: 클라 시작 → `_tables.csv`의 `Version` 확인 → 로컬<원격이면 CDN에서 CSV 다운로드 → 런타임 DataTable 채움 → **원격 실패 시 구운 첫 출시 스냅샷 폴백**(오프라인 대비).
- **로컬 내장 데이터 = 정식 요구사항(오너 확정 2026-07-08)**: 원격 없이도 게임이 돌아야 한다. 필요 이유 ①첫 실행(아직 못 받음) ②오프라인·CDN 장애 ③**알파는 서버 없어 로컬이 유일 소스** ④부팅 속도(네트워크 왕복 대기 없이 즉시 시작, 원격은 백그라운드 갱신).
- **3계층 우선순위**: **다운로드 캐시(최신) > 클라 내장(출시 스냅샷) > (서버=PvP 판정 authoritative, 표시와 별개)**. 부팅은 로컬로 즉시 시작 → 백그라운드로 원격 Version 확인 → 최신이면 캐시 저장(다음 부팅부터 적용) → 실패면 로컬 유지. **로컬 내장 = 우리 `Resource/data/` CSV가 클라 패키지에 구워진 것** — 별도 제작 아님, 단일 출처가 폴백 역할 겸함.
- **낡음 안전장치(베타 예약)**: 앱 장기 미업데이트 시 로컬 내장본이 크게 낡을 수 있음 → 원격 데이터에 "요구 최소 클라 버전" 표시로 업데이트 유도. 지금은 자리만 예약.
- **이미 절반 완료**: CSV 단일 출처(`Resource/data/`)+`Version` 컬럼이 토대. 남은 것 = 런타임 원격 페치 계층(UE HTTP 모듈 / Chunk Downloader) — **베타 추가**, 알파는 구워서 배포(구조만 CSV 분리 유지 = 로컬 내장 그 자체).
- **안전망(PvP)**: 클라 value = 표시용, 진짜 판정 = 서버 데이터 → 클라 변조·낡음도 결과는 서버 확정. 단 UX 어긋남 방지 위해 **같은 CSV를 서버·클라 양쪽 단일 배송**.

## 4. CSV 기획 데이터 규약 — top-down (오너 확정: 마스터→세부 분해 + 상위→하위 단방향 참조)

> **원 출처.** [[네이밍_폴더_규약]] §4는 이 절의 컬럼 규칙을 요약·교차참조만 한다. 실데이터 시연은 [[데이터규약_예시]].

- **테이블 3분류와 계층**:
  1. **마스터 인덱스** `_tables.csv` — 전체 테이블 목록(테이블명·버전·소유 시스템·설명). 새 테이블은 반드시 여기 등록(총괄시트).
  2. **정의(어휘) 테이블** — enum성 정의: `jobs.csv`·`races.csv`·`genders.csv`·(추후 `rarities.csv` 등). Id+NameKey 수준. 모든 테이블이 **값으로 사용 가능**(어휘 사용은 참조 계층 위반이 아님).
  3. **개체 테이블** — 조립 개념이 위, 부품이 아래: 레시피/캐릭터 규칙 → `parts.csv` → … **위가 아래의 Id를 보유**하고, 아래는 위를 모른다. 역참조·순환·형제 간 횡참조 금지.
- **동일 스키마명(명명 사전)** — 같은 의미 = 어느 테이블에서든 같은 컬럼명:
  - 0번 컬럼은 항상 `Id`(UE DataTable RowName) — **세그먼트 숫자 ID**(전역 유일, 오너 확정 2026-07-08). 예시 8자리: `[1 대분류][2 타입][1 성별][1 등급][3 일련]` → `10112001` = 캐릭터·직업01·남·2등급·001. 대분류: 1=캐릭터 2=파츠 3=스킬 4=아이템…(자릿수 배분은 축 확장 한계를 고려해 오너 확정).
    - **등급(GradeId) 세그먼트 = 유지 확정**(2026-07-13 오너 재확인 — 등급 유지+배율압축, [[알파_개발계획]] §4.5 balance#1/#2). 로직은 `GradeId` 컬럼만 읽고 ID 파싱 안 함(위 스마트 ID 안전장치 2규칙 준수).
    - **⚠ 비-등급 세그먼트 배치(타입/성별/포지션)는 미확정 — 별도 갱신 예약**: 위 예시의 `[2 타입][1 성별]`은 방향성1(수집형) 잔재다. 방향성2는 **포지션(탱/딜/힐) 축**이 밸런싱 기준([[알파_개발계획]] §4.5 balance#1, 지급이 포지션별 2개)이나 **포지션을 ID에 인코딩할지 vs 컬럼 파생할지는 열린질문#5(미확정)** — 확정 전까지 ID스킴을 방향성1식 직업/성별 구조로 굳히지 않는다(포지션 축 확정 시 이 세그먼트 배치 갱신).
  - **스마트 ID 안전장치 2규칙**: ①ID에 넣는 축은 **불변 취급**(기획 변경 시 신규 ID 발급+구 ID 폐기 — 가변값(성장 레벨 등)은 ID 인코딩 금지) ②**로직은 ID를 파싱하지 않는다** — 코드의 진실은 항상 컬럼, ID 세그먼트는 가독성·유일성·발급 규칙용(이중 진실 방지).
  - **런타임 인스턴스 ID는 별개**: 뽑기로 생성된 개별 캐릭터(SaveGame) 등 동적 개체는 스마트 ID가 아니라 단순 증가값/GUID 사용.
- **`#` 주석 규약(오너 확정 2026-07-08)**: `#`로 시작하는 **컬럼 헤더** = 주석 컬럼, `#`로 시작하는 **행(첫 셀)** = 주석 행 — 게임/임포터가 읽지 않음(임포트 전처리에서 제거, A0③ 파이프라인에 포함).
  - **`#id_txt`**: Id 바로 옆에 두는 사람용 번역 컬럼 — 숫자 ID의 중요 정보를 슬러그로: `10112001` → `char_knight_m_rare`, `24150100` → `part_cloth_s15_c1`.
  - `#id_txt`는 데이터 진실이 아님(진실은 컬럼) — 손으로 쓰면 시간이 지나며 거짓말하므로 **검증 스크립트가 컬럼에서 자동 생성/대조**(불일치 리포트)하는 것을 표준으로.
  - 외래키 = `<대상단수>Id`(`SkillId`), 다중 = `<대상단수>Ids`(구분자 `;`).
  - 문자열 원문 저장 금지 — 표시·설명은 `<대상>Key` 접미 컬럼(`NameKey`/`DescKey`)에 **dot-namespace 키**(`Skill.Slash`·`Skill.Block.Desc`)만, strings.csv에서 해석.
  - enum성 분류값(`Kind`·`EffectType`·`Target` 등)은 로컬라이즈 문자열이 아니라 **UPPER_SNAKE 원자 토큰**(`PHYS`·`MAG`·`DMG_REDUCE`·`SELF`) — 코드 분기 안정용, 표시명은 별도 `NameKey`.
  - 수치는 단위 내장: `Hp`·`Atk`·`CooldownSec`·`Rate`(0~1). 불리언 `Is~`/`Can~`. 기획 메모는 `Memo`(게임 미사용).
  - 파일명 snake_case(`parts.csv`) ↔ DataTable `DT_Parts` 1:1 — **다단어는 `_` 제거+세그먼트별 PascalCase**(`job_stats.csv`→`DT_JobStats`, `_tables.csv`→`DT_Tables`).
- **순환 예외(오너 확정 2026-07-08)**: 순환이 본질인 도메인은 예외 허용 — 예) 랜덤박스: `items` → 박스 내용(`rewards`) → reward가 다시 `items`를 참조. 단 ①예외는 `_tables.csv`에 "순환 예외" 표기로 **명시 등록**(무단 순환 금지) ②구현 시 무한 재귀 가드(중첩 깊이 상한) 필수.
