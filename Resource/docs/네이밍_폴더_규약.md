### 1-B. 신규 확정 접두사 (이 문서에서 최초 규정)

| 접두사 | 대상 애셋 | 예 | 근거·우선순위 |
|---|---|---|---|
| **`F_`** | **UserDefinedStruct**(특히 DataTable 행 struct) | `F_SkillRow` · `F_StringsRow` | **최우선.** 모든 DataTable이 행 struct를 필수로 요구하는데 기존 7종에 struct 슬롯이 아예 없다. [[A0_CSV파이프라인스파이크]]에서 기존 struct 재사용 불가가 확정됐고(엔진 15종 중 `Key,ko,ja,en` 스키마 대응 0건), 규칙 부재 상태에서 `S_Strings_SpikeTest`가 임의 명명으로 제안된 전례가 있다 — 매번 임의 결정되던 공백을 여기서 닫는다. |
| **`E_`** | **UserDefinedEnum** | `E_BattleState` · `E_DamageType` | GameInstance `BattleResult{Win,Lose,Walkover}` 등 BP enum이 이미 존재/예정. §4의 CSV enum 토큰(UPPER_SNAKE)과 짝을 이룸(값=토큰, 타입=`E_`). |

**접두사 선택 근거(`F_`/`E_`)**: UE C++ 관례(`FTableRowBase` 파생 struct=`F`, enum=`E`) 계승 — cpp-engineer 투입(MVP 이후) 시 C++ struct/enum과 무충돌. `S_`는 기존 `SM_`(StaticMesh)·`ST_`(StringTable)와 시각적으로 근접해 회피. A0③의 임의 `S_` 명명은 이 규칙으로 대체하되, **이미 생성된 `S_` struct가 있다면 소급 리네임하지 않고 §3 레지스트리(`cost-deferred-grandfather`)로 처리**한다.

> **애셋명 ≠ C++ 타입명(의도된 차이)**: UserDefinedStruct **애셋**은 언더스코어 포함(`F_SkillRow`), 향후 C++로 미러링할 때의 **타입명**은 UE 문법상 언더스코어 없는 `FSkillRow`다 — 둘이 다른 것은 의도된 것(애셋 접두사 규약과 C++ 타입 명명은 별개 레이어). 같은 논리로 `E_BattleState`(애셋) ↔ `EBattleState`(C++ enum).

## 2. 명명 순서 원칙

신규 애셋의 세그먼트 순서를 통일한다(오너가 지적한 `M_Ground`/`M_Sprite_Lit`/`M_UI_ButtonBg` 순서 제각각 문제 해소). **기존 애셋 소급 리네임은 안 함**(§3 레지스트리 등재).

```
<접두사>_[도메인]_대상[_속성][_변형]
   필수      옵션    필수   옵션    옵션
```

| 세그먼트 | 뜻 | 예 |
|---|---|---|
| 접두사 | §1 애셋 타입 | `T_`·`M_`·`MI_`… |
| [도메인] | **기능 영역**(무엇에 관한 애셋인가) — 옵션 | `UI`·`Sprite`·`FX`·`Env`·`Char` |
| 대상 | 그 안의 **개체** — 필수 | `Frame`·`Ground`·`ButtonBg` |
| [속성] | 개체의 변형 축(무엇) — 옵션 | `Lit`·`Gold` |
| [변형] | 포맷/버전/색상 — 옵션 | `9slice`·`_c2`·`_v3` |

**모범 사례**: `T_UI_Frame_9slice` = `T_` + 도메인 `UI` + 대상 `Frame` + 변형 `9slice`. `T_UI_Icon_Gold` = `T_`+`UI`+`Icon`+속성 `Gold`. `M_UI_ButtonBg` = `M_`+`UI`+대상 `ButtonBg`.

### 카브아웃 2개 (순서만으로는 재발 방지 안 되는 지점)

**(a) `WBP_`·`BP_`류는 [도메인]을 생략하고, 복합명은 원자 대상으로 취급한다.**
역할은 **폴더**가 읽는다(WBP_는 `frontend/battle/common/components`, [[UI_화면_규약]] §D 계승). 따라서 `WBP_BattleHUD`·`WBP_CharacterCard`의 "Battle"·"Character"는 **도메인 분절이 아니라 원자 PascalCase 대상**이다. 순서 원칙만 보고 `WBP_Battle_HUD`·`WBP_UI_Button`으로 **쪼개지 말 것**(각주 ※1의 언더스코어 충돌과도 연결).

**(b) [도메인] 슬롯은 통제 어휘만 쓴다.**
도메인은 임의로 만들지 않고 아래 목록에서 고른다(확장 시 이 문서에 추가):

| 도메인 | 범위 |
|---|---|
| `UI` | HUD·위젯용 텍스처/머티리얼 |
| `Sprite` | 캐릭터/유닛 스프라이트 렌더 |
| `FX` | VFX(자체제작) |
| `Env` | 환경/배경 |
| `Char` | 캐릭터 파츠/합성 |

> **판단 기준**: **도메인 = 기능 영역**(어느 시스템에 속하는가), **대상 = 그 영역 안의 개체**. `M_Ground`는 도메인 없이 대상 `Ground`(범용 지형 머티리얼), `M_Sprite_Lit`은 도메인 `Sprite`+속성 `Lit`. 도메인이 애매하면 생략이 원칙(억지로 붙이지 않는다).
>
> **타이브레이커(`Env` 오남용 방지)**: 범용 지형/환경 마스터처럼 특정 맵·배경 세트에 매이지 않는 **광역 애셋은 `Env`를 붙이지 않고 생략**한다 — `M_Ground`(O), `M_Env_Ground`(X). `Env` 도메인은 **특정 배경 세트/맵에 귀속된** 환경 애셋에만 쓴다.

## 3. 기존 이탈 레지스트리 (일원화)

규칙 확정 전 애셋의 이탈을 **한 곳에 사유태그와 함께 등재**한다. 리네임은 하지 않고 **추적성·가시성만 확보**한다.

| 이탈 패턴 | 사유태그 | 근거 | 해소 조건 | 처리 |
|---|---|---|---|---|
| `T_Party_B*`(적으로 쓰이는 소스 텍스처) | **`semantic-exception`** | `compose_party.py:148`이 전투 역할과 무관하게 **전량 `T_Party_{ID}`로 고정 출력** → A/B 시리즈 모두 `T_Party_`. "**소스=정체성 / 사용처=역할**" 원칙(아래)상 이탈은 **T_ 레이어 하나뿐** — MI 레이어(`MI_Party_A*` / `MI_Enemy_B*`)는 텍스처 파라미터만 공유 `T_Party_*`를 가리키고 **자기 이름은 이미 역할대로 정확히 갈라져 있음**(이미 검증된 패턴). 기능 crit([[기본전투무대]] plan `[S2][01]`·`[S2][06]` PASS)은 **존재·부모 배선만** 통과 — **명명 심의는 이번이 최초**(과거 QA가 "명명 예외로 승인"한 바 없음). | **balance #1**(뽑기 레시피 → JobId/GradeId 결정) 시점. **근접 아님** — 8레이어 런타임 합성 프로덕션 전환([[A0_합성머티리얼스파이크]] 후속)+balance #1 둘 다 미확정. | 리네임 안 함(`M_Sprite_PartComposite` 런타임 합성으로 대체 예정, 비용>실익). **향후 compose 스크립트 신규 합성 시 역할어(Party) 대신 중립 셋토큰 권고**(art-pipeline 소관). |
| `M_Ground` · `M_Sprite_Lit` · `M_UI_ButtonBg` (도메인/대상/속성 순서 제각각) | **`cost-deferred-grandfather`** | §2 순서 원칙 확정 전 생성. | 없음(영구 grandfather). | 리네임 안 함. **신규 머티리얼만 §2 순서 준수.** |
| (조건부·현재 비활성) A0③ 계열 임의 `S_` struct | **`cost-deferred-grandfather`** | `F_`/`E_` 확정(§1-B) 전 규칙 부재로 `S_` 임의 명명 *제안*. **2026-07-12 실측 실제 생성물 0건**(qa-critic MCP 확인) — 이탈 애셋 없음, 예방적 등재. | 없음. | [[A0_CSV파이프라인스파이크]] 재개 시 struct는 **이제 `F_` 규칙 적용**(임의 `S_` 아님). 향후 `S_` 생성물 발견 시에만 이 행 활성화(리네임 안 함). |

> **재발 방지 원칙 — "소스 = 정체성 / 사용처 = 역할"**: 소스 애셋(`T_`·`SM_`)의 도메인 토큰은 **캐릭터/합성셋의 정체성**을 담고, **전투 역할(party/enemy)은 사용처 애셋(`MI_`)·액터(`BP_`/SpawnPoint) 층위**가 담는다. 같은 캐릭터가 아군도 적도 될 수 있으므로 역할은 소스가 아니라 사용처에 속한다. 이 원칙 하에 `MI_Enemy_B1`은 정당(사용처=적), `T_Party_B1`은 토큰 오용(소스에 역할어)이다.
>
> **BP 컨테이너명 충돌**은 이탈 사례가 아니라 **예방 규칙** → §1 각주 ※1 참조.

## 4. CSV / 데이터 명명 (요약 + 교차참조)

원 출처는 [[데이터_서버_규약]] **§4**(구 [[알파_개발계획]] §2.6②-b — 오너 확정 + balance-designer 컬럼 규칙 보강, 2026-07-13 분리). 여기서는 **재서술하지 않고 요약·인용**한다.

### 4-A. 컬럼(필드) 레벨 규칙 4종 ([[데이터_서버_규약]] §4 인용)

| 규칙 | 형태 | 예 | 출처 |
|---|---|---|---|
| 외래키 접미 | `<대상단수>Id` / 다중 `<대상단수>Ids`(구분자 `;`) | `SkillId` · `PartIds`(`24;25;26`) | §4 외래키 |
| 표시·설명 문자열 | 원문 저장 금지 → `<대상>Key` 접미 컬럼에 **dot-namespace 키**만(strings.csv에서 해석) | `NameKey`=`Skill.Slash` · `DescKey`=`Skill.Block.Desc` | §4 표시명 |
| enum성 분류값 | 로컬라이즈 문자열 아님 → **UPPER_SNAKE 원자 토큰**(코드 분기 안정용) | `Kind`=`PHYS` · `EffectType`=`DMG_REDUCE` · `Target`=`SELF` | §4 enum |
| 파일명 ↔ DT_ 유도 | snake_case 파일 ↔ `DT_` PascalCase 1:1, **다단어는 `_` 제거+세그먼트별 PascalCase** | `job_stats.csv` → `DT_JobStats` · `_tables.csv` → `DT_Tables` | §4 파일명 |

(그 외 수치 단위 내장 `Hp`/`Atk`/`Rate`, `Memo` 미사용 컬럼 등은 [[데이터_서버_규약]] §4 원문 참조.)

### 4-B. 구조적 사실 (반복 질문 차단용 1줄)

> **CSV 컬럼 = DataTable 행 struct 필드 = BP 노출 멤버**는 우연이 아니라 **임포트가 강제하는 구조적 일치**다(셋 다 동일 PascalCase). CSV 헤더는 struct 필드명과 **대소문자까지 정확히 일치**해야 매핑된다([[A0_CSV파이프라인스파이크]] §2-3 경고).

### 4-C. bool 컬럼 접두 정책 — ⚠ 잠정 확정 (실증 후 재확인)

[[데이터_서버_규약]] §4는 CSV bool 컬럼을 `Is~`/`Can~`로 규정한다. 그런데 4-B의 구조적 일치 때문에 **동일 필드가 BP에서도 그 이름으로 노출**되어, §8의 BP `b`접두 규칙(`bIsAOE`)과 **같은 필드에서 충돌**한다(`IsAOE`(CSV/struct) vs `bIsAOE`(BP)).

**레이어 분리로 해소(잠정 확정)**:

| 레이어 | 규칙 | 예 |
|---|---|---|
| **데이터 레이어**(CSV 컬럼 + DataTable 행 struct 필드) | `Is~`/`Can~`, **`b` 접두 안 붙임**([[데이터_서버_규약]] §4 기존 규칙이 권위) | `IsAOE` · `CanCrit` |
| **BP 그래프 내부**(struct 유래가 **아닌** BP 저작 bool: 지역변수·계산 플래그·비-struct 멤버) | `b` 접두(§8) | `bIsSelectingTarget` |
| **경계** | DataTable/struct 유래 bool은 **§8 `b`접두 규칙의 예외** — 데이터 레이어 이름을 그대로 유지(`IsAOE`, `bIsAOE` 아님). `b`접두는 struct 경계를 넘지 않는 BP 저작 bool에만. | |

> **🚩 실증 대기(계획 명시 조건)**: UE의 struct bool 필드 표시-스트립 이력이 있어, **임포트가 CSV `IsAOE` 헤더를 `b`접두 struct 필드에 어떻게 매칭하는지가 미검증**이다. **gameplay-engineer/verifier가 실제 CSV→DataTable 임포트 1회로** (a) `IsAOE` 헤더가 `b` 없는 struct 필드로 정확히 매칭되는지 (b) BP 노출 시 실제 이름을 확인한 뒤 **최종 확정**한다. 그 전까지 위 레이어 분리는 **잠정**이다.
>
> **실제 위험도(뉘앙스)**: 알파는 오너가 에디터에서 **UserDefinedStruct** 필드를 직접 만드는 경로라, **CSV 헤더와 struct 필드명을 동일하게 타이핑하면(`IsAOE`=`IsAOE`) 그대로 매칭**된다 — `b`-스트립 불일치는 주로 **네이티브 C++ struct**(`UPROPERTY` bool이 `b`접두를 강제하고 UE가 표시명에서 스트립)에서 발현한다. 따라서 **알파 범위에선 실발생 확률이 낮고**, C++ 미러링(MVP 이후) 시점이 진짜 검증 지점이다.

## 9. 위젯 트리 요소 명명 (신규)

WBP 내부에 **오너가 수동 배치**한 자식 위젯(Button/TextBlock 등)의 이름 규칙. **`<타입약어>_<역할>`**.

| 타입약어 | 위젯 | | 타입약어 | 위젯 |
|---|---|---|---|---|
| `Btn` | Button | | `List` | ListView/TileView |
| `Txt` | TextBlock | | `Border` | Border |
| `Img` | Image | | `Overlay` | Overlay |
| `Panel` | CanvasPanel/기타 패널 | | `Icon` | 아이콘 이미지 |
| `Switcher` | WidgetSwitcher | | `Bar` | ProgressBar |
| `Chk` | CheckBox | | `Sld` | Slider |

- **형제 요소는 숫자 접미 대신 의미 role**: `Btn_LangKo`/`Btn_LangJa`/`Btn_LangEn` (O) — `Btn_LangPill_1/2/3` (X). ([[UI_화면_규약]] §C-1 `WBP_Options` `design_v1.html` 실사례 기준.)
- **role 토큰은 영어 전용**(한글 자모는 오타·인코딩 발 MCP 오진 회피).

> **🚩 의존 전제 + 미검증 (문구 정정)**: §B 파이프라인 5단계(오너 배치 → Claude MCP 바인딩)는 이 명명에 **의존을 전제**하지만, **아직 미검증**이다 — [[A0_UMG스파이크]]는 **빈 `WBP_Probe` 셸 생성까지만** 확인했고, 오너가 이름 붙여 배치한 위젯의 실제 바인딩 성공 여부는 테스트되지 않았다. → **다음 실화면(`WBP_Options`) 골격 생성 직후, 버튼 1개를 이름으로 바인딩 시도하는 스모크 테스트를 §B 5단계 진입 게이트로 둔다.** 실패해도 A0①의 오너 수동 배선 폴백이 있으므로 파이프라인은 막히지 않는다.
