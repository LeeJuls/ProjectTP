# HD-2D 정면 대치형 실시간 ATB PvP — 기술 설계 및 가능성 판단

> 작성일: 2026-06-28 / 엔진: UE 5.8 (`D:\unreal\UE_5.8`) / 상태: **설계 검토 단계, 구현 미착수**
> 캐릭터 에셋: au-pixel **heroes99** (32px 도트, 정면 중심, 상업 사용 가능)
> ⚠ **알파 현재 구현과 구분**: 이 문서의 실시간 ATB·서버 권위·GAS/네트워킹은 **베타 스코프**다. 알파(A1)의 실제 전투 로직(고정 턴 순환·핫시트, HP·데미지·사망·승패·스킬5종·상태이상2종·광폭화)은 별도 진행 중 — [[features/전투완성/청사진]] 참고.

---

## 0. 한 줄 결론

**가능하다. 단, 난이도의 무게중심은 "HD-2D 그래픽"이 아니라 "실시간 PvP 네트워킹 + 서버 권위 ATB"에 있다.**
그래픽은 검증된 기법으로 풀 수 있는 영역이고, 진짜 리스크는 _두 유저의 ATB 게이지가 동시에 실시간으로 흐르는 상태에서 누가 먼저 쳤는지를 공정하게 판정_ 하는 부분이다. 이 문서는 그 판정을 어떻게 서버 권위로 처리할지를 중심으로 설계한다.

## 1. 게임 정의 (확정된 요구사항)

| 항목 | 내용 |
|---|---|
| 장르 | 1v1 PvP, 정면 대치형 파티 배틀 |
| 전투 시간 모델 | **실시간 ATB** — 일시정지 없음. 캐릭터별 ATB 게이지가 실시간으로 충전 |
| 핵심 긴장 | 내가 커맨드를 고민하는 동안 상대 게이지가 차면 **상대가 먼저 행동** |
| 시점/연출 | HD-2D (2D 스프라이트 + 3D 환경 + 라이팅/포스트프로세싱) |
| 콘텐츠 범위 | **전투만** (필드 탐험 없음) |
| 플레이어 수 | 2명 (사람 vs 사람) |

### 이 정의에서 파생되는 결정적 제약
- 실시간 + PvP + 일시정지 없음 = **클라이언트를 신뢰할 수 없다.** 게이지 충전 속도, 행동 발동 시점, 데미지 계산은 **전부 서버가 권위**를 가져야 한다. 클라가 "내 게이지 다 찼다"고 주장하게 두면 즉시 치트.
- "고민하는 동안 상대가 먼저 친다"는 핵심 재미 = **시간이 곧 게임플레이.** 따라서 입력 지연(latency) 보정과 서버 시간 동기화가 게임 공정성의 핵심이다.

## 2. 세 가지 도전 과제와 난이도 평가

| # | 과제 | 난이도 | 리스크 |
|---|---|---|---|
| A | HD-2D 렌더링 (32px 스프라이트를 3D 라이팅 씬에) | ★★☆☆☆ | 낮음 — 검증된 기법, 에셋 한계만 관리 |
| B | 실시간 ATB 전투 로직 | ★★★☆☆ | 중간 — GAS로 구조화 가능 |
| C | **서버 권위 PvP 네트워킹 + 공정한 선후 판정** | ★★★★★ | **높음 — 프로젝트 성패가 여기 달림** |

## 3. 기술 스택 권장

| 레이어 | 선택 | 이유 |
|---|---|---|
| 스프라이트 애니메이션 | **PaperZD** (마켓 플러그인) 위에 Paper2D | Paper2D 단독은 상태머신/노티파이가 빈약. PaperZD가 애니메이션 상태머신·이벤트 노티파이 제공 → 공격 타이밍에 데미지 이벤트 걸기 좋음 |
| 전투 능력/속성 | **GAS (Gameplay Ability System)** | ATB 게이지 = `Attribute`(자동 리젠), 스킬 = `GameplayAbility`, 버프/상태이상 = `GameplayEffect`. **서버 권위 리플리케이션이 기본 내장** → C 문제의 절반을 GAS가 흡수 |
| 네트워크 토폴로지 | **Dedicated Server 권장** (초기엔 Listen Server 허용) | PvP 공정성 = 양쪽 누구도 호스트 어드밴티지 없어야. 출시 지향이면 Dedicated가 정답 |
| 캐릭터 합성(레이어) | 런타임 머티리얼 / 스프라이트 레이어 합성 | heroes99의 헤어·의상·팔레트 조합을 살리려면 |

> **핵심 아키텍처 주장:** ATB를 직접 타이머로 짜지 말고 **GAS Attribute의 리젠 메커니즘으로 모델링**하라. "게이지가 100% 차면 행동 가능" = 어트리뷰트가 max 도달 시 어빌리티 활성화 게이트 해제. 이러면 충전·소비·증감 버프(헤이스트/슬로우)가 전부 GameplayEffect로 자연스럽게 표현되고, 무엇보다 **서버 권위와 리플리케이션을 GAS가 공짜로** 준다.

## 4. HD-2D 렌더링 파이프라인 설계 (과제 A)

### 4.1 기본 구성
- 캐릭터/적: **빌보드 스프라이트**(카메라를 향해 Yaw만 정렬). 옥토패스식 "살짝 기울어진 카메라 + 직립 스프라이트".
- 배경/무대: 실제 3D 메시(전투 무대 바닥, 기둥, 소품) → 여기서 "HD"감과 깊이가 나온다.
- 카메라: 원근(Perspective) + 약한 DoF로 미니어처 디오라마 느낌.

### 4.2 스프라이트가 라이팅을 받게 하기
- 스프라이트 머티리얼을 **Unlit이 아니라 Lit**으로. 단, 32px 도트의 톤을 죽이지 않도록 라이팅 기여도를 머티리얼에서 조절.
- **노멀맵이 없다**(heroes99 미제공). 두 가지 선택:
  1. 라이팅을 "분위기"로만 사용 (방향광 약하게 + 환경광), 노멀맵 없이 평면 스프라이트에 컬러 그레이딩 → **초기 권장**, 비용 0.
  2. 노멀맵 생성(Laigter, Sprite DLight 등)으로 동적 음영 → 후순위 폴리시.

### 4.3 픽셀 퍼펙트 vs HD-2D 블룸의 충돌 (중요)
- 32px는 매우 낮은 해상도. 강한 블룸/DoF를 직접 걸면 도트가 뭉개진다.
- 해결: **스프라이트는 픽셀 퍼펙트(Nearest 필터, 정수배 스케일)로 렌더 → 포스트프로세싱(블룸/DoF/톤매핑)은 배경 위주로**. 캐릭터에는 약하게.
- Texture: Filter=Nearest, Mip 비활성, 압축 최소(UI/도트용).

### 4.4 포스트프로세싱 레시피(시작점)
- Bloom: 약하게 (배경 발광 소품 강조용)
- Depth of Field: 전경/후경에만, 캐릭터 평면은 또렷하게
- Tonemapper + 약한 Vignette + Color Grading LUT
- (선택) Lumen은 무대 3D 메시에만 의미. 도트 캐릭터엔 과함 → 초기엔 끄고 검토.

### 4.5 에셋(heroes99) 적용
- A형(정면 대치 턴제)에 **그대로 잘 맞음**: 정면 idle/공격 콤보/스킬/회피 애니메이션이 전투 화면에 직결.
- 방향 스프라이트 부재 문제 = **전투만 하므로 해당 없음.** (이게 A형을 고른 핵심 이점)
- 라이선스: 상업 사용·수정 OK / 재판매·NFT·AI학습 금지 → PvP 상용 게임 배포 가능. **단, 원본 시트 재배포 금지에 유의**(패키징된 게임 내 사용은 무방).

## 5. 실시간 ATB 전투 시스템 설계 (과제 B)

### 5.1 데이터 모델 (GAS 기준)
- `ATBCharge` (Attribute, 0~100): 매 틱 `Speed`에 비례해 서버에서 증가.
- `Speed` (Attribute): 헤이스트/슬로우 GameplayEffect로 가감.
- 행동 가능 게이트: `ATBCharge >= 100`이면 해당 캐릭터의 커맨드 입력 허용.
- 행동 발동 시: 어빌리티 실행 → `ATBCharge`를 0(또는 행동별 코스트만큼) 소비.

### 5.2 입력→발동 흐름
1. 게이지 풀 충전 → 클라 UI에 "행동 가능" 표시(예측 표시 가능, 권위는 서버).
2. 유저가 캐릭터+커맨드 선택 → 서버에 RPC.
3. 서버가 "그 시점에 정말 게이지가 찼는지" 검증 후 어빌리티 실행.
4. 어빌리티 애니메이션(PaperZD)의 타격 프레임 노티파이 → 데미지 GameplayEffect 적용(서버 권위).
5. 결과 리플리케이션 → 양 클라 연출 재생.

### 5.3 "고민하는 동안 상대가 먼저" 메커니즘
- 게이지 충전은 **서버에서 멈추지 않고 계속** 흐른다(일시정지 없음).
- 유저가 커맨드 창을 열어둔 상태에서도 서버 시간은 진행 → 상대 게이지가 먼저 100 도달하면 상대 어빌리티가 먼저 큐에 들어가 해결.
- 동시 도달 충돌은 §6.3의 타임스탬프 규칙으로 판정.

## 6. 서버 권위 네트워킹 — 프로젝트의 심장 (과제 C)

### 6.1 권위 원칙
- **서버만이 진실:** 게이지 값, 충전 속도, 행동 발동 시각, 데미지. 클라는 입력 전송 + 연출 재생만.
- 클라 ATB 바는 **서버 값의 시각적 예측/보간**일 뿐. 불일치 시 서버 값으로 스냅.

### 6.2 시간 동기화
- 서버 권위 클럭(서버 월드 타임)을 기준. 클라는 RTT 측정으로 서버 시간을 추정.
- ATB는 "초당 X% 충전"의 결정적 함수 → 클라는 마지막 서버 동기점 + 경과시간으로 부드럽게 예측, 주기적 서버 리플리케이션으로 보정.

### 6.3 선후(先後) 판정 규칙 — 공정성의 핵심
- 두 행동이 근접 시각에 발동 시: **서버가 수신·검증한 "게이지 100 도달 서버 시각"이 빠른 쪽이 우선.**
- 입력 RPC 도착 순서가 아니라 **"게이지가 찬 서버 시각"** 기준으로 정렬해야 핑 낮은 사람이 무조건 이기는 불공정을 줄인다.
- 한 행동 해결(애니메이션/데미지) 중 들어온 다른 행동은 큐잉. 해결의 원자성(누가 죽으면 그 캐릭터 행동 취소 등) 규칙 명문화 필요. → **§10 미결정**

### 6.4 지연(latency) 보정 전략
- 입력 지연: 커맨드 RPC의 왕복만큼 내 행동이 늦게 보일 수 있음 → UI 즉시 피드백(예측) + 서버 확정 시 보정.
- 연출은 클라에서 즉시 시작(반응성), 결과(데미지/사망)는 서버 확정 후 동기화.
- 핑 격차 공정성: 위 §6.3의 서버시각 기준 + 적정 입력 버퍼(예: 소량의 의도적 지연으로 양쪽 평준화)를 검토. → **§10 미결정**

### 6.5 치트 방지
- 모든 검증을 서버에서. 클라 RPC는 "이 캐릭터로 이 커맨드를 쓰겠다"는 _의사_만 전달, 가능 여부·결과는 서버 판단.
- 속도 핵/게이지 조작 불가(클라가 게이지를 안 들고 있음).

### 6.6 토폴로지 권장
- 개발 초기: **Listen Server**(빠른 반복). 단, 호스트 어드밴티지 존재 → 밸런스 측정엔 부적합.
- 출시 지향: **Dedicated Server**. 매치메이킹/세션은 추후. 초기엔 직접 접속(IP)로 2인 테스트.

## 7. 프로젝트 구조 제안 (구현 착수 시)

```
Resource/                         # 프로젝트 루트
├─ docs/                          # 본 설계 문서 등
├─ Source/<Proj>/
│  ├─ Abilities/                  # GAS: 어빌리티, 이펙트, 어트리뷰트셋(ATBCharge, Speed, HP...)
│  ├─ Battle/                     # 전투 상태기계, 행동 큐, 선후 판정 매니저(서버 권위)
│  ├─ Net/                        # 시간 동기화, RPC, 리플리케이션 헬퍼
│  ├─ Characters/                 # 스프라이트 액터, PaperZD 연동, 레이어 합성
│  └─ UI/                         # ATB 바, 커맨드 창 (로컬라이즈 키 사용, 하드코딩 금지)
└─ Content/
   ├─ Sprites/Heroes99/           # 임포트된 시트 (재배포 아님, 패키지 내 사용)
   ├─ Flipbooks/                  # PaperZD 애니메이션 소스
   ├─ Materials/                  # Lit 스프라이트 머티리얼, 포스트프로세스
   └─ Stages/                     # 3D 전투 무대 메시
```

## 8. 마일스톤 로드맵 (제안)

> 각 마일스톤은 "계획→구현→검증" 사이클로. 검증 없이 다음 단계 금지.

- **M0 — 기술 검증 스파이크 (가장 먼저, 가장 중요)**
  - heroes99 시트 1캐릭터를 UE5.8에 임포트 → PaperZD 플립북 → 3D 무대 위에 Lit 스프라이트로 배치 → 픽셀 퍼펙트 + 약한 포스트프로세스 확인.
  - **동시에** 더미 2클라 Listen Server에서 ATB 어트리뷰트 1개가 서버 권위로 충전·리플리케이트되는지 확인.
  - 목적: A와 C의 가장 큰 미지수를 _초반에_ 깨본다.
- **M1 — 단일 캐릭터 1v1 ATB 코어**: 게이지 충전 → 기본 공격 1종 → 데미지 → 사망. 서버 권위. 선후 판정 1차.
- **M2 — 파티 + 스킬/콤보 + 상태이상**: GAS 어빌리티/이펙트 확장, PaperZD 타격 노티파이.
- **M3 — 공정성/지연 보정 튜닝**: 핑 격차 테스트, 선후 규칙 확정, 입력 버퍼.
- **M4 — HD-2D 폴리시**: 노멀맵/라이팅, 무대 연출, 포스트프로세스 고도화.
- **M5 — Dedicated Server + 매치/세션** (출시 지향 시).
