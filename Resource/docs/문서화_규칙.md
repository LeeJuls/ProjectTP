# 📝 문서화 규칙 (Obsidian wiki)

> 모든 작업 산출물·중간 기록을 이 볼트에 남기는 규칙. Director와 전 에이전트가 준수한다.
> 관련: [[projectTP_허브]] · [[개발_워크플로우]]
#projectTP/규칙

---

## 1. 정리본 / raw 분리 (Karpathy RAG 방식)
- **정리본**: 기능 폴더 루트 `docs/features/<기능>/*.md` (청사진·plan·진행로그).
- **중간 산출물(raw)**: `docs/features/<기능>/raw/` — 에이전트 보고 원문, 게이트 결과, 중간 스크린샷.
- 정리본이 raw를 위키링크로 참조한다. raw는 버리지 않고 보존.

## 2. 중간 문서도 반드시 기록
- 단계 게이트 결과·TC 실행 결과·주요 결정 → 기능 폴더 `진행로그.md`에 **시간순 append**.
- 스크린샷은 `raw/`에 저장하고 `![[파일]]`로 임베드.
- "말로만 끝내지 않는다" — 실행 에이전트 보고도 요지를 진행로그에 남긴다.

## 3. 옵시디언 문법 적극 활용
- 위키링크 `[[노트]]` · `[[노트#헤딩]]` · 블록참조 `^id`.
- 계층 태그 `#projectTP/<기능>` (예: `#projectTP/기본전투무대`).
- frontmatter 필수: `type` / `feature`(기능문서) / `status` / `updated`.

## 4. 연결 원칙 (고아 노트 금지)
- 새 노트는 최소 1개 링크(허브 또는 상위 문서)로 연결한다.
- 기능 완료 시 [[projectTP_허브]] 문서목록/현재상태를 갱신한다.

## 5. 단순 유지
- 폴더 깊이 최대 3: `docs/features/<기능>/raw/`.
- 파일명 = 한글 명사형. 날짜는 frontmatter `updated:`에만(파일명에 날짜 금지).

## 기능 폴더 표준 구조
```
docs/features/<기능>/
├─ 청사진.md      ← 목표·범위·전제 (blueprint)
├─ plan.md        ← 단계·TC·진행 체크리스트 (plan)
├─ 진행로그.md     ← 게이트·결정·이슈 시간순 (log)
└─ raw/           ← 에이전트 원문·스크린샷·중간물
```
