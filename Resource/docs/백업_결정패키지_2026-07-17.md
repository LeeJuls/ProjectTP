## 5. 권고안

### 권고: **Git LFS** (first-party 폴더만, 기존 GitHub 원격에), `_savepoints` 관행은 보조 수단으로 병행 유지

**이유** (실측 크기가 근거의 중심):

1. **용량이 결정을 단순하게 만든다.** first-party 54.33 MB는 GitHub 무료 LFS 한도(10 GiB)의 0.53%에 불과하다. Content 전체(1.25 GB, Village Pack 포함해 최악의 경우까지 가정해도)조차 12.5%다. 이 프로젝트가 현재 성장 속도로 몇 배 커지더라도 무료 한도 안에 오래 머문다 — "비용" 축이 사실상 실질적 제약이 되지 않는다.
2. **"PC를 잃어도"를 실제로 만족시키는 유일한 후보다.** `_savepoints`(오늘 백업 포함)는 같은 `D:` 드라이브·같은 물리 PC에 있다 — 도난·화재·메인보드 고장처럼 PC 자체를 잃는 시나리오에는 무방비다. Perforce도 서버를 이 PC에 자체 호스팅하면 동일한 약점을 그대로 가진다(서버를 어디 두느냐가 관건이라고 비교표에 적은 이유). Git LFS는 기존 GitHub 원격(이미 연결되어 있고 이미 매일 push 중인 워크플로우)에 올라가는 순간 물리적으로 다른 위치에 사본이 생긴다.
3. **잠금은 지금 필요 없고, 나중에 공짜로 켤 수 있다.** 현재 커밋 131/131이 단일 저자(`LeeJuls`) — 동시 편집 충돌이 발생할 수 없는 구조다. Perforce의 핵심 강점(배타적 잠금)은 정확히 이 프로젝트에 지금 쓸모가 없는 기능이다. Git LFS도 잠금 기능(`git lfs lock`)을 갖고 있어 팀이 커지면 같은 비용 구조에서 활성화할 수 있다 — Perforce로 미리 옮겨갈 이유가 없다.
4. **워크플로우 분열을 피한다.** Perforce는 코드(Git)와 에셋(Perforce)을 서로 다른 시스템으로 쪼갠다. 1인 개발+MVP 단계별 게이트 워크플로우([[개발_워크플로우]])에서 이 분열은 순수 오버헤드다. Git LFS는 기존 커밋 접두사 규칙·`save/` 태그 컨벤션 위에 그대로 얹힌다.
5. **`_savepoints`를 버릴 필요는 없다.** Git LFS 커밋은 "준비됐을 때" 하는 의도적 체크포인트인 반면, `_savepoints` 복사는 오늘처럼 즉흥적·무비용으로 즉시 실행 가능하다. 두 방식은 배타적이지 않다 — Git LFS를 "정식·오프사이트" 계층으로, `_savepoints`를 "빠른·로컬" 계층으로 병행하는 것을 권한다.

**단서**: 이 권고는 "puts a copy off this PC" 문제에 대한 것이지, 완전한 해법은 아니다. 아래 §6 표에서 보듯 Git LFS도 **"커밋 후 push까지 끝난 것"에만** 유효하다 — 저장은 했지만 아직 push 전인 변경분은 여전히 이 PC에만 존재한다. 어떤 백업 방식을 택하든 "자주 push"하는 습관 자체가 §3.2 검증 질문의 실질적 절반을 차지한다.

**채택하지 않음을 권고**: Perforce — 이유는 위 4개 축 전부가 "지금 당장 이 프로젝트(1인, MVP, 소규모)에는 이점이 비용/복잡도 대비 낮음"으로 수렴하기 때문이다. 팀이 실제로 늘거나 바이너리 동시편집 충돌이 실제로 반복 발생하면 재검토할 가치가 있다.
