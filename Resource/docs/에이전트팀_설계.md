# 🤝 에이전트 팀 설계

> UE 5.8 HD-2D 턴제 PvP 배틀을 만드는 **멀티에이전트 팀 청사진**.
> Director(메인 대화)가 전문 에이전트들을 오케스트레이션한다.
> 관련: [[projectTP_허브]] · [[작업로그_HD2D아트검증_플레이북]]

---

## 📌 구조 원칙

- **메인 대화 = Director.** Subagent는 서로 못 부른다. 오직 Director만 팀을 지휘한다.
- **Fable 토큰 효율**: "머리"(설계·판단·검증)만 Fable/Opus, "손"(반복 실행)은 Sonnet/Haiku.
- **모든 에이전트는 [[projectTP_허브]] 상단 CLAUDE.md 규칙 상속**: 근본원인 없는 수정 금지 / "됐습니다" 전 실증 / 커밋 접두사 `[C] type:` / push는 오너 확인.

## 🔍 두 검증자의 구분 (중요)

| | **qa-critic** | **verifier** |
|---|---|---|
| 질문 | "논리적으로 모순 없는가?" | "실제로 돌아가는가?" |
| 방식 | 설계·코드 적대적 리뷰 (사고 기반) | 빌드·실행·스크린샷 (실증 기반) |
| 시점 | 설계 단계부터, 교차검증 | 구현 후 |
| 대상 | 기획 모순·예외처리 누락·명세불일치 | 컴파일에러·크래시·렌더 실패 |

**qa-critic 기본 태도 = 적대적(의심).** "이게 어떻게 깨지는가"를 찾는다. 발견 항목은 심각도순 리포트 → Director가 수정 지시.

## 🔄 표준 워크플로우 (설계 → 구현 → 검증)

```
① Director 계획·작업분해
      │
② 설계   system-ui-designer + balance-designer  ──┐
      │                                            │ (qa-critic 사전 검토:
③ [qa-critic] 기획 모순·구멍 검출 ◄────────────────┘  기획 모순 있나?)
      │
④ 구현   gameplay-engineer / art-pipeline / scene-builder / hd2d-art-director
      │
⑤ [qa-critic] 코드 엣지케이스·명세불일치 검출
      │
⑥ [verifier] 빌드 0·PIE·스크린샷 실증
      │
⑦ Director 결과 종합 → 오너 보고 (미적 판단은 오너)
```

- 설계·구현 결과물은 **qa-critic이 먼저 깨보고**, 통과하면 **verifier가 작동을 실증**.
- HD-2D "멋짐"의 최종 미적 판단은 **오너(인간)**. art-director는 스크린샷으로 제안만.

## 💰 모델 배분 전략 (Fable 절약)

- **Fable/Opus** (사고): Director, system-ui-designer, balance-designer, qa-critic, hd2d-art-director(룩 설계 시)
- **Sonnet** (실행): gameplay-engineer, scene-builder, art-pipeline, hd2d-art-director(적용 시), cpp-engineer
- **Haiku** (단순 확인): verifier
- **원칙**: Director가 **명확한 지시서**를 주면 실행 에이전트는 짧은 컨텍스트로 처리 → 상위 모델 토큰 최소화.

## 🎨 아트 3인방 역할 경계 (겹침 방지)

```
art-pipeline      → 재료 손질  (스프라이트 합성·임포트·플립북 슬라이스)
scene-builder     → 무대 세팅  (배경 배치·구성·레이아웃)
hd2d-art-director → 플레이팅   (라이팅·포스트프로세스·카메라 = "멋짐")
```

**아트 창작 경계**: 에셋 가공·통합·룩개발 = agent. 새 픽셀아트 원화·3D모델링·미적 최종판단 = 인간/구매.
